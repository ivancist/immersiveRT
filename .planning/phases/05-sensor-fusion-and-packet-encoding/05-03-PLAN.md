<objective>
Build the binary packet encoder — the wire contract every downstream phase depends on. `encodePacket`
writes a `SensorPacket` into a fixed 36-byte `DataView` layout (schema v1, D-14) using float16 for
the quaternion/position/displacement fields, guarded by `safeFloat` for NaN/Infinity input (V5).

the ZUPT threshold and Kalman noise. Written test-first (RED → GREEN) because the byte layout is an
exact, machine-checkable contract that Phase 6 will decode against.

Purpose: Deliver PHONE-05 (compact ~40-byte binary encoding) with a provably-36-byte packet, and the
device-adaptive calibration parameters that Plans 06/07 feed into the pipeline.
Output: `client/src/sensor/encode.ts` + `client/tests/encode.test.ts`.
</objective>

## Phase Goal (user story)

**As a** phone player in an active session, **I want to** have my phone's motion and touch fused
on-device (OS-fused orientation, dead-reckoned position, ZUPT-gated gesture displacement) and
streamed to every desktop as a compact 36-byte binary packet at my device's maximum sample rate,
**so that** each desktop can render my movements with low-latency, drift-honest fidelity.

> This plan delivers the encoding contract for that story — the exact 36-byte layout Phase 6 decodes.

<feature>
  <name>Binary sensor packet encoder + calibration math (schema v1)</name>
  <files>client/src/sensor/encode.ts, client/tests/encode.test.ts</files>
  <behavior>
    Schema v1 byte layout (D-14), little-endian throughout:
    - offset 0: uint8 version = 1
    - offset 1: uint16 seq (value written is seq mod 65536)
    - offset 3: uint32 timestamp (ms since session start)
    - offsets 7,9,11,13: float16 qw, qx, qy, qz
    - offsets 15,17,19: float16 dx, dy, dz (gesture displacement)
    - offsets 21,23,25: float16 px, py, pz (dead-reckoning position)
    - offset 27: float32 driftConfidence
    - offset 31: uint8 touchActive (1 or 0)
    - offset 32: uint16 touchX (round(clamp01(x) * 65535))
    - offset 34: uint16 touchY (round(clamp01(y) * 65535))
    - Total: 36 bytes.

    Test cases (encode.test.ts):
    - byte count: encodePacket returns a Uint8Array of byteLength 36.
    - version byte: result[0] === 1.
    - float16 quaternion round-trips within ±0.002: write qw = 0.707, read back via getFloat16(view, 7, true) ≈ 0.707.
    - seq wrap: seq = 65536 writes uint16 0 at offset 1; seq = 65537 writes 1.
    - touch encode: touchActive true → byte 31 === 1; touchX = 0.5 → uint16 at offset 32 ≈ 32768 (±1); touchActive false → byte 31 === 0.
    - safeFloat: safeFloat(NaN) === 0, safeFloat(Infinity) === 0, safeFloat(null) === 0, safeFloat(undefined) === 0, safeFloat(1.5) === 1.5; a packet with qw = NaN encodes to a finite float16 (getFloat16 of that offset is 0, not NaN).
    - computeCalibration: computeCalibration([1,1,1,1,1]) returns { threshold: 0, kalmanQ: 0 } (zero variance); for a sample set with known variance V, threshold === V * 2 and kalmanQ === V * 0.1.
  </behavior>

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Tampering / DoS | NaN / Infinity / null sensor value written as float16 | medium | mitigate | `safeFloat()` clamps every float field to a finite value before the DataView write; test asserts an NaN qw encodes to a finite float16 (0), so the Phase 6 decoder never reads a poison byte pattern |
| T-05-04 | Tampering | reused ArrayBuffer carrying stale bytes | low | mitigate | `encodePacket` overwrites all 36 offsets every call (version..touchY), so no field from a prior frame survives; test encodes distinct packets into the shared `_packetBuf` and asserts independent results |
| T-05-11 | Tampering | touchX/touchY out-of-range producing uint16 overflow | low | mitigate | encodePacket clamps touch coords to [0,1] before `Math.round(v * 65535)`, so the uint16 write can never overflow |
</threat_model>
