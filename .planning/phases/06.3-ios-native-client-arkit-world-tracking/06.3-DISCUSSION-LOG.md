## Session UX

| Option | Description | Selected |
|--------|-------------|----------|
| At pair time (session start) | Origin resets the moment ARSession starts | |
| After a hold-still calibration step | Mirrors web client's runCalibration convention | ✓ |

**User's choice:** After a hold-still calibration step

**Notes:** User questioned the purpose of a phone-side recenter given the desktop already has an R-key reset (`resetAllPlayerPositions()` in `client/src/scene.ts`, built for IMU dead-reckoning drift resets). Claude explained the R-key resets ALL players' rendered position via a desktop-side offset, while a phone-side control would let a single player re-zero just their own tracking without affecting others. User was convinced and asked to add the phone-side recenter, plus a new Disconnect/Back button (disconnect if connected; return to first screen if disconnected/error), both gated to prevent accidental triggers since the full screen is already the touch-capture surface.

| Option | Description | Selected |
|--------|-------------|----------|
| Shared hidden menu | One gesture opens a small overlay with both Recenter and Disconnect/Back | ✓ |
| Separate gestures per control | Each control gets its own distinct trigger | |

**User's choice:** Shared hidden menu

| Option | Description | Selected |
|--------|-------------|----------|
| Long-press a corner | Hold a finger in a screen corner for ~1-2s | |
| Multi-finger tap (e.g. 3-finger tap) | Requires 3 simultaneous touch points | |
| Edge swipe | Swipe in from a screen edge | |

**User's choice (free text):** "2-finger long-press, on both top corners."

| Option | Description | Selected |
|--------|-------------|----------|
| Hidden, same as webxr.ts | Camera passthrough never rendered | |
| Visible camera passthrough | Show live camera feed like a typical AR app | |

**User's choice:** Neither — user clarified this was framed wrong: "It is not about hiding anything. Only data about motion must be used." Reframed as a data-scope rule (D-14): ARKit's camera is used strictly internally for pose computation; no raw camera frames are ever displayed, stored, or transmitted.

**Additional decision (user suggestion, not from a preset option):** Use the existing DynamicToast component (`mobile/ios-app/immersiveRT/DynamicToast/`, `dynamicIslandToast(isPresented:duration:value:)` + `Toast` struct) for all user-facing messages — errors, tracking-status debug info, suggestions — with either a persistent (`duration: nil`) or timed display.

| Option | Description | Selected |
|--------|-------------|----------|
| Walk a known distance, compare reported position | Objective, repeatable room-scale test | |
| Qualitative in-game feel check | Subjective feel check using the demo/room scene | |

| Option | Description | Selected |
|--------|-------------|----------|
| Ship with CoreMotion-only baseline | Revert to 06.2 parity, position stays zero-stubbed | |
| Stop and re-scope | Treat failure as a hard blocker requiring investigation | ✓ |

**User's choice:** Stop and re-scope (open a new debug/research cycle)

| Option | Description | Selected |
|--------|-------------|----------|
| Sub-centimeter (~5-10mm) | Targets ARKit's typical published best-case accuracy | |
| True millimeter-precision (~1-3mm) | Tighter than typically-quoted accuracy | ✓ |
| You decide (Claude's discretion) | Research proposes a concrete tolerance | |

**User's choice:** True millimeter-precision (~1-3mm)
**Notes:** Initial tolerance question assumed room-scale walking (1-2m distances); user clarified the actual use case is small-volume motion within roughly a 1m³ cube, so the question was reframed around that volume before re-asking. User's rationale: "I think that the smartphone will be moved in a 1m3 cube volume, so the precision must be very high. mm if possible."
