## Accomplishments
- `TransportManager.disconnect()` (D-13): stops the sensor loop/heartbeat, closes the transport, clears `reconnectToken` so intentional disconnect never triggers auto-reconnect
- Hidden overlay menu in `ActiveSessionView`: `CornerLongPressOverlay` reveals a card with Recenter (D-11) and Disconnect (D-13); mounted only while connected, with proper window-recognizer teardown on disconnect
- Full-screen connected-session chrome: status bar hidden while connected, restored on Home/ended/error — required discovering that the toast system's persistent overlay window (not the main window) is the actual status-bar authority in this app
- Corner-hold zone narrowed from a flat 25%-of-height fraction to the device's real `safeAreaInsets.top` (notch/Dynamic Island), removing overlap with normal control touches near the top of the screen
- Single-finger touch signal rebuilt as a window-attached `UIGestureRecognizer` (`TouchSignalRecognizer`) after three raw-`UIView`-hosted trackers were all unreliable on-device, mirroring `CornerLongPressRecognizer`'s already-proven pattern
- Plain-text Back button for the disconnected/ended state, with the D-13 connected/disconnected predicate moved onto `SessionState` (synchronous) instead of `TransportManager` (async-reset), eliminating a stale-button-flash race at the start of every new session
- Recenter toast auto-dismisses after 1.5s via a new optional `Toast.duration` field

## Decisions Made
- `SessionState.isConnected` (not `TransportManager.isConnected`) is the single source of truth for all connected-session UI gating — closes a real race where the previous session's terminal state could still be read for a brief window at the start of a new session.
- Status bar hiding is set on the toast system's overlay-window controller (`forceStatusBarHidden` on `dynamicIslandToast(...)`), not via any mechanism applied to the main window — the overlay window is a separate, persistent, always-on-top `UIWindow` that is the actual status-bar authority in this app.
- The corner-hold gesture's touch signal moved from a raw `UIView`-hosted tracker to a window-attached `UIGestureRecognizer`, matching the one touch-tracking pattern in this codebase that has never had reliability problems.

## Issues Encountered
- The touch-signal bug took three implementation attempts to root-cause correctly: a `weak`→`strong` reference fix (real, but incomplete), a `Set<UITouch>`-membership rewrite (logically sound but still unreliable on-device), a plain `Int` counter (simplified per user request, still unreliable), and finally a window-attached `UIGestureRecognizer` — the pattern already proven reliable by `CornerLongPressRecognizer` in this same codebase. The lesson: a raw `UIView` hosted via `UIViewRepresentable` was the unreliable variable across all three failed attempts, not the counting/tracking logic layered on top of it.
- Status bar hiding required two attempts: a child-controller-forwarding `UIViewControllerRepresentable` trick (didn't work on-device), then SwiftUI's native `.statusBar(hidden:)` (also didn't work) — both failed for the same underlying reason: this app's toast system creates a persistent overlay window that sits above the main window and independently controls the real, visible status bar. The fix had to route through that overlay window's own controller instead.
