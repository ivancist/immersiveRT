- **D-10:** The ARKit world origin auto-resets **after a hold-still calibration step**, mirroring the web client's `runCalibration`-before-sensor-pipeline convention (`client/src/sensor/encode.ts`) — not immediately at pair time.
- **D-11:** A **phone-side manual recenter control** is added **in addition to** the existing desktop-side R-key reset (`resetAllPlayerPositions()` in `client/src/scene.ts`). - **D-12:** Recenter and a new Disconnect/Back control **share one hidden reveal gesture**: a **2-finger long-press, one finger held in each top corner**. This reveals a small overlay menu containing both buttons.
- **D-13:** Disconnect/Back button behavior: if connected, **disconnects** the session; if already disconnected or in an error state, **navigates back to the first/initial screen**.

## Summary

This phase swaps the 06.2 CoreMotion-orientation-only baseline for ARKit `ARSession` world tracking, which supplies **both** position and orientation from one VIO (visual-inertial odometry) pose per frame (`ARCamera.transform`). The integration shape is straightforward: instantiate a raw `ARSession` (no `ARSCNView`/`ARView` needed — this app never renders camera passthrough, per D-14), run it with `ARWorldTrackingConfiguration`, and read `ARFrame`/`ARCamera` via `ARSessionDelegate` callbacks (which fire on the **main thread by default** unless a `delegateQueue` is set). `ARCamera.transform` is a `simd_float4x4` — position is the translation column, and orientation converts to a quaternion with **one call**: `simd_quaternion(camera.transform)`.

The two research-flagged decisions turned out to have real substance, not just process caution:

1. **D-02 (axis convention):** ARKit's coordinate frame (right-handed, Y-up-via-gravity, camera looks down local -Z) is architecturally *different* from both W3C `DeviceOrientationEvent`'s Earth frame (which `scene.ts` already has a documented conjugate-rotation conversion for) and CoreMotion's `.xArbitraryZVertical` attitude frame (which 06.2's `CoreMotionSource.swift` explicitly left **unverified pending on-device testing**). Critically, independent evidence found during this research shows `ARCamera.transform`'s rotation is anchored to a fixed physical-sensor reference (historically documented as "landscape-right"), not auto-corrected for the app's software interface-orientation lock. This means the CoreMotion mapping this project already has cannot be reused or assumed compatible — a **new, isolated, easily-patchable conversion function is required**, and it must go through the same on-device verification discipline that caught two prior axis bugs (`260710-w83`, the huge-position-drift debug cycle).
2. **D-17 (mm tolerance):** Every independent accuracy source found — including a peer-reviewed 200+-environment study (arXiv 2109.14757) and an industrial benchmark using submillimeter optical ground truth (Feigl et al.) — reports ARKit position drift/error in the **centimeter-to-meter** range, never millimeters, even for the simplest tested actions. No Apple source publishes a mm-level accuracy claim. This is strong evidence (not proof, since no source tested this project's exact "small ~1m³ handheld volume, mostly rotation + short translation" regime) that the **1–3mm quantitative target is at serious risk of failing D-16's checkpoint**, which is precisely the scenario D-18's "stop and re-scope, do not silently degrade" rule anticipates. The planner should treat the go/no-go checkpoint as genuinely uncertain in outcome, not a formality.

**Primary recommendation:** Build the ARKit integration with the axis conversion isolated in one small, well-named function (mirroring `phone.ts`'s `rotateDeviceToWorld` / `scene.ts`'s local axis-mapping block) so it can be corrected without touching unrelated code, exactly as happened twice already in this project. Run the D-16 quantitative checkpoint **early** in the plan sequence (not last) — if D-17's tolerance fails, the rest of the phase's UX polish work (recenter, hidden gesture, touch capture) may need to be re-scoped around a different position-accuracy expectation before it's worth building.

### Pattern 1: Headless `ARSession` (no `ARSCNView`/`ARView`)

**What:** ARKit does not require a rendering view to run world tracking. Instantiate `ARSession` directly, set its `delegate`, and call `session.run(ARWorldTrackingConfiguration())`. The `ARSessionDelegate.session(_:didUpdate:)` callback still fires every frame with a full `ARFrame` (including `camera.transform` and `camera.trackingState`) with zero rendering surface involved. `[CITED: developer.apple.com/forums/thread/113797, general ARKit architecture]`

**When to use:** Exactly this phase's case — D-14 forbids any passthrough view, so a rendering surface would be pure waste.

**Example:**
```swift
// Source: pattern synthesized from Apple ARSession/ARSessionDelegate docs — verify signature on-device
final class ARPoseSource: NSObject, ARSessionDelegate {
    private let session = ARSession()
    var onPose: ((simd_float4x4, ARCamera.TrackingState) -> Void)?

    func start() {
        guard ARWorldTrackingConfiguration.isSupported else { return }
        let config = ARWorldTrackingConfiguration()
        // config.worldAlignment defaults to .gravity — do not override without a reason (Pitfall 2)
        session.delegate = self
        // session.delegateQueue = <background queue>  // optional — see Pitfall 3 re: MainActor
        session.run(config)
    }

    func stop() { session.pause() }

    func recenter() {
        // D-10/D-11: re-anchor world origin WITHOUT a full session restart (see Pattern 2)
        session.setWorldOrigin(relativeTransform: matrix_identity_float4x4)
    }

    // ARSessionDelegate
    func session(_ session: ARSession, didUpdate frame: ARFrame) {
        onPose?(frame.camera.transform, frame.camera.trackingState)
    }
}
```

### Pattern 2: Recenter via `setWorldOrigin(relativeTransform:)`, not a full `resetTracking` restart

**What:** `ARSession.RunOptions` includes `.resetTracking` and `.resetSceneReconstruction`, passed to `session.run(config, options:)` — but this **restarts the SLAM pipeline**, briefly re-entering `.limited(.initializing)`/`.limited(.relocalizing)` tracking state. `ARSession.setWorldOrigin(relativeTransform:)` instead redefines the coordinate space's origin/basis relative to the **current** tracking state, without interrupting tracking. `[CITED: developer.apple.com/documentation/arkit/arsession/2942278-setworldorigin]`

**When to use:** Both D-10 (auto-reset after hold-still calibration) and D-11 (manual phone-side recenter) — neither should cause a visible tracking-loss blip, so prefer `setWorldOrigin` over `.resetTracking` for both. Reserve a full `session.run(config, options: [.resetTracking])` for genuine recovery scenarios (e.g., tracking is `.notAvailable` and needs a hard restart), not routine recentering.

**Example:**
```swift
// Source: Apple docs synthesis — verify exact matrix construction on-device
func recenterToCurrentPose(_ session: ARSession) {
    // Passing the identity transform sets the origin to the device's CURRENT pose —
    // i.e. "wherever the phone is right now becomes (0,0,0)".
    session.setWorldOrigin(relativeTransform: matrix_identity_float4x4)
}
```

### Pattern 3: Corner-anchored two-finger long-press (D-12)

**What:** SwiftUI's built-in gesture composition (`SimultaneousGesture`, `.gesture()`, `LongPressGesture`) has no primitive for "N touches, each constrained to a specific screen region, held for a duration." The reliable approach is a **custom `UIGestureRecognizer` subclass** wrapped in `UIViewRepresentable`, overriding `touchesBegan`/`touchesMoved`/`touchesEnded`/`touchesCancelled` to track each `UITouch`'s `location(in:)` independently, checking each against its own corner rect, and using a `DispatchWorkItem`/`Timer` armed only once both corners have an active touch (cancelled on early lift-off or excessive movement). `[ASSUMED — synthesized from general iOS UIGestureRecognizer/UIViewRepresentable patterns; no single authoritative source covers this exact compound gesture, so this is a recommended approach, not a verified one]`

**When to use:** D-12's hidden reveal gesture, layered as an overlay `UIViewRepresentable` on top of `ActiveSessionView`'s full-screen touch-capture surface (D-04) — must NOT consume/block the normal single-finger tap+drag touch capture, only observe alongside it.

**Example:**
```swift
// Source: [ASSUMED] pattern — standard UIGestureRecognizer subclass shape
final class CornerLongPressRecognizer: UIGestureRecognizer {
    var onReveal: (() -> Void)?
    private var cornerTouches: [ObjectIdentifier: (touch: UITouch, corner: Corner)] = [:]
    private var armTimer: Timer?
    private let holdDuration: TimeInterval = 0.6 // Claude's discretion per CONTEXT.md

    enum Corner { case topLeft, topRight }

    override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent) {
        guard let view else { return }
        for touch in touches {
            let pt = touch.location(in: view)
            if let corner = corner(for: pt, in: view.bounds), !occupied(corner) {
                cornerTouches[ObjectIdentifier(touch)] = (touch, corner)
            }
        }
        armIfBothCornersHeld()
    }

    override func touchesMoved(_ touches: Set<UITouch>, with event: UIEvent) {
        // Cancel if a tracked touch drifts out of its corner region beyond tolerance
    }

    override func touchesEnded(_ touches: Set<UITouch>, with event: UIEvent) { reset() }
    override func touchesCancelled(_ touches: Set<UITouch>, with event: UIEvent) { reset() }

    private func armIfBothCornersHeld() {
        guard cornerTouches.values.contains(where: { $0.corner == .topLeft }),
              cornerTouches.values.contains(where: { $0.corner == .topRight }) else { return }
        armTimer?.invalidate()
        armTimer = Timer.scheduledTimer(withTimeInterval: holdDuration, repeats: false) { [weak self] _ in
            self?.onReveal?()
        }
    }

    private func corner(for pt: CGPoint, in bounds: CGRect) -> Corner? { /* top-left/top-right hit-test regions */ nil }
    private func occupied(_ corner: Corner) -> Bool { cornerTouches.values.contains { $0.corner == corner } }
    private func reset() { cornerTouches.removeAll(); armTimer?.invalidate(); armTimer = nil }
}
```

### Pattern 4: Session-start capability + permission gate (D-09)

**What:** Check `ARWorldTrackingConfiguration.isSupported` (static `Bool`, device-capability check — no LiDAR/A12+ requirement for basic world tracking, but older/unsupported devices return `false`) AND camera permission via `AVCaptureDevice.authorizationStatus(for: .video)` BEFORE calling `session.run(...)`. `[CITED: developer.apple.com/documentation/avfoundation/avcapturedevice/1624613-authorizationstatus]`

**Example:**
```swift
// Source: Apple AVFoundation + ARKit docs synthesis
enum ARStartupError: Error { case deviceUnsupported, cameraDenied, cameraRestricted }

func checkARStartupPreconditions() async -> ARStartupError? {
    guard ARWorldTrackingConfiguration.isSupported else { return .deviceUnsupported }

    switch AVCaptureDevice.authorizationStatus(for: .video) {
    case .authorized:
        return nil
    case .notDetermined:
        let granted = await AVCaptureDevice.requestAccess(for: .video)
        return granted ? nil : .cameraDenied
    case .denied:
        return .cameraDenied
    case .restricted:
        return .cameraRestricted
    @unknown default:
        return .cameraDenied
    }
}
```
D-09 requires this to **block session start with a DynamicToast error**, not fall back to CoreMotion-only. NSCameraUsageDescription must already exist in Info.plist (verify during planning — ARKit implicitly requires camera access, so this string may already be needed for `QRScannerView.swift`'s existing camera use; confirm it's present and its copy makes sense for both QR-scan AND ARKit tracking).

### Anti-Patterns to Avoid

- **Running a `Timer`-based poll loop for ARKit pose reads:** ARKit already delivers pose data via a push-based delegate callback at its own frame rate (~60Hz on supported devices) — do not add a redundant polling `Timer` on top of it. This mirrors why 06.2's sensor loop is *not* actually timer-driven (see Pitfall below).
- **Re-deriving CoreMotion's axis mapping and assuming it transfers to ARKit:** CoreMotion's `.xArbitraryZVertical` attitude frame and ARKit's VIO camera-pose frame are constructed by entirely different subsystems with different reference conventions — do not copy `CoreMotionSource.swift`'s (already-unverified) quaternion field-order mapping onto `ARPoseSource`.
- **Calling `session.run(config, options: [.resetTracking])` for routine recentering:** causes a visible tracking interruption; use `setWorldOrigin(relativeTransform:)` instead (Pattern 2).
- **Treating `ARCamera.transform`'s rotation as auto-corrected for the app's UI orientation lock:** independent evidence (Apple Developer Forums thread on `ARCamera` intrinsics/UI orientation) indicates the camera-local axis reference is tied to the physical sensor housing, not the current interface orientation — this must be accounted for explicitly in the conversion function, not assumed away.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Quaternion extraction from a 4x4 pose matrix | Manual 3x3 submatrix → quaternion math | `simd_quaternion(camera.transform)` | One-line stdlib call, avoids sign/normalization bugs — `simd` is Apple's own vetted math library |
| World-origin recentering | Manually offset every subsequent pose sample in app code | `ARSession.setWorldOrigin(relativeTransform:)` | ARKit already solves this natively without a session restart; hand-rolled offset tracking would duplicate what ARKit does internally and risks drift between the two bookkeeping systems |
| Camera/motion permission state machine | Custom permission-polling logic | `AVCaptureDevice.authorizationStatus(for:)` + `requestAccess(for:)` | Standard, well-tested system API; already used implicitly by `QRScannerView.swift` for camera access |
| Tracking-quality signal | Inventing a custom heuristic (e.g. feature-count thresholds) for "is tracking good" | `ARCamera.trackingState` (`.normal`/`.limited(reason:)`/`.notAvailable`) | ARKit's own internal SLAM confidence signal — reinventing this from raw frame data would be strictly worse and non-standard |

**Key insight:** ARKit already solves the three hardest sub-problems this phase touches (pose-to-quaternion math, origin recentering, tracking-quality signaling) via small, direct API calls — the actual engineering risk in this phase is entirely in the **axis-convention conversion** and the **mm-tolerance accuracy question**, both of which no library can solve; they require empirical on-device verification per D-16.

## Common Pitfalls

### Pitfall 1: No `pollTimer` exists in the 60Hz sensor loop — the phase description's assumption is wrong

**What goes wrong:** The task brief describes integrating with "the existing 60Hz `pollTimer` encode loop in `ActiveSessionView.swift`." This does not exist. `ActiveSessionView.swift`'s only `pollTimer` is `SessionViewModel`'s **0.25s throttled UI-state poll** (connection status text/symbol), explicitly documented as "well below the 60Hz sensor rate" and NOT the sensor path.

**Why it happens:** The real 60Hz sensor loop is **callback-driven**, not timer-driven: `CoreMotionSource.onOrientation` fires per CoreMotion sample (background queue) → `TransportManager.handleOrientation(_:)` (hopped to `@MainActor` via `Task { @MainActor in ... }`) → encode → send. There is no `Timer` anywhere in this path.

**How to avoid:** `ARPoseSource` should follow the exact same shape as `CoreMotionSource`: a delegate/closure callback (`onPose`) that `TransportManager` (or its extended equivalent) wires up once at `startSensorLoopIfNeeded()`-time, hopping to `@MainActor` per sample exactly like `handleOrientation(_:)` already does. Do not introduce a `Timer`-based poll loop for ARKit reads — `ARSessionDelegate.session(_:didUpdate:)` already fires once per ARKit frame.

**Warning signs:** A plan that says "add ARKit reads inside `pollTimer`'s closure" is planning against code that doesn't exist and will need correction during execution.

### Pitfall 2: `ARSessionDelegate` callbacks fire on the main thread by default — conflicts with the project's "never touch shared state off dedicated queue" convention

**What goes wrong:** Unlike `CoreMotionSource`'s explicit dedicated background `OperationQueue`, `ARSession`'s delegate callbacks default to the **main queue** unless `session.delegateQueue` is explicitly set otherwise. `[CITED: Apple Developer Forums — "ARSessionDelegate methods are by default called on the main queue... delegateQueue property... if not specified, defaults to the main thread"]`

**Why it happens:** This is architecturally different from CoreMotion, which always requires you to supply a queue.

**How to avoid:** Given this project's `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` build setting, letting ARKit callbacks land on the main thread by default is actually **compatible** with the project's `@MainActor` convention (unlike CoreMotion, which explicitly avoids the main queue to prevent blocking UI at 60Hz sensor rates). Decide explicitly during planning whether to leave `delegateQueue` unset (simplest, main-thread-native, consistent with `TransportManager`'s existing `@MainActor` hop pattern) or set a dedicated background queue (mirrors `CoreMotionSource`'s pattern but adds a queue hop `ARSessionDelegate → @MainActor` and a documented added-latency tradeoff per Apple's own note). Document whichever choice is made — do not assume it silently mirrors CoreMotion's pattern without an explicit decision.

### Pitfall 3: Axis-convention risk is real, not process paranoia — evidence from this project's own history AND from Apple's own forum guidance

**What goes wrong:** Assuming ARKit's `camera.transform` orientation can be dropped into the wire's `qw/qx/qy/qz` fields the same way CoreMotion's or DeviceOrientationEvent's quaternion was (or worse, assuming it's "probably fine because it's still a quaternion").

**Why it happens:** Three independent facts compound here: (1) `scene.ts` already has a documented, non-obvious conjugate-rotation conversion for W3C `DeviceOrientationEvent`'s Earth frame; (2) `CoreMotionSource.swift`'s CoreMotion→packet mapping is *still marked unverified* pending an on-device checkpoint that hasn't run yet; (3) an Apple Developer Forums thread on `ARCamera` intrinsics indicates the camera-local rotation reference is tied to the device's physical sensor housing orientation (historically documented relative to landscape-right), not auto-corrected for whatever interface orientation the app's UI is locked to. `[CITED: developer.apple.com/forums/thread/89594]` None of these three conventions are guaranteed compatible with each other.

**How to avoid:** Do not attempt to derive the "correct" final axis-mapping constants analytically before the D-16 checkpoint — this project has twice shipped an analytically-reasoned axis fix that was wrong on-device (`260710-w83`; the `rotateDeviceToWorld` conjugate/direct-rotation reversal in the huge-position-drift debug cycle). Instead: (a) isolate the ARKit→packet conversion in one small, named function; (b) ship a reasoned starting-point mapping (document it as unverified, exactly like `CoreMotionSource.swift` already does); (c) make the D-16 on-device checkpoint's qualitative pass explicitly include a single-axis-at-a-time motion test (move phone straight up only, straight forward only, etc.) BEFORE the quantitative mm-tolerance test, using the existing numeric HUD (`scene.ts`'s `q:`/`pos:`/`drift:` per-player text) as the feedback loop — this is the fastest way to catch a sign/axis-swap bug, as it was in every prior incident on this project.

**Warning signs:** Any plan step that hard-codes final axis-negation/reorder constants without a code comment saying "UNVERIFIED — pending Plan N's on-device checkpoint" is repeating the exact mistake `260710-w83` and the huge-position-drift debug cycle already made twice.

### Pitfall 4: The 1–3mm tolerance (D-17) has no supporting evidence in any accuracy benchmark found

**What goes wrong:** Planning the checkpoint (D-16) as a formality/final-polish step, when independent evidence suggests real risk of failure.

**Why it happens:** Every accuracy source found in this research — `[CITED: arxiv.org/abs/2109.14757]` ("Here To Stay," 200+ environments, 3 phone models: 2.9cm–25cm drift for the *simplest* tested user action, 10cm+ for others), and `[CITED: doi.org/10.3390/s22145382]`-adjacent Feigl et al. industrial study (submillimeter optical ground truth, ~17m error per 120m path / 14.4cm/m scaling error) — measures drift and error in **centimeters to meters**, not millimeters, and none specifically tested a small (~1m³) handheld working volume with mostly-rotational, short-translation motion (this project's actual use case, which differs from both the papers' room-scale-walking and industrial-large-area conditions).

**How to avoid:** Sequence the D-16 quantitative test **early** in the plan — do not build the full UX polish (D-11/D-12/D-13 hidden-menu, D-15 toast wiring, etc.) before confirming the core premise holds. If the checkpoint fails per D-18, the re-scope decision should happen before, not after, the UX layer is built on top of a position source that may need replacing or degrading in scope.

**Warning signs:** A plan sequence that puts the D-16 checkpoint as the LAST wave/plan in the phase, after all UX work is already built — this inverts the correct risk order given the evidence above.

### Freeze-on-loss position (D-07 — direct port of `webxr.ts`'s last-known-good pattern)

```swift
// Source: pattern ported from client/src/sensor/webxr.ts's WebXrPoseTracker (freeze-on-lost, never
// switch tracking source live mid-session)
final class ARPoseTracker {
    private var lastGoodTransform: simd_float4x4 = matrix_identity_float4x4
    private(set) var driftConfidence: Double = 0

    func ingest(camera: ARCamera) {
        driftConfidence = self.driftConfidence(for: camera.trackingState)
        // Freeze: only update last-known-good on a state that still yields a usable pose.
        // ARKit still reports a transform even in .limited/.notAvailable, but per D-07 the
        // WIRE value should freeze — do not forward a degrading pose while confidence < 1.0
        // unless product decision says otherwise (confirm during planning: does "freeze" mean
        // "stop updating lastGoodTransform on ANY non-.normal state" or only on .notAvailable?
        // webxr.ts freezes only on a NULL pose (driftConfidence 0), not on .limited (0.5) —
        // mirror that: update through .limited, freeze only on .notAvailable).
        if camera.trackingState != .notAvailable {
            lastGoodTransform = camera.transform
        }
    }
}
```

**Open sub-question captured for the planner:** `webxr.ts`'s freeze behavior is keyed on a *null pose* (driftConfidence 0 only), and still updates position through the `.limited`/emulatedPosition (driftConfidence 0.5) state. D-07 says "position freezes at last-known-good on loss" — confirm during planning whether "loss" means only `.notAvailable` (driftConfidence 0, matching `webxr.ts`'s exact freeze condition) or also `.limited` (driftConfidence 0.5). The `webxr.ts` porting guide implies freeze-only-on-zero-confidence is the precedent to follow.

## Open Questions

1. **Does D-07's "freeze at last-known-good on loss" apply only to `.notAvailable`, or also to `.limited`?**
   - What we know: `webxr.ts`'s precedent (the explicit porting guide named in CONTEXT.md's canonical refs) freezes only on a null pose (driftConfidence 0), and continues updating position through the `.limited`/emulatedPosition state (driftConfidence 0.5).
   - What's unclear: D-07's phrasing ("position freezes at last-known-good on loss") doesn't specify which confidence tier counts as "loss."
   - Recommendation: Follow the `webxr.ts` precedent (freeze only on driftConfidence 0 / `.notAvailable`) unless the planner or a future discuss-phase session decides otherwise — flag this explicitly in the plan so it's a deliberate choice, not an oversight.

2. **Is `CoreMotionSource.swift` deleted, kept-dormant, or repurposed in this phase?**
   - What we know: D-01 says ARKit "supersedes" it; D-09 (from 06.2, carried forward) forbids using it as a live fallback.
   - What's unclear: Whether the file/class itself should be physically removed from the target, or just have its call sites in `TransportManager.swift` swapped out (leaving the class present but unused, e.g. for future reference or a possible future non-ARKit-device code path).
   - Recommendation: Planner should make an explicit decision here — likely delete `CoreMotionSource`'s wiring from `TransportManager` but consider whether to keep the file itself (unused) given the amount of doc-comment history it carries about the still-unverified axis mapping, which may be useful context if ARKit's checkpoint fails and IMU-only ever needs revisiting per D-18.

3. **What exact hold-still detection signal drives D-10's auto-reset?**
   - What we know: CONTEXT.md says it should mirror `runCalibration`'s hold-still convention from `client/src/sensor/encode.ts`, and the huge-position-drift debug history shows the web client's ZUPT/hold-still detection had a documented signal-choice bug (accelerationIncludingGravity vs. linear acceleration) that took two debug passes to resolve.
   - What's unclear: Whether ARKit itself provides a usable "device is still" signal (e.g. via `ARCamera.transform` frame-to-frame delta, since ARKit already does its own internal stationary-detection for tracking quality) or whether CoreMotion's accelerometer should be sampled in parallel purely for stillness detection (NOT for orientation, respecting D-01's "no parallel CoreMotion orientation pipeline" — a stillness detector reading raw acceleration magnitude is a different concern than an orientation pipeline, so this may not conflict with D-01, but the planner should confirm this framing explicitly).
   - Recommendation: Prefer deriving stillness from `ARCamera.transform`'s own frame-to-frame position delta (avoids the accelerationIncludingGravity-vs-linear-acceleration bug class entirely, since it's a smoothed VIO pose rather than raw noisy accelerometer data) unless research/planning finds a reason ARKit's pose isn't responsive enough for this purpose.
