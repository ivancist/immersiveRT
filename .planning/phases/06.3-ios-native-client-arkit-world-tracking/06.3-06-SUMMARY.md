# Phase 06.3 Plan 06: ARKit Availability Gate + Tracking Feedback Summary

**D-09 hard-block gate (reachable at both real camera-use entry points) and debounced D-08 local tracking-limited toasts, refined through three rounds of on-device hardware feedback**

## Performance

- **Duration:** ~2h across 3 on-device iteration rounds (initial build, QR-scan gate gap, toast-accumulation debounce)
- **Completed:** 2026-07-15
- **Tasks:** 3/3 (2 auto + 1 human-verify checkpoint, iterated 3 times)
- **Files modified:** 6 across 4 commits, plus 2 new files

## Accomplishments
- Added `Toast.arUnavailable`/`Toast.cameraPermissionDenied` (D-09) and a per-reason `Toast.trackingLimited(_:)` factory (D-08), plus accurate `NSCameraUsageDescription` copy covering both QR scanning and ARKit motion tracking
- Wired a hard session-start precondition gate — no CoreMotion fallback exists in this app at all, so denial/unsupported blocks the session entirely with a toast (D-09/D-18 no-silent-degrade)
- **Fixed a real reachability gap found on-device:** the D-09 gate only lived at ARKit-session-start, but `QRScannerView` (the actual first camera use in the real user flow) had zero permission handling of its own and showed a black screen on denial — the gate now also runs at the "Scan QR" button tap
- **Fixed a real UX bug found on-device:** tracking-limited toasts re-triggered on every ARKit classification flicker (strict equality dedup); added `TrackingLimitedMessageDebouncer` (500ms min continuous dwell, pure/testable, mirrors `HoldStillDetector`'s style) to stabilize this
- Confirmed on-device that wire `driftConfidence` stays flat 0.5 for any `.limited` reason throughout all of the above changes

## Decisions Made
- D-09 check reused (not duplicated) at both entry points via the same `ARPoseSource.checkARStartupPreconditions()` function
- Debounce dwell set to 500ms (Claude's Discretion, ~15-30 ARKit frames) — long enough to suppress flicker, short enough not to swallow genuinely sustained states
- Kept per-sub-reason toast specificity per explicit user preference, even knowing violent-motion-then-stop can still occasionally produce two sequential distinct toasts as classification settles — judged an acceptable, honest reflection of real changing conditions rather than a bug

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] QR-scan entry point had no D-09 gate**
- **Found during:** On-device Task 3 verification (round 1)
- **Issue:** The plan scoped D-09 only to ARKit-session-start (`ActiveSessionView`), but `QRScannerView` — the actual first real camera use — had no permission handling and silently showed a black screen on denial, making the gate unreachable in practice
- **Fix:** `ContentView`'s "Scan QR" tap now runs the same precondition check before presenting the scanner sheet
- **Files modified:** `ContentView.swift`
- **Verification:** On-device retest confirmed toast now appears immediately, no black screen
- **Committed in:** `5c012eb`

**2. [Rule 2 - Missing Critical] Tracking-limited toasts lacked flicker debouncing**
- **Found during:** On-device Task 3 verification (round 1-2)
- **Issue:** Strict equality-only dedup re-fired the local toast on every ARKit classification flicker, reading as "accumulating" toasts including repeats of the same type
- **Fix:** Added `TrackingLimitedMessageDebouncer` (500ms min continuous dwell) between `ARPoseSource`'s per-frame classification and the local toast callback
- **Files modified:** `ARPoseSource.swift`, new `TrackingLimitedMessageDebouncer.swift` + tests
- **Verification:** 7 new unit tests plus on-device retest confirmed substantially reduced accumulation
- **Committed in:** `b0b64ca`

## Issues Encountered
- One SendMessage-continuation of a completed executor agent (whose worktree had already been merged/removed) bypassed the worktree isolation guard and committed fix #1 directly to `main` instead of halting for re-provisioning, as its own instructions required. No harm resulted (verified build/test green, no concurrent work in flight), but subsequent fixes were dispatched via fresh `Agent()` calls with an explicit no-bypass instruction to avoid recurrence.
- Residual UX characteristic (not fixed): during violent motion followed by an abrupt stop, ARKit's classification can settle through two different stable `.limited` reasons in quick succession, producing two sequential (not stacked) toasts. User explicitly chose to keep per-reason specificity and accept this rather than collapse to a single generic message.
