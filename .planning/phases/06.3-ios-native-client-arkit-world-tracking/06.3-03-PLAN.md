<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 1: On-device single-axis convention verification (D-02) — MUST run before quantitative test</name>
  <action>Run the single-axis motion verification procedure below on a physical iPhone against the desktop numeric HUD, iterating ONLY the isolated `ARPoseConversion.swift` mapping until every axis/sign/rotation is correct (D-02). Manual-only — ARKit does not run in the Simulator.</action>
  <what-built>
    ARPoseSource (Plan 02) streams real ARKit 6DOF pose through the unchanged 36-byte schema to the desktop, which renders it via the existing Phase 6 decode/SLERP pipeline and shows the per-player numeric HUD (`q:` / `pos:` / `drift:` lines in `client/src/scene.ts`).
  </what-built>
  <how-to-verify>
    1. Build and run the iOS app on a physical iPhone (ARKit does NOT run in the Simulator). Confirm the exact device model in the checkpoint record (LiDAR models may materially affect D-17 outcome per RESEARCH.md Environment Availability).
    2. Start the desktop room client and join a room; scan the QR from the phone to pair. Grant camera permission when prompted.
    3. With the desktop numeric HUD visible (`pos:` and `q:` lines), move the phone along ONE axis at a time and record HUD response:
       - Straight UP (~30cm), watch which `pos:` component changes and its sign.
       - Straight FORWARD (~30cm), same.
       - Straight RIGHT (~30cm), same.
       - Pure yaw / pure pitch / pure roll rotations, watch `q:`.
    4. For each: note whether the axis, sign, and rotation direction match the expected desktop convention (i.e. moving the phone up moves the rendered object up, not down/sideways).
    5. If any axis is inverted or swapped: fix ONLY the isolated mapping in `ARPoseConversion.swift` (the UNVERIFIED starting-point mapping from Plan 01), rebuild, and re-test — do NOT touch decode.ts (desktop is a byte-format passthrough with zero axis logic). Record the final corrected mapping.
  </how-to-verify>
  <resume-signal>Type "axis-verified" once every single-axis motion and rotation maps correctly on the HUD (record the final mapping in 06.3-CHECKPOINT.md), or describe the mismatch to iterate.</resume-signal>
</task>

  <action>Execute the measured-point accuracy protocol below on-device, recording ARKit-reported position vs. ground truth per point and summarizing error against the ~1-3mm target (D-16a/D-17). Manual-only.</action>
  <what-built>
    With the axis convention verified (Task 1), the desktop `pos:` HUD reports ARKit-derived position in a known unit frame, usable as the measurement readout against physical ground truth.
  </what-built>
  <how-to-verify>
    1. Define a measured-point protocol within a ~1m³ working volume (Claude's Discretion per CONTEXT.md — e.g. a taped grid or ruler rig with marked points at 0, 25cm, 50cm along each of X/Y/Z, plus a few diagonal points). Record the exact protocol used.
    2. Move the phone to each marked ground-truth point, hold still, and record the ARKit-reported position from the desktop `pos:` HUD.
    3. Compute per-point error (reported vs. ground truth) and summarize (min/max/mean error per axis, and worst-case).
    4. Compare against the D-17 target of ~1-3mm. RESEARCH.md Pitfall 4 sets the expectation that error will likely land in the centimeter range — record actual numbers regardless.
  </how-to-verify>
  <resume-signal>Type "quant-recorded" with the measured error summary written into 06.3-CHECKPOINT.md.</resume-signal>
