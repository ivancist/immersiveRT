# Phase 06.3 Plan 07: Corner Long-Press Gesture Recognizer Summary

**Custom `UIGestureRecognizer` subclass detects a hidden 2-finger both-top-corners hold via UIWindow-level touch observation, with a pure unit-tested hit-test function underneath and zero interference with the existing single-finger touch-capture surface.**

## Accomplishments
- `corner(for:in:)` is a pure, static, unit-tested function (`Corner?` in/out, no touch runtime) classifying a point against a landscape-locked bounds' top-left/top-right corner regions (top 25% height, left/right 30% width bands)
- `CornerLongPressRecognizer` (`UIGestureRecognizer` subclass) tracks each `UITouch` independently, arms a 0.6s hold timer only once both `.topLeft` and `.topRight` corners are simultaneously held, and cancels on early lift-off, excessive drift, or `touchesCancelled`
- `CornerLongPressOverlay` (`UIViewRepresentable`) correctly attaches the recognizer to the enclosing `UIWindow` (not a ZStack-sibling marker view) so it observes every touch on screen without ever winning hit-testing or blocking `ActiveSessionView`'s existing full-screen touch-capture surface (D-04/D-12)

## Decisions Made
- Corner region fractions: top 25% height, left/right 30% width (documented in source doc comments) — matches RESEARCH.md Pattern 3's suggested example fractions
- `holdDuration` default 0.6s and `driftTolerance` 24pt — both Claude's Discretion per CONTEXT.md, chosen to be forgiving of natural finger micro-movement while staying intentional enough to be very hard to trigger by accident
- **Architectural adjustment (Rule 2 — missing critical functionality):** RESEARCH.md's Pattern 3 example and the plan's action both describe wrapping the recognizer in a `UIViewRepresentable` "overlay" without addressing how a ZStack-sibling view's gesture recognizer would ever receive touches destined for a different sibling view (the actual touch-capture surface). Implemented as-literally-described, the recognizer would never fire in practice — a `UIGestureRecognizer` only receives touches for the ancestor chain of whichever view UIKit hit-tests, and a plain sibling overlay view either intercepts nothing (if its own hit-test returns false) or has to win hit-testing and thereby block the surface underneath (violating D-04/D-12's core "never block" requirement) if its hit-test returns true. Resolved by attaching the recognizer to the enclosing `UIWindow` instead of the marker view itself — the window is the one common ancestor of every on-screen view, so the recognizer observes every touch with zero hit-test interference. This is a same-file addition with no new external dependency or schema change, so it falls under Rule 2 rather than Rule 4.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] UIWindow-attachment instead of marker-view attachment for the SwiftUI overlay wrapper**
- **Found during:** Task 2 (SwiftUI wrapper implementation)
- **Issue:** A `UIViewRepresentable` overlay hosted as a ZStack sibling of `ActiveSessionView`'s touch-capture surface cannot have its own attached gesture recognizer observe touches destined for that sibling surface — `UIGestureRecognizer` delivery only traverses the ancestor chain of the hit-tested view, never unrelated siblings. As literally described in RESEARCH.md Pattern 3 / the plan's action text, the recognizer would either never fire, or (if the marker view's own hit-test intercepted touches) would block the primary touch-capture surface, violating the plan's explicit "never consume/block" requirement (D-04/D-12).
- **Fix:** `CornerLongPressOverlay.Coordinator.attach(to:holdDuration:)` walks to `view.window` (deferred one runloop tick via `DispatchQueue.main.async`, since `window` is nil at `makeUIView` time) and calls `window.addGestureRecognizer(recognizer)` instead of attaching to the marker view. The marker view itself is a zero-size, `isUserInteractionEnabled = false`, hidden placeholder used only to reach the window.
- **Files modified:** `mobile/ios-app/immersiveRT/Gestures/CornerLongPressRecognizer.swift`
- **Verification:** `xcodebuild build` succeeds; source assertions confirm `cancelsTouchesInView = false` and both-corners gating are present as required by the plan's acceptance criteria. Full on-device gesture reliability (including confirming the window-attachment approach behaves correctly with two real simultaneous touches) is deferred to Plan 08's on-device checkpoint per this plan's own `<verification>` section (Simulator cannot reliably simulate two sustained independent touches).
- **Committed in:** `2239eff` (Task 2 commit)
