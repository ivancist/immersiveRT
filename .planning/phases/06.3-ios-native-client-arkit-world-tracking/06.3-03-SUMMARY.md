# Phase 06.3 Plan 03: On-Device ARKit Go/No-Go Checkpoint Summary

**GO decision on real iPhone 15 Pro hardware after two rounds of on-device-verified axis fixes (quaternion swizzle + position swizzle), both isolated to `ARPoseConversion.swift`**

## Performance

- **Duration:** ~2h across the full interactive checkpoint (setup, 2 fix iterations, quantitative + qualitative passes)
- **Completed:** 2026-07-15
- **Tasks:** 3/3 (all human-verify checkpoints)
- **Files modified:** 2 (`ARPoseConversion.swift`, `ARPoseConversionTests.swift`) across 2 commits, plus 1 new doc (`06.3-CHECKPOINT.md`)

## Accomplishments
- Diagnosed and fixed a rotation axis-coupling bug (roll/yaw swapped) caused by desktop `scene.ts` applying a W3C-DeviceOrientationEvent-specific quaternion swizzle to ARKit's already-compatible quaternion
- Diagnosed and fixed a position axis bug (up→backward, right→left, forward→up) caused by the same class of mismatch in `scene.ts`'s device-frame position swizzle vs. ARKit's world-frame position
- Identified a desktop-side (non-iOS) `positionMode` display toggle issue (`'gesture'` default reads the wrong wire fields) — no code fix needed, user-facing toggle
- Ran an informal quantitative accuracy pass (20cm range, 5cm intervals) — "very accurate," better than RESEARCH.md's pessimistic centimeter-scale expectation
- Ran a qualitative in-game feel pass and recorded a **GO** decision, with two observed characteristics explicitly identified as expected ARKit behavior (not bugs) that Plan 05 will directly address

## Files Created/Modified
- `mobile/ios-app/immersiveRT/Sensor/ARPoseConversion.swift` — `arKitPacketQuaternion` and `arKitPacketPosition` updated from UNVERIFIED straight pass-through to on-device-VERIFIED compensating swizzles
- `mobile/ios-app/immersiveRTTests/ARPoseConversionTests.swift` — expected values recomputed by hand for the new mappings; one new regression test added (`test_arKitPacketQuaternion_swapsAndNegatesYZForRotationWithNonzeroYZComponents`) that actually discriminates the old pass-through from the new mapping
- `.planning/phases/06.3-ios-native-client-arkit-world-tracking/06.3-CHECKPOINT.md` — new: full record of both axis fixes, accuracy pass, feel-pass notes, and GO decision

## Decisions Made
- Both axis fixes were derived by treating `scene.ts`'s existing render-side swizzles as fixed/immutable (per RESEARCH.md's mandated isolated seam) and solving backward for the `ARPoseConversion.swift` mapping that cancels them out — never touched `scene.ts`/`decode.ts`
- Each fix was independently re-derived twice (once analytically by the orchestrator, once by the executor agent reading the live code) before being applied, and validated against real on-device symptom reports rather than committed on analysis alone
- Desktop `positionMode` default (`'gesture'`) was left unchanged — out of scope, and touching it could affect other client paths (WebXR/Android gesture clients)

## Deviations from Plan

None beyond the plan's own explicit "iterate until every axis/sign/rotation is correct" instruction — two iteration rounds were required (quaternion, then position), which is exactly what Task 1's `<how-to-verify>` anticipated ("If any axis is inverted or swapped... rebuild, and re-test").

## Issues Encountered
- A worktree assigned to an early diagnostic-only agent session was auto-cleaned up (no commits had been made yet) before the fix could be applied in that same session — recovered by re-provisioning a fresh worktree from `main` (which already had all prior work) and re-dispatching the same verified fix instructions. No work was lost.
- One flaky, unrelated `PeerConnectionManagerTests` WebRTC test failure surfaced during the position-fix test run — confirmed flaky by isolated re-run (passed) and a second full-suite run (fully green).
