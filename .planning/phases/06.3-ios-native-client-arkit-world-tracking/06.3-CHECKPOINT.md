# 06.3-03 On-Device ARKit Go/No-Go Checkpoint

## Task 1: Single-Axis Convention Verification (D-02) — ✅ COMPLETE

**Device:** iPhone 15 Pro (LiDAR)

**Method:** Desktop numeric HUD (`pos:`/`q:`/`drift:` lines), raw wire values read directly (HUD displays untransformed `state.px/py/pz`/`qw/qx/qy/qz`), rendered cube observed separately after each fix. Iterated twice against real hardware feedback before converging.

### Issue 1 — Rotation: roll/yaw swapped and coupled

**Symptom:** Roll and yaw were switched with each other; yawing also produced coupled roll rotation. Root cause: `client/src/scene.ts`'s quaternion render path applies a fixed W3C `DeviceOrientationEvent`-frame-specific conjugation (`scratchQuat.set(state.qx, state.qz, -state.qy, state.qw)`) to every incoming wire quaternion. ARKit's raw camera-transform quaternion is already in a frame structurally compatible with Three.js (right-handed, Y-up, camera facing -Z) — applying the W3C-specific swizzle on top introduced an unwanted rotation, manifesting as axis coupling.

**Fix (isolated to `ARPoseConversion.swift`, per RESEARCH.md's mandated seam — `scene.ts`/`decode.ts` untouched):**
```swift
func arKitPacketQuaternion(from transform: simd_float4x4) -> (qw: Double, qx: Double, qy: Double, qz: Double) {
    let quaternion = simd_quaternion(transform)
    return (
        qw: Double(quaternion.real),
        qx: Double(quaternion.imag.x),
        qy: Double(-quaternion.imag.z),
        qz: Double(quaternion.imag.y)
    )
}
```
Commit: `1a7b9bf` (merged via `b91a194`).

### Issue 2 — Position: stuck at 0,0,0 (desktop display mode, no code fix)

**Symptom:** Position never changed from `0,0,0` regardless of motion, even though `driftConfidence` correctly reached `1.0` on movement (confirming ARKit tracking itself was fine).

**Root cause:** Desktop `client/src/scene.ts` defaults `positionMode = 'gesture'`, which reads `state.dx/dy/dz` (WebXR/Android gesture-based dead-reckoning fields) for both the HUD and the rendered mesh — fields the iOS ARKit client never populates. Pressing the `P` key (documented in the HUD as "[P to cycle]") switches to the mode that reads `state.px/py/pz`, which is what the ARKit client populates.

**Resolution:** No code change — user-side toggle. Confirmed out of scope for this plan; `scene.ts`'s `positionMode` default/gating logic left untouched to avoid affecting other client paths (WebXR/Android gesture clients).

### Issue 3 — Position: axes wrong once visible

**Symptom (raw wire values, single-axis motions):**
| Motion | Raw wire component | Direction | Rendered cube (before fix) |
|--------|--------------------|-----------|-----------------------------|
| Lift UP | `y` increases | — | moved BACKWARD |
| Move RIGHT | `x` increases | — | moved LEFT |
| Move FORWARD | `z` decreases | — | moved UP |

**Root cause:** `scene.ts`'s position-render formula (`three.x=-wire.px, three.y=-wire.pz, three.z=wire.py`) was tuned for the original web client's device-frame dead-reckoning position semantics, not ARKit's world-frame position. Verified consistent against all 3 reported axes before deriving the fix.

**Fix (isolated to `ARPoseConversion.swift`):**
```swift
func arKitPacketPosition(from transform: simd_float4x4) -> (px: Double, py: Double, pz: Double) {
    let translation = transform.columns.3
    return (px: Double(-translation.x), py: Double(translation.z), pz: Double(-translation.y))
}
```
Commit: `66a2f5b` (merged via `35eee9a`).

### Final corrected mapping (both VERIFIED on-device, iPhone 15 Pro)

- `arKitPacketPosition`: `px = -tx, py = tz, pz = -ty` (where `tx/ty/tz` = `transform.columns.3`)
- `arKitPacketQuaternion`: `qw = q.real, qx = q.imag.x, qy = -q.imag.z, qz = q.imag.y`

**Re-test result after both fixes:** User confirmed — "It works." Lift → cube up, move right → cube right, move forward → cube moves into the scene; yaw/pitch/roll independent, no coupling.
