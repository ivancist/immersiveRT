# Phase 06.3: iOS Native Client — ARKit World Tracking - Context

**Gathered:** 2026-07-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Swap the native iOS phone client's position source from CoreMotion dead-reckoning (the Phase 06.2 parity baseline, zero-stubbed position) to ARKit `ARSession` world tracking for precise 6DOF position, sourcing orientation from the same ARKit pose (no parallel CoreMotion orientation pipeline — decided in Phase 06.2 as D-02, locked here). Includes a mandatory on-device ARKit tracking-precision verification checkpoint that is a go/no-go gate for the project.

**Scope was expanded during this discussion to also include native touch input capture** — `SensorPacketEncoder`'s `touchActive`/`touchX`/`touchY` fields exist on the wire but nothing on the native iOS side populates them yet. This is native-client parity work in the same spirit as what 06.2 left stubbed, not a new capability, so it's included here rather than deferred.

**Explicitly deferred (not this phase):** a desktop-side WebRTC throughput + Three.js rendering FPS monitor overlay — unrelated to ARKit/iOS, see `<deferred>` below.

</domain>

<decisions>
## Implementation Decisions

### Position & Orientation Source (carried forward and locked from Phase 06.2)
- **D-01:** ARKit's `ARSession`/camera-transform pose is the source for **both** position and orientation. No parallel CoreMotion (`CMDeviceMotion`) orientation pipeline runs alongside it — this supersedes `CoreMotionSource.swift`'s role from 06.2 (carries forward 06.2's D-02).
- **D-02 (research flag, mandatory):** ARKit's camera-transform axis convention must be verified against the existing desktop decode/SLERP pipeline (Phase 6) before implementation proceeds — Phase 6 already hit an axis-inversion bug once porting position (`quick/260710-w83`). Carries forward 06.2's D-03.

### Touch Input Capture (new scope added to this phase)
- **D-03:** Capture single tap + drag only — continuously update `touchActive`/`touchX`/`touchY` while a finger is down. No separate swipe-gesture recognition, no configurable on-screen button regions.
- **D-04:** Captured across the **full** `ActiveSessionView` screen — not a dedicated touch-only sub-region.
- **D-05:** `touchX`/`touchY` normalized against the app's **explicit landscape-locked bounds** (not raw UIKit view bounds) — must be reconciled with ARKit's own coordinate conventions during implementation (relates to D-02).
- **D-06:** The phone shows **local visual feedback** when touch is active (in addition to the desktop's existing emissive-flash feedback in `scene.ts`).

### Tracking Loss & Fallback
- **D-07:** `ARCamera.TrackingState` → `driftConfidence` mapping matches `webxr.ts` exactly: `normal` → 1.0, `limited` → 0.5, `notAvailable` → 0.0. Position **freezes at last-known-good** on loss; the tracking source is **never switched live mid-session** (mirrors 06.1's D-06 convention).
- **D-08:** ARKit's `limited` sub-reasons (`initializing`, `excessiveMotion`, `insufficientFeatures`, `relocalizing`) are **not** differentiated on the wire (still a flat 0.5) but **are** surfaced individually in the phone's own local UI via DynamicToast (D-15) — useful during the on-device precision checkpoint (D-16).
- **D-09:** If ARKit is entirely unavailable at session start (unsupported device, or camera permission denied), **block session start with an error message** — do NOT silently fall back to the 06.2 CoreMotion-only baseline. (This is a start-time gate, distinct from D-07's "never switch mid-session" runtime rule.)

### Session UX
- **D-10:** The ARKit world origin auto-resets **after a hold-still calibration step**, mirroring the web client's `runCalibration`-before-sensor-pipeline convention (`client/src/sensor/encode.ts`) — not immediately at pair time.
- **D-11:** A **phone-side manual recenter control** is added **in addition to** the existing desktop-side R-key reset (`resetAllPlayerPositions()` in `client/src/scene.ts`). The desktop R-key resets **all** players' rendered position at once via an offset; the phone-side recenter lets a **single player** re-zero just their own ARKit tracking origin. (Confirmed additive, not redundant, after discussing the existing R-key mechanism with the user.)
- **D-12:** Recenter and a new Disconnect/Back control **share one hidden reveal gesture**: a **2-finger long-press, one finger held in each top corner**. This reveals a small overlay menu containing both buttons. Chosen because the full screen is already the touch-input capture surface (D-04) — both controls must be very hard to trigger by accident.
- **D-13:** Disconnect/Back button behavior: if connected, **disconnects** the session; if already disconnected or in an error state, **navigates back to the first/initial screen**.
- **D-14 (data-scope rule, not a UI/visibility choice):** ARKit's camera is used **strictly internally** to compute the tracking pose. No raw camera frames are ever displayed (no passthrough view of any kind), stored, or transmitted anywhere in the app — only motion/pose data leaves the ARSession. Stronger than `webxr.ts`'s "never request the raw-camera-pixel feature" rule (T-06.1-01) — there is no passthrough view at all here.
- **D-15:** Use the existing **DynamicToast** component (`dynamicIslandToast(isPresented:duration:value:)` view modifier + `Toast` struct, `mobile/ios-app/immersiveRT/DynamicToast/`) for all user-facing messages in this phase — errors (e.g. ARKit unavailable/blocked, D-09), tracking-limited-reason debug info (D-08), and suggestions. `duration: nil` for persistent, or a specific seconds value for auto-dismiss.

### Go/No-Go Checkpoint (mandatory on-device verification gate)
- **D-16:** Test protocol is **both**: (a) a quantitative measured-motion test — phone moved through known distances/positions within a ~1m³ working volume, comparing ARKit-reported position against ground truth; and (b) a qualitative in-game feel pass using the existing demo/room scene.
- **D-17:** Quantitative tolerance target: **~1-3mm** — tighter than ARKit's typically-quoted best-case accuracy (a few mm to ~1cm). User's explicit rationale: phone motion stays within a small ~1m³ volume (not room-scale walking), so every mm of error is a meaningfully large fraction of the total range. **Research flag:** confirm during research whether ARKit's VIO can realistically hit this tolerance for small-volume motion specifically — if it can't even under ideal conditions, that is exactly the scenario D-18's fallback exists for.
- **D-18:** If the checkpoint fails, **stop and re-scope** (open a new debug/research cycle) — do **NOT** silently ship a degraded CoreMotion-only fallback. ARKit position tracking is the entire point of this phase, so a failed checkpoint is a hard blocker, not a soft-degrade.

### Claude's Discretion
- Exact SwiftUI implementation of the hidden 2-finger-long-press gesture recognizer (D-12).
- Exact wording/symbol choices for individual DynamicToast messages (D-15).
- Precise hold-still duration for the calibration step before ARKit origin locks (D-10).
- Specific measured-point layout/protocol for the quantitative go/no-go test (D-16) — e.g. number of sample points, which axes are tested.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- DynamicToast component — a complete Dynamic-Island-style user-facing messaging system already built and in use (D-15); no need to build a new notification/error UI.
- `resetAllPlayerPositions()` desktop R-key reset — complements rather than replaces the new phone-side recenter (D-11).
- `SensorPacketEncoder`'s existing `px`/`py`/`pz` + `touchActive`/`touchX`/`touchY` fields — the 36-byte wire schema needs no changes; this phase just populates real values into fields that already exist and are already zero-stubbed correctly.

### Established Patterns
- "Never touch raw camera pixels" — `webxr.ts`'s T-06.1-01 pattern, extended in this phase to "no passthrough view at all" (D-14).
- `safeFloat()` sanitization before every `DataView` write (`SensorPacketEncoder.swift`) — camera- and touch-derived values must pass through this guard too.
- `_packetBuf`/`Data` module-scope buffer reuse — no per-tick GC at 60Hz; ARKit pose reads must fit into this existing encode-loop structure.
- CoreMotion's dedicated-background-queue pattern (`CoreMotionSource.swift`, never main queue/actor per the project's `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` build setting) — ARSession delegate callbacks should follow the same concurrency model.

### Integration Points
- `ActiveSessionView.swift`'s `pollTimer` / 60Hz encode loop is where ARKit pose reads, touch state, and DynamicToast triggers all plug in alongside the existing callback structure that currently only wires CoreMotion.
- `TransportManager.swift`'s existing connect/pair/fan-out/reconnect loop is what the new Disconnect control (D-13) hooks into.

</code_context>

<specifics>
## Specific Ideas

- "I think that the smartphone will be moved in a 1m3 cube volume, so the precision must be very high. mm if possible" — user's explicit rationale for the tight ~1-3mm tolerance target (D-17); this is a hand-held small-volume tracking use case, not room-scale walking.
- "It is not about hiding anything. Only data about motion must be used." — user's correction that reframed the camera-passthrough question from a UI/visibility choice into a hard data-scope rule (D-14).
- "We have a key to reset the cube position on server" — user's reference to the existing desktop R-key reset (`resetAllPlayerPositions`), which led to confirming the phone-side recenter is additive, not redundant (D-11).
- "This button should be hidden, accessible only with specific actions, we want to prevent unintentional clicks" — user's explicit requirement behind the shared hidden-menu design (D-12).
- "Use the existing DynamicIslandToast to report messages to the user" — user's explicit suggestion, adopted as D-15.

</specifics>
