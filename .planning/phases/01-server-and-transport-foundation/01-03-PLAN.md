## Phase Goal

**As a** developer on a network that blocks UDP/QUIC, **I want to** connect to the server
over WebSocket and have signaling messages echo back, **so that** the ImmersiveRT platform
works even when WebTransport is unavailable.

<objective>
Implement the WebSocket fallback listener in `server/src/ws_server.rs` as a production-ready
TCP listener that upgrades connections to WebSocket using tokio-tungstenite and echoes messages
back. Wire it into `main.rs` alongside the WebTransport listener using `tokio::try_join!`.
Add an automated integration test that verifies the WebSocket echo round-trip.

Purpose: Satisfies INFRA-05 and completes the Phase 1 success criteria. After this plan, the
server binary handles both QUIC-capable and QUIC-blocked clients simultaneously. The automated
integration test provides regression coverage for the WebSocket path going forward.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| internet → TCP:8080 | Untrusted WebSocket connections arrive; plain ws:// (no TLS in Phase 1) |
| WebSocket client → echo handler | Client-supplied messages echoed verbatim; no parsing in Phase 1 |
| tokio::try_join! error boundary | Panic or fatal error in one listener kills both via try_join propagation |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-08 | Tampering | Plain WebSocket ws:// on port 8080 | medium | accept | Phase 1 WS is LAN-dev-only; this is an explicit RESEARCH.md Open Question 1 decision — WSS with TLS added in Phase 2 Docker deployment; acceptable risk until internet exposure |
| T-01-09 | DoS | ws_server accept loop | high | mitigate | Task 1: each connection dispatched to `tokio::spawn`; errors logged with `tracing::warn!` and connection dropped; accept loop continues; unbounded concurrent connection growth addressed in Phase 2 with max-connection tracking |
| T-01-10 | DoS | Large WebSocket frame echoed verbatim | medium | mitigate | Phase 1 echo is fire-and-forget; in Phase 2 add max-message-size config to tokio-tungstenite config builder (WebSocketConfig::max_message_size); acceptable for Phase 1 where only trusted dev clients connect |
| T-01-11 | Spoofing | tokio::try_join! failure propagation | low | accept | If wt_server panics, try_join propagates the error to main and exits the process; this is a clean-fail behavior acceptable for Phase 1; Phase 2 adds restart logic in Docker Compose |
</threat_model>
