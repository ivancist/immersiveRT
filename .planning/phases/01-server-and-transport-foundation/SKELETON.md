# Walking Skeleton — ImmersiveRT Phase 1

> The thinnest possible end-to-end slice: `cargo run` starts the server, Chrome connects
> via WebTransport (or falls back to WebSocket), a latency probe echoes back.
> All architectural decisions made here govern subsequent phases without renegotiation.

---

## Project Structure

```
immersiveRT/
├── Cargo.toml                     # workspace root — members = ["server"]
├── Cargo.lock                     # committed to repo
├── .gitignore                     # includes certs/, target/
├── certs/                         # gitignored — mkcert output
│   ├── localhost+2.pem
│   └── localhost+2-key.pem
└── server/
    ├── Cargo.toml                 # package: immersive-rt-server
    └── src/
        ├── main.rs                # #[tokio::main] — spawns wt + ws listeners

```

## Architectural Decisions

These decisions are locked for all subsequent phases. Changing any of these requires a
phase-level discussion and CONTEXT.md update.

### Framework

| Layer | Decision | Rationale |
|-------|----------|-----------|
| Async runtime | tokio 1.x (rt-multi-thread) | De-facto Rust async runtime; required by wtransport; work-stealing scheduler saturates cores without thread-per-connection overhead |
| WebTransport server | wtransport 0.7.1 | Only pure-Rust WebTransport implementation; pairs natively with tokio; ~17k weekly downloads; actively maintained by BiagioFesta |
| WebSocket fallback | tokio-tungstenite 0.29 | Lightest tokio-native WS upgrade; no router overhead needed for raw signaling; 4M weekly downloads |
| Error handling | anyhow 1.x | Ergonomic error propagation with `?` across async boundaries; no custom error hierarchy needed at this stage |
| Logging | tracing + tracing-subscriber | Structured JSON-capable logging; same ecosystem as tokio; respects RUST_LOG env var |
| Serialization (probe) | serde + serde_json 1.x | Echo probe uses JSON at Phase 1; binary MessagePack (msgpackr) comes in Phase 5 for sensor packets |

### Network Architecture

| Property | Value | Rationale |
|----------|-------|-----------|
| WebTransport port | 4433 (UDP/QUIC) | Standard WebTransport port; QUIC requires UDP |
| WebSocket fallback port | 8080 (TCP) | Separate port mandatory — QUIC (UDP) and WebSocket (TCP) cannot share a port on the same address |
| Concurrency model | Two `tokio::spawn` loops joined with `tokio::try_join!` | Both listeners run concurrently; failure in either propagates to main and terminates the process |
| Session handling | One `tokio::spawn` per accepted connection | Each connection gets an independent task; errors log but do not kill the accept loop |

### TLS / Certificate Strategy

| Environment | Certificate Source | Chrome Requirement |
|-------------|--------------------|--------------------|
| Local dev | mkcert — generates a locally-trusted cert signed by a mkcert CA installed in the OS trust store | `chrome://flags/#webtransport-developer-mode` MUST be enabled — Chrome's QUIC stack does not consult the OS trust store by default |
| Production (future) | Let's Encrypt or equivalent trusted CA | No Chrome flag needed; real CA certs pass QUIC validation by default |

**Cert files (generated by mkcert, gitignored):**
- `certs/localhost+2.pem` — TLS certificate (SAN: localhost, 127.0.0.1, ::1)
- `certs/localhost+2-key.pem` — Private key

**Runtime config via environment variables:**
- `CERT_PATH` — defaults to `certs/localhost+2.pem`
- `KEY_PATH` — defaults to `certs/localhost+2-key.pem`
- `WT_PORT` — defaults to `4433`
- `WS_PORT` — defaults to `8080`
- `RUST_LOG` — defaults to `info`

### Latency Echo Probe

| Property | Value | Rationale |
|----------|-------|-----------|
| Transport | WebTransport bidirectional stream | Bidirectional stream ensures echo arrives (reliable); datagrams are unreliable and unsuitable for measuring round-trip |
| Message format | JSON: `{"type":"ping","client_ts":<ms>}` / `{"type":"pong","client_ts":<ms>,"server_ts":<ms>}` | Simple, human-readable at Phase 1; replaced by binary MessagePack in Phase 5 |
| Target latency | < 10ms on LAN | Baseline metric; WebTransport latency advantage over WebSocket becomes visible under load |

### Testing Strategy

| Layer | Approach |
|-------|----------|
| Unit tests | `cargo test -p immersive-rt-server` — covers echo logic, message parsing |
| Integration (WebSocket) | `cargo test test_ws_echo` — connects a local WebSocket client, sends echo, verifies round-trip |
| Integration (WebTransport) | Manual — requires Chrome with `webtransport-developer-mode` flag; no headless WT client available in Phase 1 |
| CI gate | `cargo build 2>&1` with zero warnings; `cargo test --workspace` green |

## One Real Connection

The walking skeleton is verified when:

1. `mkcert -install && cd certs && mkcert localhost 127.0.0.1 ::1` generates cert files
2. `cargo run` starts with log lines:
   ```
   INFO immersive_rt_server: WebTransport listening on :4433
   INFO immersive_rt_server: WebSocket fallback listening on :8080
   ```
3. Chrome (with dev flag enabled) opens a WebTransport connection to `https://localhost:4433` and the server logs a session accepted
4. A WebSocket client connecting to `ws://localhost:8080` echoes a ping message back
5. `cargo test --workspace` passes with zero warnings
