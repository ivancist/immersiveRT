## User Constraints (from CONTEXT.md)

### Locked Decisions

**Packet Schema Integration**
- **D-01:** Camera-tracked position reuses the existing `px/py/pz` fields in the fixed 36-byte `SensorPacket` (schema v1, D-14) — no schema version bump, no explicit position-source flag on the wire.
- **D-02:** WebXR's tracking-state enum (`normal` / `limited` / `lost`) maps to discrete `driftConfidence` values: `normal` → 1.0, `limited` → 0.5, `lost` → 0.0.
- **D-03:** `gestureDisplacement` (dx/dy/dz) is recomputed every frame as the delta between the current camera position and the camera position from a rolling window (~300ms) ago — no ZUPT-style discrete reset event when camera tracking is active.

**Fallback & Detection**
- **D-04:** When WebXR `immersive-ar` is unsupported (non-ARCore Android device, or iOS), the phone keeps running the existing ZUPT+Kalman dead-reckoning pipeline unchanged and keeps writing `px/py/pz` from it — same best-effort/drifty behavior as today (SENS-03/04).
- **D-05:** Phone detects `navigator.xr.isSessionSupported('immersive-ar')` and shows a tracking-mode indicator ("Camera Tracking" vs "Motion Only") on the existing `view-calibrating` screen (phone.ts, D-08 calibration scene).
- **D-06:** Tracking source is chosen once at calibration and locked for the session. If the WebXR session drops mid-game (tracking lost, app backgrounded, etc.), the phone reports `driftConfidence=0` and waits for WebXR to recover — no live hand-off to the Kalman/ZUPT pipeline mid-session.

**Platform Scope**
- **D-07:** Phase 06.1 delivers Android WebXR camera tracking only. No browser-based iOS camera-tracking spike is attempted — MindAR/AR.js are marker-tracking only, and no free, full 6DOF browser alternative was identified worth spiking.
- **D-08:** iOS's real position-tracking solution is a native ARKit companion app, planned as a near-term follow-up — but building it is explicitly OUT of scope for 06.1. iOS continues on the ZUPT+Kalman fallback (D-04) until that native-app phase lands.

- Specific WebXR session/frame-loop integration with the existing rAF loop in phone.ts.
- Visual design of the "Camera Tracking" / "Motion Only" badge on the calibration screen.

- **Native iOS companion app (ARKit)** for camera-assisted position tracking — belongs in its own future roadmap phase; explicitly out of scope for 06.1 (D-08).

## Summary

Phase 06.1 replaces (on supported Android devices) the phone's IMU-only Kalman/ZUPT position dead-reckoning — which the prior debug session conclusively found cannot be tuned to usable in-hand accuracy — with WebXR `immersive-ar` camera-assisted 6DOF pose tracking. The critical architectural fact discovered during research is that **`client/src/phone.ts` has no Three.js renderer at all** (only the desktop's `scene.ts` imports `three`), so the "opaque-layer trick" language in the phase goal does not map to a Three.js `XRManager`/`ARButton` integration on the phone side. Instead, the phone client should call the **raw browser `navigator.xr` WebXR API directly** — request an `immersive-ar` session with `dom-overlay` as an optional feature (rooted at `document.body`, i.e. the existing phone UI), which visually replaces the raw camera passthrough with the app's own opaque UI. A minimal (1x1 or hidden) canvas + `XRWebGLLayer` must still be created and passed to `session.updateRenderState()` because the WebXR spec requires a base layer to exist even when no rendering is performed — this is the literal "opaque layer trick": the layer exists structurally to satisfy the API contract, but is never drawn into, and `dom-overlay` (or, as an explicit fallback, a solid-color-cleared canvas) is what actually hides the passthrough visually from the user.

Position data is read every `XRSession.requestAnimationFrame` tick via `XRFrame.getViewerPose(referenceSpace)`. WebXR has no first-class `normal`/`limited`/`lost` tracking-state enum — the spec-correct way to derive it is: `getViewerPose()` returns `null` → treat as `lost` (driftConfidence 0.0); pose returned with `pose.emulatedPosition === true` → treat as `limited` (driftConfidence 0.5, per D-02); pose returned with `emulatedPosition === false` → `normal` (driftConfidence 1.0). This `XRSession.requestAnimationFrame` loop runs independently of `window.requestAnimationFrame` and, critically, independently of the existing `devicemotion`/`deviceorientation` event listeners that currently drive the entire sensor-tick-and-broadcast pipeline in `phone.ts` — these are plain DOM events, not tied to any rAF loop, so no loop-conflict exists. The simplest integration is a shared closure variable: the XR frame loop writes the latest camera position + tracking state into a small piece of session-local state; the existing `devicemotion` handler (unchanged in its overall structure) reads that state instead of/alongside the Kalman filter output when building each `SensorPacket`.

**Primary recommendation:** Implement Android camera tracking as a self-contained module (e.g. `client/src/sensor/webxr.ts`) that exposes `isSupported()`, `start()`, and a per-tick `getPose()`-style read, using raw `navigator.xr` — not Three.js's XR helpers. Wire it into `phone.ts`'s existing `startSensorPipeline()` as a parallel branch selected once at calibration time per D-06, leaving `ZUPTDetector`/`Kalman1D`/the existing `devicemotion` handler completely untouched in the fallback branch.

### Pattern 1: Minimal WebXR session for pose-only tracking (no visible rendering)
**What:** Request an `immersive-ar` session, satisfy the spec's mandatory base-layer requirement with a throwaway canvas/WebGL context that is never drawn into, and read pose data only.
**When to use:** Any app that needs 6DOF device pose but has no 3D scene to render (this project's phone client).
**Example:**
```typescript
// Source: MDN XRSession.updateRenderState + immersive-web/webxr explainer
// [CITED: developer.mozilla.org/en-US/docs/Web/API/XRSession/updateRenderState]
async function startWebXrPoseTracking(): Promise<XRSession> {
  const session = await (navigator as any).xr.requestSession('immersive-ar', {
    // 'local' is sufficient for a controller app (no floor-relevance) — see State of the Art.
    requiredFeatures: ['local'],
    // dom-overlay is OPTIONAL: session must still succeed on devices that lack it.
    optionalFeatures: ['dom-overlay'],
    domOverlay: { root: document.body }, // reuses existing phone UI as the opaque cover
  });

  // Spec requires a base layer even if it is never rendered into (the "opaque layer" here
  // is structural, not visual — dom-overlay/solid-clear fallback handles the visual hiding).
  const canvas = document.createElement('canvas'); // 1x1, never appended to DOM
  const gl = canvas.getContext('webgl', { xrCompatible: true }) as WebGLRenderingContext;
  await session.updateRenderState({ baseLayer: new (window as any).XRWebGLLayer(session, gl) });

  return session;
}
```

### Pattern 2: Per-frame tracking-state → driftConfidence mapping (D-02)
**What:** Derive a discrete normal/limited/lost signal from `XRFrame.getViewerPose()` since WebXR has no explicit tracking-state enum on the base spec.
**When to use:** Every XR animation frame, feeding straight into D-02's fixed 1.0/0.5/0.0 mapping.
**Example:**
```typescript
// Source: immersive-web/webxr spatial-tracking-explainer + MDN XRPose.emulatedPosition
// [CITED: immersive-web.github.io/webxr/spatial-tracking-explainer.html]
function readPoseAndConfidence(frame: XRFrame, refSpace: XRReferenceSpace):
  { pos: { x: number; y: number; z: number } | null; driftConfidence: 0 | 0.5 | 1 } {
  const pose = frame.getViewerPose(refSpace);
  if (!pose) {
    return { pos: null, driftConfidence: 0 }; // D-02: lost
  }
  const p = pose.transform.position;
  if (pose.emulatedPosition) {
    return { pos: { x: p.x, y: p.y, z: p.z }, driftConfidence: 0.5 }; // D-02: limited
  }
  return { pos: { x: p.x, y: p.y, z: p.z }, driftConfidence: 1 }; // D-02: normal
}
```

### Pattern 3: Rolling ~300ms gestureDisplacement (D-03) — no discrete reset
**What:** Maintain a small ring buffer of `{t, x, y, z}` camera-position samples; each frame compute the delta between "now" and the oldest sample still within the ~300ms window.
**When to use:** Only while the WebXR branch is active (D-03 explicitly says no ZUPT-style reset event applies here — this replaces, not extends, the existing ZUPT-gated `gestureOrigin` reset logic from `phone.ts`).
**Example:**
```typescript
// Applies D-03. Distinct from the existing ZUPT-gated gestureOrigin logic in phone.ts,
// which remains used only in the fallback (non-WebXR) branch.
interface PosSample { t: number; x: number; y: number; z: number; }
const ring: PosSample[] = [];
const WINDOW_MS = 300;

function pushAndComputeDelta(now: number, cur: { x: number; y: number; z: number }) {
  ring.push({ t: now, x: cur.x, y: cur.y, z: cur.z });
  while (ring.length > 0 && now - ring[0].t > WINDOW_MS) { ring.shift(); }
  const base = ring[0] ?? { x: cur.x, y: cur.y, z: cur.z };
  return { dx: cur.x - base.x, dy: cur.y - base.y, dz: cur.z - base.z };
}
```

### Anti-Patterns to Avoid
- **Requesting `hit-test` or `plane-detection` as required/optional features:** Not needed for pure position tracking (no surface placement in this app); adds unnecessary ARCore CPU overhead and battery/thermal cost, directly working against the phase's own stated escape-hatch concern. [CITED: developers.google.com/ar/develop/performance]
- **Requesting the `camera-access` (raw camera) feature:** This is a *separate*, much higher-privacy-sensitivity WebXR module (raw pixel access) from the base `immersive-ar` pose tracking this phase needs. Never request it — it triggers extra consent flows and exposes user camera imagery for no benefit here. [CITED: immersive-web.github.io/raw-camera-access/]
- **Wiring a Three.js `XRManager`/`ARButton` into `phone.ts`:** phone.ts has no renderer; adding one purely to drive session lifecycle imports an entire rendering stack for zero visual output.
- **Assuming a single `trackingState` property exists on `XRFrame`/`XRPose`:** the base WebXR spec has no such field — it must be derived from `getViewerPose()` nullability + `emulatedPosition`, per Pattern 2.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Camera-assisted 6DOF position tracking (VIO/SLAM) | A custom visual-inertial odometry algorithm | WebXR `immersive-ar` (ARCore-backed on Chrome/Android) | ARCore's VIO is a mature, hardware-accelerated, years-refined SLAM implementation; hand-rolling any equivalent is explicitly the "don't" this phase exists to correct (the prior debug session already proved IMU-only tuning is a dead end — camera VIO is strictly harder to hand-roll than IMU-only was) |
| Hiding the AR camera passthrough visually | Custom shader/compositing tricks to mask the camera feed | `dom-overlay` optional feature (with solid-clear-canvas fallback) | `dom-overlay` is a standardized WebXR module purpose-built for exactly this use case (reusing existing page DOM as the visible content during an immersive session) |
| Tracking-state detection | Custom heuristics on raw pose jitter/velocity | `getViewerPose()` nullability + `pose.emulatedPosition` | This is the spec-documented, cross-browser-consistent signal — reinventing jitter heuristics would be strictly worse and non-portable |

**Key insight:** This entire phase is fundamentally about *not* hand-rolling VIO — it deliberately trades the prior phase's from-scratch IMU dead-reckoning approach for a platform-provided, hardware-accelerated tracking system. Every sub-decision in this research (dom-overlay, tracking-state derivation, reference space choice) follows the same "use the spec-provided primitive" principle.

## Common Pitfalls

### Pitfall 1: WebXR base spec has no `trackingState` field — assuming one exists breaks D-02
**What goes wrong:** Code written assuming `frame.trackingState` or `pose.trackingState` exists will be `undefined` at runtime with no compile error if not using a properly typed WebXR lib definition, silently breaking D-02's driftConfidence mapping (always falls to some default).
**Why it happens:** Some AR/VR SDKs (Unity, native ARKit/ARCore) do expose a tracking-state enum, and it is easy to assume the browser API mirrors that.
**How to avoid:** Use the two-signal derivation in Pattern 2 (`getViewerPose()` null-check + `emulatedPosition`) exclusively.
**Warning signs:** TypeScript compiles clean but driftConfidence is always the same value regardless of actual device movement/occlusion.

### Pitfall 2: `dom-overlay` is optional — assuming it's always available leaves the raw camera visible
**What goes wrong:** If `dom-overlay` isn't supported on a given Android/Chrome build (older Chrome versions, some OEM browsers), and the code has no fallback, the user sees the raw camera feed underneath a transparent WebGL canvas instead of the app's UI — jarring and privacy-uncomfortable (their own camera feed full-screen).
**Why it happens:** `dom-overlay` must be requested via `optionalFeatures`, not `requiredFeatures` — the session will still succeed without it, and it's easy to not check `session.domOverlayState`/whether the overlay actually activated.
**How to avoid:** After session start, check `session.domOverlayState?.type` (or the vendor-prefixed equivalent used at the time of implementation); if absent, fall back to clearing the WebGL canvas to a solid opaque color every frame as the "opaque layer" hiding mechanism.
**Warning signs:** Feature works in Chrome Canary/dev builds but raw camera flashes through on some real devices during QA.

### Pitfall 3: Requesting `camera-access` or `hit-test` "just in case" inflates permission scope and battery cost
**What goes wrong:** Requesting features the app doesn't use (raw camera pixels, hit-test/plane-detection) triggers extra permission prompts and measurably increases ARCore CPU/battery usage — directly undermining the phase's own stated battery/heat escape-hatch condition.
**Why it happens:** Copy-pasting from typical WebXR AR tutorials (most are hit-test/placement demos) without pruning the requiredFeatures/optionalFeatures list to only what's needed.
**How to avoid:** requiredFeatures should be `['local']` only (or omitted — the base spec guarantees `viewer`/`local` availability); optionalFeatures `['dom-overlay']` only.
**Warning signs:** Extra permission dialogs beyond camera access appear; device heats up faster than expected during manual testing.

### Pitfall 4: WebXR position values need the same `safeFloat()`/`POSITION_MAX` treatment as Kalman output
**What goes wrong:** If camera-derived px/py/pz are written straight into the `SensorPacket` without passing through `safeFloat()` and the existing `POSITION_MAX` clamp, a single bad/NaN pose sample (which can happen transiently during a tracking-state transition) could poison a float16 wire value or exceed the desktop's expected bounds.
**Why it happens:** The existing bounds/sanitization logic lives inline in the `devicemotion` handler around the Kalman-specific variables (`rawPx`, `clamp()`); a naive WebXR integration might bypass that code path entirely since it's a "different" position source.
**How to avoid:** Route camera-derived x/y/z through the exact same `clamp(v, POSITION_MAX)` + `safeFloat()` treatment already applied to `rawPx/rawPy/rawPz` before assignment into the `SensorPacket`.
**Warning signs:** Desktop cube occasionally teleports to POSITION_MAX bounds or NaN-derived garbage right after a WebXR tracking-state transition.

### Pitfall 5: "No live hand-off" (D-06) implies position must FREEZE, not go stale/garbage, on tracking loss
**What goes wrong:** D-06 says the phone reports `driftConfidence=0` and "waits" when WebXR drops — but the sensor packet cadence is still driven by the `devicemotion` event (which keeps firing independent of WebXR state). If the WebXR branch's `getPose()` read returns `null`/stale on every subsequent tick with no explicit "freeze at last known good value" logic, px/py/pz could silently reset to 0,0,0 or NaN→0 via `safeFloat()`'s fallback, visually snapping the desktop cube to origin instead of holding still.
**Why it happens:** The natural default when a pose read fails is "return zero/undefined," which is the wrong behavior here — the correct behavior is "hold the last known-good camera position."
**How to avoid:** `webxr.ts` should retain the last-known-good `{x,y,z}` in closure state and return that (not zero, not NaN) whenever `getViewerPose()` returns null, alongside `driftConfidence: 0`.
**Warning signs:** Desktop cube snaps to origin (rather than freezing in place) the moment the phone's camera is occluded or the app is backgrounded.

### Pitfall 6: Desktop `scene.ts` has no freeze-on-`driftConfidence=0` behavior today — review needed per CONTEXT.md
**What goes wrong:** `scene.ts` currently just renders whatever `px/py/pz`/`dx/dy/dz` values arrive each packet with no confidence-gated freeze logic (confirmed via source read — `driftConfidence` is displayed in the HUD only, never used to gate rendering). Combined with Pitfall 5's freeze-at-phone-side fix, this should be *sufficient* (frozen phone-side position → frozen desktop rendering), but if the phone-side freeze (Pitfall 5) is NOT implemented correctly, the desktop has no independent safety net.
**Why it happens:** `driftConfidence` was added in Phase 5/6 purely as a diagnostic HUD number, not as a behavioral gate.
**How to avoid:** Treat Pitfall 5 as the primary fix (phone freezes correctly), and treat any desktop-side freeze-on-`driftConfidence===0` logic as optional defense-in-depth, not a substitute.
**Warning signs:** None expected if Pitfall 5 is handled — flag only if QA sees jitter/snapping on the desktop during a simulated WebXR tracking-loss test.

### Pitfall 7: `isSessionSupported()` can throw/reject on browsers without `navigator.xr` at all
**What goes wrong:** Calling `navigator.xr.isSessionSupported(...)` when `navigator.xr` is `undefined` (Safari, older Android browsers, non-HTTPS contexts) throws a `TypeError` synchronously, which if unguarded crashes the calibration-mode-detection code path (D-05).
**Why it happens:** `navigator.xr` only exists on WebXR-capable browsers; must feature-detect its presence before calling any method on it. This mirrors the exact same pattern already used for `DeviceMotionEvent.requestPermission` feature-detection in `attachGrantButton()` — apply the same style here.
**How to avoid:** `if (typeof navigator !== 'undefined' && 'xr' in navigator) { ... }` guard before calling `isSessionSupported`; treat absence as immediate "unsupported" → fallback branch (D-04), same as iOS.
**Warning signs:** Uncaught exception in browser console on iOS/older Android instead of a graceful fallback to Motion-Only mode.

## Code Examples

Verified patterns from official/community sources (see Sources):

### Feature detection with safe guard (D-05)
```typescript
// Source: MDN XRSystem.isSessionSupported() + existing DeviceMotionEvent feature-detect
// pattern already used in phone.ts's attachGrantButton()
// [CITED: developer.mozilla.org/en-US/docs/Web/API/XRSystem/isSessionSupported]
async function detectWebXrSupport(): Promise<boolean> {
  if (typeof navigator === 'undefined' || !('xr' in navigator)) { return false; }
  try {
    return await (navigator as any).xr.isSessionSupported('immersive-ar');
  } catch (_e) {
    return false; // SecurityError (non-HTTPS) or any other rejection → treat as unsupported
  }
}
```

### dom-overlay activation check + solid-clear fallback
```typescript
// Source: W3C WebXR DOM Overlays Module + MDN updateRenderState
// [CITED: w3.org/TR/webxr-dom-overlays-1/]
function isDomOverlayActive(session: XRSession): boolean {
  return !!(session as any).domOverlayState?.type;
}

// If dom-overlay did not activate, clear the WebGL canvas to opaque black every XR frame
// as the fallback "opaque layer" so the raw camera passthrough is not visible.
function clearOpaqueFallback(gl: WebGLRenderingContext): void {
  gl.clearColor(0, 0, 0, 1);
  gl.clear(gl.COLOR_BUFFER_BIT);
}
```

## Open Questions

1. **Exact property/state names for detecting dom-overlay activation and current environment blend mode across Chrome-on-Android versions**
   - What we know: The DOM Overlays Module spec defines `domOverlayState` conceptually; `environmentBlendMode` is a documented `XRSession` attribute (`opaque`/`alpha-blend`/`additive`).
   - What's unclear: Whether `@types/three` or a WebXR-specific `@types/webxr` package needs to be added for TypeScript typings on `navigator.xr`/`XRSession`/`XRWebGLLayer`/`XRFrame` — none of these are currently typed in this project (no WebXR types exist in `client/src/types.ts` or elsewhere).
   - Recommendation: Planner should include a task to either (a) add minimal local `.d.ts` ambient declarations for the small subset of WebXR types actually used (avoids a new dependency, consistent with "no new packages" recommendation above), or (b) add `@types/webxr` (a types-only devDependency, effectively zero runtime/security risk) if broader typing coverage is preferred. Given this project's existing style of hand-written minimal type surfaces (see `types.ts`), option (a) is the better fit and should go through the Package Legitimacy Gate only if option (b) is chosen.

2. **Actual on-device frame rate WebXR position tracking will deliver on the target Android hardware**
   - What we know: The phase goal mentions "~30fps" as a target; ARCore/WebXR frame rate is generally tied to the device's display refresh/compositor rate (often 30-60Hz) and is not directly configurable by the app.
   - What's unclear: Whether the actual achieved *tracking update* rate (as opposed to raw session rAF rate) matches or falls short of 30fps on the specific test devices available to this project — this can only be confirmed via on-device testing.
   - Recommendation: Treat 30fps as an expectation to validate empirically during Wave 0/1 human-verify testing, not a guaranteed contract; the rolling ~300ms gestureDisplacement window (D-03) should be tolerant of variable tick rates (it already is, by design — see Pattern 3).
