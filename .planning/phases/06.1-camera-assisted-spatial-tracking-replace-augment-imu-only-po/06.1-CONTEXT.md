# Phase 06.1: Camera-Assisted Spatial Tracking (INSERTED) - Context

**Gathered:** 2026-07-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Add camera-assisted 6DOF position tracking on Android via WebXR `immersive-ar` (ARCore), replacing the existing IMU-only Kalman/ZUPT dead-reckoning as the *primary* position source on supported devices — while keeping that same Kalman/ZUPT pipeline as an automatic fallback wherever WebXR AR isn't available (unsupported Android devices, and iOS). Orientation continues to come from `DeviceOrientationEvent` unchanged, on every platform.

**This phase is Android-only for camera tracking.** iOS gets no browser-based camera-tracking work in 06.1 — it stays on the existing IMU fallback. The real iOS solution (a native ARKit companion app) is explicitly deferred to its own future phase.

</domain>

<decisions>
## Implementation Decisions

### Packet Schema Integration
- **D-01:** Camera-tracked position reuses the existing `px/py/pz` fields in the fixed 36-byte `SensorPacket` (schema v1, D-14) — no schema version bump, no explicit position-source flag on the wire.
- **D-02:** WebXR's tracking-state enum (`normal` / `limited` / `lost`) maps to discrete `driftConfidence` values: `normal` → 1.0, `limited` → 0.5, `lost` → 0.0.
- **D-03:** `gestureDisplacement` (dx/dy/dz) is recomputed every frame as the delta between the current camera position and the camera position from a rolling window (~300ms) ago — no ZUPT-style discrete reset event when camera tracking is active.

### Fallback & Detection
- **D-04:** When WebXR `immersive-ar` is unsupported (non-ARCore Android device, or iOS), the phone keeps running the existing ZUPT+Kalman dead-reckoning pipeline unchanged and keeps writing `px/py/pz` from it — same best-effort/drifty behavior as today (SENS-03/04).
- **D-05:** Phone detects `navigator.xr.isSessionSupported('immersive-ar')` and shows a tracking-mode indicator ("Camera Tracking" vs "Motion Only") on the existing `view-calibrating` screen (phone.ts, D-08 calibration scene).
- **D-06:** Tracking source is chosen once at calibration and locked for the session. If the WebXR session drops mid-game (tracking lost, app backgrounded, etc.), the phone reports `driftConfidence=0` and waits for WebXR to recover — no live hand-off to the Kalman/ZUPT pipeline mid-session.

### Platform Scope
- **D-07:** Phase 06.1 delivers **Android WebXR camera tracking only**. No browser-based iOS camera-tracking spike is attempted — MindAR/AR.js are marker-tracking only (per ROADMAP goal), and no free, full 6DOF browser alternative was identified worth spiking.
- **D-08:** iOS's real position-tracking solution is a **native ARKit companion app**, planned as a near-term follow-up — but building it is explicitly OUT of scope for 06.1. iOS continues on the ZUPT+Kalman fallback (D-04) until that native-app phase lands.

### Claude's Discretion
- Exact opaque-layer-trick implementation to skip WebXR passthrough rendering in Three.js r185.
- Specific WebXR session/frame-loop integration with the existing rAF loop in phone.ts.
- Visual design of the "Camera Tracking" / "Motion Only" badge on the calibration screen.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ZUPTDetector` (client/src/sensor/zupt.ts) and `Kalman1D` (client/src/sensor/kalman.ts) — kept as-is for the fallback path (D-04); not being removed or replaced.
- Existing calibration screen state machine in phone.ts — extend (add tracking-mode indicator, D-05), don't replace.
- `safeFloat()` in encode.ts — camera-derived position values must pass through the same guard before the DataView write.

### Established Patterns
- `POSITION_MAX`/`GESTURE_MAX` clamping (phone.ts) — camera-derived positions likely need the same bound applied before encoding.
- `_packetBuf` module-scope allocation (encode.ts) — no per-tick GC; camera tracking integration must respect this (Pitfall 5).

### Integration Points
- `startSensorPipeline()` in phone.ts is where WebXR session detection/branching plugs in alongside the existing `deviceorientation`/`devicemotion` listeners.
- `driftConfidence` is already wired end-to-end (encode → decode → playerStore → scene.ts) — reusing it (D-02) avoids touching the desktop side at all.

</code_context>
