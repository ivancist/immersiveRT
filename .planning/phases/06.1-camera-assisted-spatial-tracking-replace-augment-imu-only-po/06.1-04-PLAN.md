  <name>Task 1: On-device WebXR tracking, freeze-on-loss, and thermal verification</name>
  <action>Blocking human-verify checkpoint — no automated action. A human runs the on-device verification steps in &lt;how-to-verify&gt; on a physical ARCore Android device (and an iOS/non-ARCore device if available) and returns the &lt;resume-signal&gt;.   <what-built>
    Android WebXR (ARCore) camera-assisted 6DOF position tracking as the primary position source in the phone client, with the ZUPT+Kalman IMU pipeline retained as automatic fallback. The full desktop stack (WebTransport + WebRTC + Three.js render) already exists from Phases 4-6 and is unchanged.
  </what-built>
  <how-to-verify>
    Prerequisite: a physical ARCore-capable Android phone (Chrome, recent version, "Google Play Services for AR" installed), plus a desktop running the room client — both over HTTPS/WSS per existing dev setup (mkcert). Start the full stack (`docker compose up` or the existing dev workflow) and open the desktop room; scan the QR code with the Android phone.

    1. Calibration badge (D-05): During the "Hold your phone still" calibration screen, confirm a badge reads "Camera Tracking" on the ARCore phone. Grant the camera permission prompt when it appears.
    2. Drift resolution (SENS-V2-03): With tracking active, move the phone through a known physical path (e.g. a 1-metre square) and back to start. Confirm the desktop cube's position follows the motion and returns near the origin WITHOUT the runaway drift documented in .planning/debug/huge-position-drift-makes-pho.md. Compare against the prior IMU-only behaviour described there.
    3. Freeze-on-loss (D-06): Cover the phone camera with your hand (or background the app briefly). Confirm the desktop cube FREEZES at its last position (does not snap to origin, does not fly off), and — via the phone debug overlay / desktop HUD drift value — that driftConfidence reads 0. Uncover the camera and confirm tracking resumes without a Kalman hand-off jump.
    4. Battery/thermal escape hatch: Run the immersive-ar session continuously for 15+ minutes. Note device temperature and battery drain. Report whether heat/battery is acceptable or whether the ROADMAP escape-hatch (native companion app) should be reconsidered for Android.
    5. Fallback (D-04): If an iOS device or a non-ARCore Android device is available, scan and confirm the badge reads "Motion Only" and the phone still connects and drives position via the existing IMU pipeline (drifty but functional, as before).
  </how-to-verify>
  <resume-signal>Type "approved" if steps 1-3 and 5 pass and step 4 thermal is acceptable; otherwise describe which step failed (badge text, drift observed, snap-instead-of-freeze, thermal concern) so gap-closure planning can address it.</resume-signal>
