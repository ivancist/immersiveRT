# Phase 06.1 Plan 03: Wire WebXR Pose Tracking Into phone.ts Summary

**Camera-assisted 6DOF position becomes the primary phone position source on WebXR-capable Android (branch locked once at calibration via D-06), while the existing ZUPT+Kalman fallback runs completely unchanged on iOS/unsupported devices — same 36-byte wire packet, same clamp+safeFloat sanitisation for both sources**

## Accomplishments

- `#tracking-mode-badge` added to `client/phone.html`'s `view-calibrating` block; `onPlayerReady` sets its text to `'Camera Tracking'` or `'Motion Only'` from `isXrSupported()` before the calibration screen shows (D-05)
- `webXrSupported` module-scope flag set once per calibration; `startSensorPipeline` reads it into a closure-local `useWebXr` const decided once and never re-evaluated mid-session (D-06)
- `xrTracker = new WebXrPoseTracker(300)` constructed and `startWebXrPoseTracking(xrTracker)` fired (fire-and-forget) only when `useWebXr` is true; existing `ZUPTDetector`/`Kalman1D` construction preserved unconditionally for the fallback branch (D-04)
- `devicemotion` handler now branches the position-source read: camera branch reads `xrTracker.getState()` for `px/py/pz`, `dx/dy/dz` (D-03 continuous rolling delta, no ZUPT-gated reset) and `driftConfidence` (D-02); fallback branch is the pre-existing ZUPT+Kalman logic, moved into an `else` unchanged
- Both branches converge on the identical `clamp(POSITION_MAX)` / `clamp(GESTURE_MAX)` bounding before packet construction — camera values get the same double-guard (`clamp` + `encodePacket`'s `safeFloat()`) as Kalman values (Pitfall 4 / V5, T-06.1-02 mitigation)
- Wire format untouched: `encode.ts` `SCHEMA_VERSION` still `1`, `BUF_SIZE` still `36`; `client/src/types.ts` (the `SensorPacket` interface) not modified (D-01)

## Decisions Made

- `isStill` declared as a mutable `let isStill = false;` outside the if/else branch (rather than `const` inside only the fallback arm) so the pre-existing dev-overlay call site (`updateOverlay(pkt, mq, isStill, hz)`) needed zero changes — kept the orientation/touch/motion-indicator/dev-overlay code completely untouched, per the plan's explicit "do NOT restructure the handler" instruction.
- `wa` (`rotateDeviceToWorld` output) and `mag` (ZUPT accel magnitude) remain computed unconditionally before the branch, even though only the fallback `else` arm consumes them — left the existing unconditional pre-branch computation block as-is rather than gating it, since moving it would have meant restructuring code the plan asked to leave untouched. The extra computation is negligible at the 60-100Hz `devicemotion` rate and has no observable effect on the WebXR branch's packet output.
