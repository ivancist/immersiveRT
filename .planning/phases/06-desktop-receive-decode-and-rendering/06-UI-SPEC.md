### TAB Overlay Background

`rgba(0, 0, 0, 0.6)` тАФ semi-transparent black overlay. Not a CSS variable; inline value. Allows the Three.js scene to remain dimly visible beneath the overlay while text is legible.

Source: RESEARCH.md Pattern 5 `#game-tab-overlay` background.

### Three.js Scene Colors (data-driven, not CSS tokens)

| Element | Color Formula | Result |

## Game View Layout Contract

### Canvas Placement (D-04, D-05)

```
position: fixed; top: 0; left: 0; width: 100vw; height: 100vh;
display: none; /* activated by showGameView() on first player-ready */
```

The canvas is embedded in the existing `index.html` as `<canvas id="game-canvas">`. On `player-ready`, all existing views (`#view-lobby`, `#view-room`, `#view-phone`) are hidden via `[hidden]` attribute; the canvas and HUD become visible. The existing DOM is not removed тАФ it is hidden.

A `<div id="game-container" style="position:relative;width:100vw;height:100vh">` wraps both the canvas and the `CSS2DRenderer` domElement so label coordinates share the same stacking context.

Source: CONTEXT.md D-04, D-05; RESEARCH.md Pattern 5 (Pitfall 5).

Source: CONTEXT.md D-15; RESEARCH.md Open Question 2.

### Per-Player Name Label (D-11)

One `CSS2DObject` per player, attached as child of the player's `THREE.Mesh`, positioned at `(0, 1.2, 0)` in object-local space (above a 1-unit box). Text content: `username`. Source: CONTEXT.md D-11; RESEARCH.md Pattern 2 (CSS2DObject).

## Keyboard Interaction Contract

All keys operate as toggles or cyclers. No modifier key required. Keys active only when game view is displayed (canvas visible).

| Key | Action | Default State | HUD Reflection |
|-----|--------|---------------|----------------|
| `P` | Cycle position mode: `gestureDisplacement` тЖТ `deadReckoningPosition` тЖТ back | gestureDisplacement | `pos: gesture` / `pos: dead-reckoning` |
| `G` | Toggle grid floor (`GridHelper.visible`) | on | `G:on` / `G:off` |
| `A` | Toggle axes gizmos per object (`AxesHelper.visible` per mesh) | on | `A:on` / `A:off` |
| `H` | Toggle per-player numeric HUD panel (`display: none` toggle) | on | `H:on` / `H:off` |
| `T` / `D` | Toggle motion trail / drama mode (`THREE.Line.visible` per player) | off | `T:on` / `T:off` |
| `TAB` (hold) | Show roster overlay while held | hidden | тАФ |
| `TAB` (release) | Hide roster overlay | тАФ | тАФ |

Note on `T`/`D`: Both keys map to the same trail toggle. `D` is the "drama mode" name from CONTEXT.md D-14; `T` is the keyboard binding from D-15. Implement a single handler that responds to both `'t'` and `'d'` `key` values.

Source: CONTEXT.md D-13, D-14, D-15.

## Touch Flash Interaction Contract (D-14)

When `SensorPacket.touchActive === true` for a player:
1. Immediately set `mesh.material.emissive.setHex(0xffffff)` тАФ no delay.
2. After 100ms, set `mesh.material.emissive.setHex(0x000000)` (clear).
3. Guard against re-triggering while flash is active (`flashing` boolean per player).

This is a diagnostic feature: the instant white flash makes phone-to-screen latency perceptible at a glance. No CSS or DOM animation involved тАФ pure WebGL emissive material mutation.

Source: CONTEXT.md D-14.

## States and Transitions

| State | Visible Elements | Trigger |
|-------|-----------------|---------|
| Lobby | `#view-lobby` | Page load / before join |
| Room (waiting) | `#view-room` | After `join-ack` received |
| Game active | `#game-canvas` + `#game-hud` | First `player-ready` event |
| Game + roster | `#game-canvas` + `#game-hud` + `#game-tab-overlay` | While `TAB` is held |

Transition from Room тЖТ Game active: `showGameView()` sets `[hidden]` on `#view-lobby`, `#view-room`, `#view-phone`; removes `display:none` from `#game-canvas` and `#game-hud`. The transition is instant (no animation). The reverse (Game тЖТ Room) is out of scope for Phase 6.

### Empty State (No Phones Connected)

The game canvas activates on first `player-ready`. Before any `player-ready`:
- Existing `#view-room` remains visible with QR code and roster.
- No empty-state copy needed for the 3D view itself тАФ it is hidden.

If the game view is active but all phones disconnect (edge case, Phase 6 scope):
- HUD shows: "0/N connected"
- All player meshes remain in scene but stop updating (target-state store holds last known state).
- No "empty state" screen is shown тАФ the static last-frame of the 3D scene persists.

### Error State Copy

| Scenario | Copy |
|----------|------|
| `decodePacket` returns null (malformed or wrong version) | Silent drop тАФ no user-visible message. Console: `[decode] dropped malformed packet from {phoneId}` |
| Seq-drop (out-of-order) | Silent drop тАФ no user-visible message. Console: `[decode] dropped out-of-order seq {newSeq} (last {lastSeq}) from {phoneId}` |
| WebTransport connect fail тЖТ WS fallback | No user-visible indicator. Console: `[WT] Desktop connect failed, falling back to WS`. HUD connection status unchanged. |
| WebTransport closed unexpectedly | No user-visible indicator in Phase 6. Console: `[WT] transport closed`. WS reconnect is out of Phase 6 scope. |

Source: CONTEXT.md D-01, D-02, D-09; RESEARCH.md security domain (silent drops are correct for sensor data).
