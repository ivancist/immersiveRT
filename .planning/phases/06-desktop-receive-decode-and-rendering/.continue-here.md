## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| desktop-self-close-only | leaveRoom() calls wtTransport.close() / ws.close() on the DESKTOP side only. The phone's session keeps running. Correct flow: signal the server (or send over the data channel) to close the phone's session BEFORE tearing down desktop transport. | blocking | Send `leave-room` server message before any local close calls; verify phone shows disconnected state in manual test. |
| position-drift-accumulation | dx/dy/dz are ZUPT-processed absolute accumulated displacements. Dead-zone threshold (0.002) reduces micro-noise but does not stop drift at real-motion scales. R key applies an offset that persists until next reload — adequate for testing, not for production UX. | advisory | Investigate ZUPT threshold in phone.ts / sensor pipeline; consider resetting accumulator to zero on each `player-ready` (new session baseline). |

All three axes were inverted. Fixed: `set(-rdx, -rdz, rdy)` (dead-reckoning) and `set(-rpx, -rpz, rpy)` (gesture). User verified all three backwards; negated all three components.

**BUG 1 (blocking, was) — Leave Room: phone stays connected — FIXED**
After clicking "Leave Room", the desktop tears down its own WT/WS transport but the phone remains connected. The phone can still move the cube (which no longer exists on screen). The phone's WebRTC data channel and WT session are not closed.

Fix approach:
- Option A: Send a signaling message to the server BEFORE closing desktop transport. The server then sends a disconnect/leave notification to the phone over its WT session. Check what message type the Rust server handles (look at server/src/main.rs or equivalent for "leave", "disconnect", "close" message types).
- Option B: Send a message directly over the WebRTC data channel to the phone telling it to disconnect, then close the data channel.
- After fix: phone should show its reconnecting/disconnected UI within ~1s of desktop clicking "Leave Room".

**BUG 2 (advisory) — Position drift too high**
Even with POSITION_DEADZONE=0.002, position drift accumulates rapidly in gesture mode. The cube drifts off-screen within ~30-60 seconds of normal use.
- R key mitigates but is a workaround
- Root cause: ZUPT threshold in phone.ts may be too permissive; or accumulated dx/dy/dz values grow faster than the deadzone filters
- Consider increasing POSITION_DEADZONE to 0.005 or 0.01 as a first experiment
- Long-term: investigate phone.ts ZUPT implementation and whether resetting accumulator on player-ready would help

**BUG 3 (advisory) — Occasional rendering lag**
User reported "sometimes rendering seems to lag" — not consistently reproducible. Likely candidate: rAF loop doing too much per frame (SLERP + position + numeric HUD textContent updates + trail ring buffer all in one frame). Not blocking for phase completion but worth noting.
