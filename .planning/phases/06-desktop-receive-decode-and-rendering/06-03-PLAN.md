## Phase Goal (MVP user story)

**As a** platform developer evaluating motion precision, **I want to** see each connected phone's live orientation and position drive a distinct 3D object on my desktop with SLERP smoothing and diagnostic overlays, **so that** I can directly assess sensor fidelity and phone-to-screen latency before building a real game.

> Derived from ROADMAP.md §Phase 6 Goal + CONTEXT.md `<specifics>`.

This plan delivers the first visible increment of that story: on first player-ready the room UI gives way to a live full-viewport 3D scene (empty grid stage, ready for player cubes in plan 04).

<objective>
Provision Three.js and stand up the game view shell: install `three@0.185.1` + `@types/three@0.185.1`, add the canvas/HUD/overlay DOM and CSS to `index.html` per the UI-SPEC, create `client/src/scene.ts` with a guarded `initScene` that builds renderer + fixed camera + lights + grid floor + CSS2DRenderer and runs a requestAnimationFrame loop, and wire `room.ts` to call `showGameView()` + `initScene()` on the first `player-ready`.

Purpose: DESK-05 needs a Three.js render surface. Separating the (SUS-flagged) dependency install + DOM shell + empty-scene activation from the per-player rotation logic (plan 04) keeps each plan inside its context budget and puts a human legitimacy gate in front of the only package install in this phase.
Output: three installed, game DOM shell live, an empty 3D grid scene that activates on first player-ready.
</objective>
