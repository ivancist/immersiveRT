## Phase Goal (MVP user story)

**As a** platform developer evaluating motion precision, **I want to** see each connected phone's live orientation and position drive a distinct 3D object on my desktop with SLERP smoothing and diagnostic overlays, **so that** I can directly assess sensor fidelity and phone-to-screen latency before building a real game.

> Derived from ROADMAP.md §Phase 6 Goal + CONTEXT.md `<specifics>`.

This plan delivers the pure decode + drop + store logic that turns received bytes into per-player state — fully unit-tested, no transport or rendering coupling.

<objective>
Create the desktop's binary receive core as two dependency-free, unit-testable modules: `client/src/sensor/decode.ts` (the exact inverse of `encode.ts` plus the uint16 sequence-drop and finite-guard predicates) and `client/src/playerStore.ts` (the per-player target-state store). Written test-first: failing tests define the wire contract and store behavior, then the implementations make them green.

Purpose: DESK-03 (decode + per-sender out-of-order drop) and DESK-04 (per-player target-state store) are pure computation with no server, WebRTC, or WebGL involvement — isolating them into tested modules keeps the later room.ts wiring (plan 04) trivial and correct.
Output: decode.ts, playerStore.ts, and three green vitest suites.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-03 | Tampering | decodePacket truncated/garbage buffer | high | mitigate | `buf.byteLength < BUF_SIZE` returns null before any DataView read — no OOB read, no crash |
| T-06-04 | Tampering | decodePacket wrong schema version | medium | mitigate | byte 0 !== SCHEMA_VERSION returns null (future-phone / stale-desktop mismatch dropped) |
| T-06-05 | Spoofing | replay / delayed / out-of-order packet | medium | mitigate | isNewerSeq half-distance uint16 comparison drops duplicates and old packets per sender |
| T-06-06 | Tampering | non-finite float16 quaternion (NaN/Infinity) | high | mitigate | isSafePacket isFinite guard rejects poison quaternions before they reach any THREE.Quaternion.set (applied at render wiring) |
| T-06-07 | DoS | high-frequency packet flood | low | accept | updateTargetState overwrites a single Map entry per sender; no accumulation, no unbounded growth |
</threat_model>
