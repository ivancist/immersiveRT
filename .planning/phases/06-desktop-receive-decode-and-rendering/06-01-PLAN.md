## Phase Goal (MVP user story)

**As a** platform developer evaluating motion precision, **I want to** see each connected phone's live orientation and position drive a distinct 3D object on my desktop with SLERP smoothing and diagnostic overlays, **so that** I can directly assess sensor fidelity and phone-to-screen latency before building a real game.

> Derived from ROADMAP.md §Phase 6 Goal + CONTEXT.md `<specifics>` ("the user wants to evaluate motion precision directly"). The ROADMAP Goal line is descriptive prose, not user-story format; this story restates it faithfully without adding scope.

This plan delivers the transport foundation of that story: the desktop speaks WebTransport like the phone already does.

<objective>
Migrate `client/src/room.ts` signaling from WebSocket-only to the same WebTransport-first dual-path that `client/src/phone.ts` already uses: try WebTransport (QUIC/HTTP3, port 4433) first, fall back to WebSocket (WSS, port 9090) when WT is unsupported or fails. All signaling message types travel on one active transport (D-03). No server changes.

Purpose: DESK-01 requires the desktop to hold a persistent WebTransport connection. This is a behavior-preserving transport swap — every existing handler (join-ack, offer, ice-candidate, player-ready, room-event, pair-ack) keeps working; only the wire underneath changes.
Output: `room.ts` with ported WT helpers, a transport-agnostic `signalSend`, and WT-first connect with automatic WS fallback.
</objective>
