## Phase Goal (MVP user story)

**As a** platform developer evaluating motion precision, **I want to** see each connected phone's live orientation and position drive a distinct 3D object on my desktop with SLERP smoothing and diagnostic overlays, **so that** I can directly assess sensor fidelity and phone-to-screen latency before building a real game.

> Derived from ROADMAP.md §Phase 6 Goal + CONTEXT.md `<specifics>`.

This is the payoff plan: real phone motion becomes a smoothly rotating cube on the desktop, one per player.

<objective>
Close the sensor hot path on the desktop: set the WebRTC data channel to binary, wire `ondatachannel` to decode → finite-guard → seq-drop → store, and give `scene.ts` real per-player boxes that SLERP toward the latest orientation and follow the active position mode each frame. After this plan, moving a phone rotates its box on screen, and two phones drive two independent boxes.

Purpose: DESK-02 requires the desktop to accept each phone's channel and consume its binary packets with no server relay; DESK-05 requires SLERP interpolation of orientation in the Three.js loop. This plan connects the tested decode/store core (plan 02) to the live scene (plan 03).
Output: rotating, per-player colored boxes driven by real decoded sensor packets.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| phone → desktop (WebRTC data channel) | Untrusted binary packets consumed directly P2P and applied to scene state |
| decoded packet → THREE render | Packet-derived numbers drive quaternion/position — poison values could break rendering |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-09 | Tampering | non-finite quaternion applied to mesh | high | mitigate | isSafePacket finite-guard in dc.onmessage rejects the packet before updateTargetState / any slerp — a NaN quaternion never reaches THREE |
| T-06-05b | Spoofing | out-of-order / replayed packet applied to store | medium | mitigate | isNewerSeq per-sender check in dc.onmessage drops backward/duplicate seq before store update |
| T-06-10 | Injection | username rendered in CSS2D name label | medium | mitigate | label uses `textContent` (not innerHTML) — no HTML/script injection via player name |
| T-06-11 | Tampering | Firefox delivers packet as Blob (decode fails) | low | mitigate | dc.binaryType set to 'arraybuffer' as first statement before channel opens (Pitfall 3) |
</threat_model>
