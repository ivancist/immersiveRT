<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form→WS join-room | Untrusted room code and username entered by user; client-side validation is UX only; server validates authoritative |
| QR URL→phone browser | HTTPS required for camera-app auto-open; HTTP QR will silently fail on iOS (Pitfall 5) |
| sessionStorage reconnect_token | Stored client-side; must be treated as a credential; room.js must not log its value to console |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-02 | DoS | Lobby form rapid join-room submission | medium | mitigate | btn-join-submit is disabled immediately on click (pointer-events: none) until join-ack/join-error received. Prevents client from sending multiple join-room messages from one connection. Server enforces max 8 per room regardless. |
| T-03-09 | Information Disclosure | reconnect_token logged to console | low | mitigate | room.js must not call console.log with the reconnect_token value. Log only "Reconnect token stored" without the value. Review all console.log calls. |
| T-03-10 | Spoofing | pairing_url HTTP vs HTTPS in QR | high | mitigate | pairing_url comes from server join-ack; server constructs it from BASE_URL env var. If BASE_URL starts with http:// (not https://), QR scan will not auto-open on iOS. The BASE_URL required-var check and nginx HTTPS (Plan 03-03) together prevent this. No client-side mitigation needed; trust the server URL. |
</threat_model>
