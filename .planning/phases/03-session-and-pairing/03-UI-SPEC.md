## Spacing Scale

Declared values (all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-adjacent gaps, status dot margins |
| sm | 8px | Input internal padding, button vertical padding |
| md | 16px | Default element spacing, card internal padding |
| lg | 24px | Section padding, between QR card and short code block |
| xl | 32px | Between major page sections |
| 2xl | 48px | Lobby card top/bottom padding |
| 3xl | 64px | Page-level max-width breathing room |

Exceptions:
- Status dots: 8×8px circle (2px — intentionally sub-scale for inline affordance)
- QR canvas: fixed 256×256px (library-constrained, not on spacing scale)
- Touch target minimum: 44px height on all interactive elements (mobile-browser requirement for phone landing page — /phone view)

## Typography

All sizes apply to both UI-font and monospace-font roles; monospace is used only for room codes and short codes.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Label | 13px | 600 | 1.4 | Form labels, status text, event log timestamps, metadata |
| Body | 15px | 400 | 1.5 | Paragraphs, event log entries, error messages |
| Heading | 20px | 600 | 1.2 | Section headers, card titles, "Your Room: ABCD23" |
| Display | 28px | 600 | 1.1 | Room code block, short code display (monospace font at this size) |

Weight declarations: Regular (400) + Semibold (600). No other weights used.

*Source: Default scale — no upstream artifact specified typography. Chosen to match "developer tool" context (RESEARCH.md "Lobby and room UI" capability) with legibility priority.*

## Color

Theme: Dark (game developer tool aesthetic — game platforms conventionally use dark UI).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#111111` | Page background |
| Secondary (30%) | `#1e1e1e` | Cards, panels, input backgrounds, event log container |
| Accent (10%) | `#7c6af7` | Primary CTA buttons (Create Room, Join Room, Continue, Join), own-slot roster highlight border, QR card outer border |
| Destructive | `#ef4444` | Error messages, room-full badge, rejected slot indicator |
| Text primary | `#f0f0f0` | All standard body and heading text |
| Text secondary | `#888888` | Timestamps, placeholder text, empty slot labels, secondary metadata |
| Status: connected | `#22c55e` | Connected slot dot |
| Status: hold (reconnecting) | `#eab308` | Disconnected-hold slot dot (animated — see Interaction States) |
| Status: empty | `#444444` | Empty slot dot |

Accent reserved for: Create Room button, Join Room button, Continue button, Join Room submit button, own-slot row left-border highlight, QR card outer border (2px). Nowhere else.

Exception — QR canvas section: The `<canvas>` element itself and its immediate containing `<div>` use `background: #ffffff` regardless of theme. The QR code must render dark-on-white for camera-app scannability. The white box is scoped to the canvas wrapper only; the outer card remains `#1e1e1e`.

*Source: Dark theme — inferred from game developer tool context (RESEARCH.md). No upstream artifact specified colors. Destructive value derived from error codes in D-10 (join-error: room_full, room_not_found). QR white requirement from CONTEXT.md Specifics and RESEARCH.md Pitfall 5.*

| Element | Copy |

### Error States

| Trigger | Copy |
|---------|------|
| join-error: room_not_found | Room not found. Double-check the code and try again. |
| join-error: room_full | This room is full (8/8 players). Ask the host for a different code. |
| join-error: (any other reason) | Could not join room. Check your connection and try again. |
| Connection to server failed | Connection failed. Check your network and try again. |
| pair-error: token_expired | This pairing code has expired. Ask the desktop player to share a new one. |
| pair-error: token_used | This pairing code has already been used. Ask the desktop player to share a new one. |
| pair-error: (any other reason) | Pairing failed. Try scanning the QR code again. |

### Lifecycle Event Log Text

| Event type (D-21) | Log entry text |
|---|---|
| player-joined | {username} joined — slot {N} |
| player-left (hold expired) | {username} left — slot {N} |
| player-reconnected | {username} reconnected — slot {N} |
| room-full | Room is full (8/8 players) |
| player-disconnected (hold started) | {username} disconnected — waiting 60s for reconnect |

Format: `[HH:MM:SS] {text}` in label size, timestamp in text-secondary color.

### Phone Landing Page (minimal — full phone app is Phase 4)

| State | Copy |
|-------|-------|
| Pairing in progress | Pairing your phone... |
| Pairing succeeded | Paired successfully. Phone app coming in Phase 4. |
| Pairing failed | Pairing failed: {reason}. Go back and scan again. |

### Destructive Actions

None in Phase 3. No room deletion, no player kick, no slot force-release from the client side.

## UI Surfaces

Phase 3 delivers three client surfaces served as a single-page app (SPA) via `history.pushState`. All surfaces are in `client/dist/index.html` + `client/dist/room.js`.

### Surface 1: Lobby (`/`)

**Layout:** Centered card (max-width 480px) on full-page dominant background. Vertical flex. Top-to-bottom: title, subtitle, two CTA buttons.

**Structure:**
```
[Page background: #111111]
  [Card: #1e1e1e, padding: 2xl (48px), border-radius: 8px]
    [Title: "ImmersiveRT" — heading size, text-primary]
    [Subtitle: "Create or join a room to get started." — body size, text-secondary]
    [Button row: flex, gap: md (16px)]
      [Create Room — accent bg, white text, full-width]
      [Join Room   — accent bg, white text, full-width]
    [Form area: hidden initially, revealed on button click]
```

**Create Room sub-form (revealed on "Create Room" click):**
```
[Label: "Game Type" — label size]
[Select: <select> with single option "Placeholder Game" — secondary bg, 1px secondary border]
[Submit: "Continue" button — accent, full-width]
[Loading state: button text "Creating..." + disabled]
```
On join-ack: `history.pushState` to `/room/{room_code}`, render Surface 2.

**Join Room sub-form (revealed on "Join Room" click):**
```
[Label: "Room Code" — label size]
[Input: 6-char uppercase, auto-uppercase on keyup, placeholder "ABCD23" — secondary bg]
[Label: "Username" — label size]
[Input: 1-64 chars, placeholder "Your name" — secondary bg]
[Submit: "Join Room" button — accent, full-width]
[Error area: hidden until error, destructive text color, body size, below room code input]
```
On join-ack: `history.pushState` to `/room/{room_code}`, render Surface 2.
On join-error: show error message in error area.

**Input validation (client-side, pre-submit):**
- Room code: strip non-CHARSET chars (`[^A-Z2-9]` after toUpperCase), max 6 chars, min 1 char
- Username: trim whitespace, min 1 char, max 64 chars, printable ASCII only

**Pre-warming (D-11):** WS/WT connection opened when lobby page loads — before user clicks any button. No UI indicator needed; happens silently.

### Surface 2: Room Page (`/room/{code}`)

**Layout:** Two-column on viewports >= 768px; single-column stacked on < 768px. Max-width 960px, centered.

**Left column (QR + codes):**
```
[Card: #1e1e1e, padding: lg (24px), border: 2px solid #7c6af7 (accent), border-radius: 8px]
  [Label: "Scan to pair your phone" — heading size, text-primary, margin-bottom: md]
  [QR wrapper: background: #ffffff, padding: 8px, border-radius: 4px, display: inline-block]
    [<canvas id="pairing-qr"> — 256×256px]
  [Separator: margin: lg (24px) 0]
  [Label: "Or type this code on your phone:" — label size, text-secondary]
  [Short code: "{ROOM_CODE}-{SLOT_ID}" — monospace, display size (28px), weight 600, text-primary, letter-spacing: 0.1em]
```

QR render call (exact options, no deviation):
```javascript
QRCode.toCanvas(
  document.getElementById('pairing-qr'),
  pairingUrl,
  { width: 256, margin: 2, color: { dark: '#000000', light: '#ffffff' }, errorCorrectionLevel: 'M' },
  (err) => { if (err) console.error('[QR]', err); }
);
```

**Right column (roster + events):**
```
[Section: "Players" — heading size]
[Roster: 8 slot rows, vertical list, gap: sm (8px)]
  [Slot row (each): flex, align-items: center, gap: sm (8px), padding: sm (8px)]
    [Status dot: 8×8px circle, color by state]
    [Slot number: "Slot {N}" — label size, text-secondary, min-width: 48px]
    [Username: body size, text-primary | "Empty" in text-secondary | "{username} (reconnecting...)" in text-secondary]
  [Own slot: left-border 3px solid #7c6af7 (accent), background: rgba(124,106,247,0.08)]
[Separator]
[Section: "Room Events" — heading size]
[Event log: scrollable container, max-height: 200px, overflow-y: auto, #1e1e1e bg, padding: sm]
  [Each entry: "[HH:MM:SS] {text}" — label size, timestamp in text-secondary, text in text-primary]
  [Max 50 entries — oldest discarded when limit reached]
  [Auto-scrolls to bottom on new entry]
```

**Room page header (above columns):**
```
[Room title: "Your Room: {ROOM_CODE}" — heading size (20px)]
[Room code: shown inline in monospace at heading size, text-primary]
[Connection status: text-secondary, label size — "Connected" | "Reconnecting..." | "Disconnected"]
```

### Surface 3: Phone Landing (`/phone?token=...`) — Phase 3 scope: pairing handshake only

**Layout:** Single centered card (max-width 400px), full-page dominant background. Optimized for mobile viewport (< 480px wide screens). Min button height: 44px.

**States:**
```
Pairing state (initial):
  [Card: #1e1e1e, padding: xl (32px)]
    [Title: "Pairing your phone..." — heading size]
    [Spinner: CSS-only animation (see Interaction States)]
    [Hint: "Keep this page open." — body size, text-secondary]

Success state (pair-ack received):
  [Card: #1e1e1e]
    [Title: "Paired successfully." — heading size]
    [Body: "Phone app coming in Phase 4. Keep this tab open." — body size, text-secondary]

Error state (pair-error or token expired/used):
  [Card: #1e1e1e]
    [Title: "Pairing failed" — heading size, destructive color (#ef4444)]
    [Error message: {human-readable reason from copy table above} — body size]
    [Hint: "Go back and scan the QR code again." — label size, text-secondary]
```

No navigation elements on this surface. No "Back" button — user exits via browser back gesture.

## Interaction States

### Buttons

| State | Style |
|-------|-------|
| Default | background: `#7c6af7`, color: `#ffffff`, border: none, border-radius: 6px, padding: 10px 20px |
| Hover | background: `#6a58e8` (10% darker accent), cursor: pointer |
| Active / Pressed | background: `#5946d9` (15% darker accent) |
| Disabled / Loading | opacity: 0.5, cursor: not-allowed, pointer-events: none |
| Focus (keyboard) | outline: 2px solid `#7c6af7`, outline-offset: 2px |
| Loading text | button label replaced with "{action}..." (e.g., "Creating...", "Joining...") |

### Inputs

| State | Style |
|-------|-------|
| Default | background: `#1e1e1e`, border: 1px solid `#444444`, color: `#f0f0f0`, border-radius: 4px, padding: 8px 12px |
| Focus | border: 2px solid `#7c6af7`, outline: none, padding: 7px 11px (compensate for wider border) |
| Error | border: 2px solid `#ef4444`, padding: 7px 11px |
| Placeholder text | color: `#888888` |

Error message element: `display: block`, color: `#ef4444`, font-size: 13px (label), margin-top: 4px, hidden by default, shown via `display: block` on error.

### Select Element (game type)

Same style as inputs. Uses OS-native select widget (no custom dropdown for Phase 3).

### Status Dots

| State | Color | Animation |
|-------|-------|-----------|
| Connected | `#22c55e` | none |
| Disconnected — hold window active | `#eab308` | CSS keyframe: `opacity 1→0.3→1`, duration 1.5s, iteration: infinite, timing: ease-in-out |
| Empty | `#444444` | none |

CSS for pulsing dot:
```css
@keyframes dot-pulse {
  0%, 100% { opacity: 1; }
  50%       { opacity: 0.3; }
}
.dot--hold { animation: dot-pulse 1.5s ease-in-out infinite; }
```

### CSS-Only Loading Spinner (no icon library)

Used on phone landing page (pairing state) and during server round-trips:
```css
@keyframes spin {
  to { transform: rotate(360deg); }
}
.spinner {
  display: inline-block;
  width: 24px;
  height: 24px;
  border: 3px solid #444;
  border-top-color: #7c6af7;
  border-radius: 50%;
  animation: spin 0.8s linear infinite;
}
```

### Slot Row — State Transitions

| Slot state | Username text | Dot color | Row style |
|------------|--------------|-----------|-----------|
| Empty | "Empty" in text-secondary (`#888888`) | `#444444` | Default row bg |
| Connected | `{username}` in text-primary | `#22c55e` | Default row bg |
| Hold (reconnecting) | `{username} (reconnecting...)` in text-secondary | `#eab308` pulsing | Default row bg |
| Own slot (any connected state) | Same as above | Same as above | Left-border 3px `#7c6af7`, bg `rgba(124,106,247,0.08)` |

Transition on hold expiry (timer fires, player-left event received):
1. Dot: yellow pulsing → grey static
2. Username text: `{username} (reconnecting...)` → "Empty" in text-secondary
3. Event log appended: `[HH:MM:SS] {username} left — slot {N}`

### Event Log

- New events appended to bottom
- Container auto-scrolls to bottom on each append
- Maximum 50 entries; on overflow, `removeChild(firstChild)` before `appendChild(newEntry)`
- No user interaction (read-only)

### QR Code Fallback

If `QRCode.toCanvas()` returns an error (e.g., CDN failed to load), show a text fallback in the QR wrapper:
```
[Short pairing URL in monospace, body size, dark text on white bg]
"Open: {pairingUrl}"
```
This lets the user manually type the URL if the QR is unavailable.

No third-party shadcn registry blocks. No npm install in Phase 3 client. CDN is the only external client dependency.

CDN integrity note: Executor should consider adding a `<script integrity="sha384-...">` SRI hash for `qrcode.min.js` if the CDN URL is pinned to `@1.5.4`. This prevents CDN compromise from injecting malicious code. Executor can obtain hash via `openssl dgst -sha384 -binary qrcode.min.js | base64`.
