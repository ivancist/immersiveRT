## Phase Goal

**As a** third-party game developer, **I want to** add a one-line latency overlay, **so that** I can see per-player latency, jitter, packet-loss, and ICE state on the desktop without any extra configuration.

> Slice: the developer overlay — the SDK's single deliberate DOM exception (D-02). After this plan a consumer calls `platform.attachLatencyOverlay()` and sees live per-player transport metrics, satisfying SDK-04 with the research-corrected metric sources.

<objective>
Implement `attachLatencyOverlay(container?)` (D-02) and its metrics. Latency from the packet `timestamp` field vs `Date.now()`; RTT from `getStats()` candidate-pair; ICE state from the live `pc.iceConnectionState`; and — per RESEARCH Pitfall 1 — jitter and packet-loss computed application-side from the packet's existing `seq`/arrival-time data, NOT from `getStats()` (which cannot provide them for a data-channel-only connection). No wire-schema change (D-16 outcome preserved).

Purpose: SDK-04 requires a single-line-include overlay rendering rolling avg latency, jitter, packet-loss %, and ICE state per player. This corrects D-16's literal mechanism while delivering exactly what it wanted.
Output: latencyOverlay.ts + a per-sender loss/jitter tracker fed by the packet path + jsdom DOM tests.
</objective>
