## Phase Goal

**As a** third-party game developer, **I want to** poll `getPlayerInput(id)` / `getRawInput(id)` and subscribe with `on(...)` against typed APIs, **so that** I can drive a Three.js game from phone motion without reading platform source.

> Slice: the public API core. After this plan a consumer can (in a unit test and a strict-typecheck fixture) read the exact 5-field player input, opt into raw orientation, and subscribe to lifecycle + IMU events — the headless heart of the SDK (D-01). Live data arrives once transport lands in 07-04; here the API is proven by driving the store directly.

<objective>
Build the SDK's own rAF/interval-driven interpolation tick (D-08) and the `Platform` class that wraps a native `EventTarget` behind a typed `.on()/.off()` facade (D-12), exposing `getPlayerInput()` (SDK-02), `getRawInput()` (SDK-06), and the `imuUpdate`/`playerJoin`/`playerLeave`/`playerReconnect` events (SDK-03). Add a consumer typecheck fixture proving `tsc --strict` compiles a game against the published types (SDK-01, ROADMAP success criterion 1).

Purpose: This is the only consumer-facing surface of the headless SDK. It must apply SLERP to orientation only (D-10), hold on gaps (D-11), fire imuUpdate per tick (D-14), and keep raw vs interpolated state separate (SDK-06).
Output: tick.ts, platform.ts, their tests (both scheduling paths), and a strict-typecheck fixture.
</objective>
