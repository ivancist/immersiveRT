# Phase 06.2 Plan 04: WebSocket Signaling Fallback Transport Summary

**`WebSocketSignaling` — a zero-dependency `URLSessionWebSocketTask` conformance of `SignalingTransport`, ported from `phone.ts`'s `connectPhoneWS`/`sendWsMsg`/`onPhoneWsMessage`, guaranteeing a working transport regardless of the WebTransport spike outcome (D-05 fallback-of-last-resort).**

## Accomplishments
- `WebSocketSignaling` implements `connect()`/`send(_:)`/`request(_:)`/`close()` against `URLSessionWebSocketTask`, with URL always built from an injected host (`wss://{host}:9090`) — never `location.hostname`, never hardcoded (Pitfall 5)
- `myId` is stored once (`let`) and reused across every register/reconnect, satisfying the Shared Pattern
- `request(pair)`/`request(reconnect)` round-trip via `CheckedContinuation`, resolved/rejected from a directly testable `handle(_:)` dispatcher on `pair-ack`/`pair-error`/`join-ack`/`join-error`; all other envelope types route to `onServerPush`
- Close handling (`handleClosure(_:)`) always rejects any in-flight request with `ws-closed`, and conditionally suppresses the `onClosed` "ended" signal via a manager-settable `isReconnecting` flag — matching phone.ts's asymmetric `_pairReject`-always / `showView('view-ended')`-guarded behavior
- 17 unit tests exercise all of the above without a real socket, using synthetic envelopes and a bounded polling helper (`waitUntil`) to deterministically synchronize with the unstructured `Task` created inside `request(_:)`

## Decisions Made
- `WebSocketSignaling` takes `host: String` via constructor injection instead of calling `QRTokenParser.host(from:)` internally. The plan's own `files_modified` list scopes this plan to `Transport/WebSocketSignaling.swift` + its test file only, and `QRTokenParser.host(from:)` is owned by the parallel wave-2 Plan 02 (not a declared dependency of this plan — `depends_on: [06.2-01]` only). Constructor injection satisfies every acceptance criterion ("URL built from injected host... never hardcoded") without creating a cross-worktree file-ownership conflict; the future `TransportManager` (Plan 06) is expected to call `QRTokenParser.host(from:)` and pass the result in.
- `onClosed` suppression is implemented as a settable `isReconnecting` flag consulted only inside `handleClosure(_:)`'s final `onClosed?(reason)` call — the continuation-rejection behavior (`resolvePair`/`resolveReconnect` with `.requestFailed(reason: "ws-closed")`) is unconditional and runs before that check, so a managed reconnect never weakens the "in-flight request always resolves/rejects" guarantee, only the terminal-UI-facing signal.
- `handle(_:)` and `handleClosure(_:)` are internal (not `private`) specifically to be testable via `@testable import` with synthetic envelopes, avoiding the need for a `URLProtocol`-based WebSocket mock.
