## Decisions Made

See `06.2-SPIKE-FINDINGS.md` for full detail on each bug. Key decisions:
- WT is confirmed working and kept as the primary transport (D-04 ordering unchanged: WT first, WS fallback)
- The desktop web client's packet-decode error was flagged but explicitly NOT fixed — out of scope for this iOS-only phase
- An early concurrent-polling approach to dead-connection detection was tried, found to cause a worse regression (broke fresh connects, not just reconnects — likely Network.framework's declarative QUIC API isn't safe for concurrent `conn.state` access while streams are in flight), and was reverted in favor of a synchronous check on the existing send path

## Deviations from Plan

### Auto-fixed Issues (Rule 3 — blocking; discovered via on-device verification, not simulator-testable)

All six bugs in `06.2-SPIKE-FINDINGS.md` were blocking issues discovered only through real on-device testing (Xcode console + server logs), each of which prevented completing the plan's own Task 3/5/6 checkpoints. Fixed directly rather than deferred, since VALIDATION.md and this plan's own `<verification>` section require these checkpoints to actually pass, not just be attempted. Summarized:

1. **QR scanner double-fire → connection stomping** — `AVCaptureMetadataOutput` delegate fired repeatedly per scan; fixed with a one-shot guard.
2. **Ack/error envelope decode failures** — server's ad-hoc JSON responses omit `from`/`to`; fixed with lenient `Codable` decode.
3. **WT control stream lifetime bug** — local variable, not retained; fixed by storing on `self`.
4. **WT extended-CONNECT status-200 detection too narrow** — only recognized one of two valid QPACK encodings; fixed with a minimal field-line walker + regression tests against real captured bytes.
5. **Dead WT connection undetected after backgrounding** — first attempt (concurrent polling Task) caused a regression, reverted; final fix checks `conn.state` synchronously on the existing send path.
6. **Heartbeat not sent immediately on resume** — implemented the behavior the code's own doc comment already claimed but didn't do.

**Total deviations:** 6 auto-fixed (all Rule 3 — blocking, discovered during physical-device verification).
**Impact on plan:** All six were necessary for the phase's own must-haves (WebRTC fan-out working, WT/WS transport parity, Wake Lock/reconnect correctness) to actually be verifiable, not scope creep.

## Issues Encountered

Iterative on-device debugging required more back-and-forth than a typical automated task — each fix needed a real rebuild + physical-device retest cycle from the user. One fix attempt (concurrent connection-state polling) was itself a regression and had to be diagnosed and reverted. Not all findings were airtight on the first pass — one early theory (a 107-second WT hang) was later walked back after the user correctly pointed out the evidence was ambiguous (see SPIKE-FINDINGS.md's "Correction" section).
