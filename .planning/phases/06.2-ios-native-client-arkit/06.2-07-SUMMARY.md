**Swift port of `client/src/phone.ts`'s `startPhoneClient()` + `attemptReconnect()` — the connect->pair->fan-out->stream->reconnect orchestrator, with WT-first/WS-fallback ordering (D-04) enforced end-to-end and a 13-attempt reconnect loop matching the web client's retry shape exactly.**

## Accomplishments
- Implemented `TransportManager`, a `final class` orchestrating WT-first/WS-fallback connect (D-04), pairing, WebRTC data-channel fan-out (one `PeerConnectionManager.openChannel` per peer), heartbeat startup, and a CoreMotion-driven orientation-only sensor encode/send loop (D-01/D-09 — position/gesture/drift stay zero)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Unwrapped `WebSocketSignalingError.requestFailed(reason:)` instead of collapsing it into a generic message**
- **Found during:** Task 1 (`start()`'s pairing) and Task 2 (`tryWsReconnect()`)
- **Issue:** `WebSocketSignaling.request(_:)` rejects a pending `pair`/`reconnect` continuation by THROWING `WebSocketSignalingError.requestFailed(reason:)` on a structured `pair-error`/`join-error` server response — unlike `WebTransportSignaling`, which returns the same failure shape as a normal envelope. A naive blanket `catch` would discard the server-provided reason: for `start()`, every WS pairing failure (even ones the server explained, e.g. "room_full") would show the generic "Server connection dropped during pairing" message instead of the real reason phone.ts surfaces (`.catch(reason => ({ type: 'pair-error', payload: { reason } }))`). For the reconnect loop, it's a correctness bug, not just a UX one: a genuine terminal `join-error` reason (e.g. "banned") would have been collapsed into the always-retryable `"ws-closed"` bucket, causing the loop to retry to the 13-attempt ceiling instead of terminating on the first iteration.
- **Fix:** `start()` now catches `WebSocketSignalingError` specifically and synthesizes the equivalent `pair-error` envelope so the reason reaches the same extraction logic as the WT path. `tryWsReconnect()` now separates the `connect()` failure path (legitimately `"ws-closed"`, no server reason exists) from the `request()` failure path (unwraps `.requestFailed(reason:)`).
- **Files modified:** `mobile/ios-app/immersiveRT/Transport/TransportManager.swift`
- **Verification:** Regression tests `test_start_wsPairError_surfacesServerProvidedReason` and `test_attemptReconnect_wsJoinError_preservesReason_notCollapsedToWsClosed`, both passing.
- **Committed in:** `dcb5b60` (Task 1 GREEN), `19a7149` (Task 2 GREEN)

**2. [Rule 2/3] Added `PeerConnectionManager.openDataChannels`**
- **Found during:** Task 1 (sensor loop implementation)
- **Issue:** The sensor encode/send loop needs to broadcast one encoded packet to every currently-open "sensor" data channel, but `PeerConnectionManager`'s per-peer state map is (correctly) `private`, and this file wasn't in Plan 07's declared `files_modified` list.
- **Fix:** Added a small read-only computed property (`openDataChannels: [RTCDataChannel]`) projecting only the open channels — `peers` itself stays private, no roster leak. Reviewed and confirmed reasonable in-scope extension per the executor prompt's explicit guidance.
- **Files modified:** `mobile/ios-app/immersiveRT/WebRTC/PeerConnectionManager.swift`
- **Verification:** Full `immersiveRTTests` suite green, including existing `PeerConnectionManagerTests`.
- **Committed in:** `dcb5b60` (Task 1 GREEN)

**3. [Rule 1 - Test-infra flake] Fixed a nondeterministic test-double race**
- **Found during:** Task 2, while confirming reconnect-loop tests were reliably green (not just once)
- **Issue:** The reconnect tests' fast/virtual `TransportManagerClock` initially returned with zero suspension. `withTransportTimeout`'s internal `withThrowingTaskGroup` races the real operation against `clock.sleep(seconds)` as the timeout arm — with a zero-length sleep, that race against `ScriptedSignalingTransport`'s equally near-instant `connect()`/`request()` became genuinely nondeterministic at the Swift concurrency scheduler level, flaking roughly 1-in-4 runs (`test_start_wtFails_wsSucceeds_activeTransportIsWS`, `test_attemptReconnect_wsJoinError_preservesReason_notCollapsedToWsClosed`, `test_attemptReconnect_slotNotHeld_skipsWSSubAttempt` each failed in different runs).
- **Fix:** The test clock now sleeps a nominal 5ms instead of zero — still negligible test wall-clock time (13 retry-ceiling iterations cost ~65ms total) while reliably losing the timeout race to any non-timing-out fake operation.
- **Files modified:** `mobile/ios-app/immersiveRTTests/TransportManagerTests.swift`
- **Verification:** 4 consecutive full-suite runs, all green, after the fix (previously 2 of 3 runs had a different spurious failure each time).
- **Committed in:** `19a7149` (Task 2 GREEN)
