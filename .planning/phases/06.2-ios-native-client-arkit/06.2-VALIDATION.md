---

## Sampling Rate

- **After every task commit:** Run whatever XCTest unit tests exist for pure-logic pieces (packet encoder, state-machine transitions with fake transports) — simulator-safe, no CoreMotion/real network required
- **After every plan wave:** Full XCTest suite (once it exists) + confirm the app still builds for a **real device** target, not just Simulator (CoreMotion/WebRTC/`Network.framework` code behaves differently, or isn't available at all, in the Simulator)
- **Before `/gsd-verify-work`:** Full XCTest suite green AND the `checkpoint:human-verify` on-device tasks (WebTransport spike result, CoreMotion axis verification, WebRTC fan-out to a real desktop) completed
- **Max feedback latency:** ~60 seconds

## Manual-Only Verifications

| Behavior | Requirement | Why Manual | Test Instructions |
|----------|-------------|------------|-------------------|
| WebTransport handshake success (open QUIC/HTTP3 connection, complete WT-over-HTTP/3 handshake, round-trip a JSON envelope) | D-04/D-05 | Requires a real network path, TLS trust, and the live dev server — inherently an integration/interop test against infrastructure outside the app's own control; not simulator-testable | On the physical iPhone (mkcert CA trusted per Pitfall 2), attempt the WT handshake against the running dev server; confirm a bidirectional stream opens and a `register` envelope round-trips (or the connection stays open, matching `wt_server.rs`'s no-response-on-register behavior). If it fails within the spike's time-box, fall back to WS-only and document findings. |
| CoreMotion axis mapping correctness (Pitfall 1) | D-09 | CoreMotion sensors are unavailable in the iOS Simulator; requires the physical iPhone (D-07) | Rotate the phone through known reference rotations (90° yaw, 90° pitch, 90° roll); confirm the rendered desktop cube rotates in the matching direction/axis, comparing against the web client's existing behavior on the same physical device where possible. |
| WebRTC fan-out + DTLS passive-role SDP patch behavior on the real `stasel/WebRTC` iOS stack | PHONE-03 | Native DTLS stack behavior may differ from Safari's; needs a real peer connection to a real desktop, not simulator-testable | Pair a real iPhone running the native app to a real desktop browser session; confirm the `RTCDataChannel` opens and packets arrive/decode correctly on the desktop. |
| mkcert CA trust chain installed on the physical test iPhone | Pitfall 2 (infra prerequisite) | One-time manual device-trust step; not exercisable via automated test | Export mkcert root CA (`mkcert -CAROOT`), transfer to iPhone (AirDrop or hosted URL), install as a configuration profile (Settings → General → VPN & Device Management), then enable full trust (Settings → General → About → Certificate Trust Settings). |
