## Pattern Assignments

### `Transport/SignalingEnvelope.swift` (model, transform)

**Analog:** `client/src/phone.ts` / `client/src/room.ts` message shape (used throughout both files, e.g. `phone.ts` lines 156-162, 285-288, 349-352)

**Core shape to replicate:**
```typescript
// phone.ts:156-162 — every signaling call takes this shape
function signalSend(type: string, to: string, payload: object): void {
  if (useWt && transport) {
    sendWtMessage(transport, { type: type, from: myId ?? '', to: to || '', payload: payload as Record<string, unknown> });
  } else {
    sendWsMsg(type, to, payload);
  }
}
```
Envelope is always `{ type: string, from: string, to: string, payload: Record<string, unknown> }`, JSON-serialized (`JSON.stringify(envelope)` in `sendWtMessage`/`sendWtRequest`, `phone.ts:170,190`). Swift `Codable struct SignalingEnvelope { let type: String; let from: String; let to: String; let payload: [String: AnyCodable] }` — payload needs an `AnyCodable`-style wrapper since Swift `Codable` has no native `Record<string, unknown>` equivalent. Known message `type` values to model (from `room.ts` grep): `register`, `pair`, `pair-ack`, `pair-error`, `reconnect`, `join-ack`, `join-error`, `heartbeat`, `offer`, `answer`, `ice-candidate`, `rtc-channel-ready`, `phone-state`, `player-ready`, `peer-joined`, `peer-left`.

### `Transport/WebSocketSignaling.swift` (service, event-driven + request-response)

**Analog:** `client/src/phone.ts` lines 251-313

**URL construction pattern** (line 252) — **MUST replace `location.hostname` with the scanned QR host** (Pitfall 5):
```typescript
const url = 'wss://' + location.hostname + ':9090';
```
Native equivalent: `wss://\(scannedHost):9090` via `URLSessionWebSocketTask`.

**Connect + register pattern** (lines 251-283):
```typescript
function connectPhoneWS(onReadyCallback: (() => void) | null): void {
  ws = new WebSocket(url);
  ws.onopen = function() {
    if (!myId) { myId = crypto.randomUUID(); }  // preserve myId across reconnects
    ws!.send(JSON.stringify({ type: 'register', from: myId, to: '', payload: {} }));
    wsReady = true;
    registered = true;
    if (onReadyCallback) { onReadyCallback(); }
  };
  ws.onmessage = function(evt) { onPhoneWsMessage(JSON.parse(evt.data)); };
  ws.onclose = function() {
    wsReady = false; registered = false;
    if (_pairReject) { _pairReject('ws-closed'); ... }
    if (_reconnectReject) { _reconnectReject('ws-closed'); ... }
    if (heartbeatInterval !== null) { clearInterval(heartbeatInterval); heartbeatInterval = null; }
    if (!_reconnecting) { showView('view-ended'); }
  };
}
```
Key behaviors to replicate: (1) `myId` persists across reconnects (generate once, reuse), (2) register message sent immediately on open, (3) close handler rejects any in-flight pair/reconnect promise/continuation with `'ws-closed'`, (4) close handler suppresses the "ended" UI state while a managed reconnect loop (`_reconnecting`) is active.

**Message dispatch pattern** (lines 290-313): `pair-ack`/`pair-error` resolve/reject a pending pair continuation; `join-ack`/`join-error` resolve/reject a pending reconnect continuation; everything else routes to `handleServerPush`. Swift: use `CheckedContinuation` stored in optional ivars mirroring `_pairResolve`/`_pairReject`/`_reconnectResolve`/`_reconnectReject`.

### `Transport/WebTransportSignaling.swift` (service, streaming + request-response)

**Analog:** `client/src/phone.ts` lines 164-248 (WT helpers) + 334-368 (`tryWtReconnect`)

**Request/response over bidi stream** (lines 167-184) — one-shot request pattern to replicate structurally (open stream → write JSON → close write side → read until done → parse JSON response):
```typescript
async function sendWtRequest(t: WebTransport, envelope: SignalingMessage): Promise<SignalingMessage> {
  const stream = await t.createBidirectionalStream();
  const writer = stream.writable.getWriter();
  await writer.write(new TextEncoder().encode(JSON.stringify(envelope)));
  await writer.close();
  const reader = stream.readable.getReader();
  let buf = new Uint8Array(0);
  while (true) {
    const chunk = await reader.read();
    if (chunk.done) { break; }
    // ...append chunk to buf...
  }
  return JSON.parse(new TextDecoder().decode(buf)) as SignalingMessage;
}
```
Fire-and-forget variant (`sendWtMessage`, lines 187-198) is identical but discards the parsed response — still drains the readable side to release server-side backpressure.

**Server-push listener** (lines 204-248): reads `t.incomingBidirectionalStreams` via `.getReader()` (NOT async iteration — comment at 200-203 explicitly warns iOS WebKit pre-26.4 lacks `Symbol.asyncIterator` on `ReadableStream`; irrelevant for native Swift, but the underlying pattern — pull-based reader loop, not iterator — is still the correct structural analog for whatever native QUIC stream API is used). Each incoming stream is drained to EOF, JSON-parsed, and routed to `handleServerPush`.

**Reconnect (`tryWtReconnect`, lines 334-368)** — exact control-flow to replicate:
1. Build fresh transport (5s connect timeout via `Promise.race`).
2. Register (`sendWtMessage` with `type: 'register'`) BEFORE starting the push listener (comment lines 347-348: avoids push-listener-err noise from a short-lived transport).
3. Send `reconnect` request (`sendWtRequest` with `reconnect_token`).
4. Only on `join-ack` response: commit to the shared `transport`/`useWt` state, THEN start the push listener, THEN wire the closed-handler.
5. On any other response or network error: close the temp transport, return the failure envelope (`{ type: 'join-error', payload: { reason: 'wt-net' } }` on network failure).

**Transport-closed handler** (lines 317-328): on WT `.closed` resolving (success or rejection, both treated the same via `.then(onWtClose).catch(onWtClose)`), if not already reconnecting: clear registered/transport state, stop heartbeat, call `attemptReconnect()`.

**URL construction** (line 335, 378) — same host-substitution requirement as WS: `'https://' + location.hostname + ':4433'` → `https://\(scannedHost):4433`.

### `Transport/TransportManager.swift` (controller/state-machine, event-driven + request-response)

**Analog:** `client/src/phone.ts` lines 371-631 (`startPhoneClient` + `attemptReconnect`)

**Initial connect sequence** (`startPhoneClient`, lines 371-486) — exact ordering to replicate:
1. Read token from scan (native: from `QRTokenParser`, not `location.search`).
2. **Attempt 1 — WebTransport**: if WT type/capability exists, try `new WebTransport(wtUrl)` + `.ready` (no explicit timeout on this specific call, unlike reconnect's 5s race); verify stream-reader capability; start push listener BEFORE sending anything (comment line 389: avoids dropped pushes); generate `myId`; register; on any thrown error, null out transport, fall through to WS.
3. **Attempt 2 — WebSocket fallback**, only if `wtOk` is false: connect with a 10s timeout (lines 412-429); on failure, surface a TLS-trust-flavored error message and abort to an error view (ties directly to RESEARCH.md Pitfall 2 — mkcert CA).
4. **Pair**: over WT use `sendWtRequest` with `type: 'pair'`; over WS use a promise/continuation pair resolved by `onPhoneWsMessage`'s `pair-ack`/`pair-error` handling. Either path sends `{ token }` as payload.
5. On `pair-ack`: extract `slot`, `room_code`, `username`, `ice_servers`, `peers`, `reconnect_token` from `payload` (lines 467-473) — defensive type-checks (`typeof payload['slot'] === 'number'`, etc.) should be replicated in Swift's `Codable` decoding (optional/defaulted fields, not force-unwraps).
6. Fan out one `openChannelToPeer` call per entry in `peers` (line 485).

**Reconnect loop** (`attemptReconnect`, lines 507-631) — exact retry shape (this is a hard requirement per D-04/Pattern 3, not simplifiable):
- Guard: no `reconnectToken` → go straight to ended state.
- **3-second initial delay** before the first attempt (comment 513-515: iOS network instability immediately after backgrounding).
- **Up to 13 attempts total**, **10-second wait between attempts**.
- Each iteration: try WT first via `tryWtReconnect()`. If response reason is `slot_not_held`, skip WS entirely for that iteration (both transports would return the same result) and just wait 10s. Otherwise, on any other WT failure, fall through to try WS.
- WS retry sub-path: if the WS socket isn't already closed/closing, close and reopen it (10s timeout), then send a `reconnect` envelope and await `join-ack`/`join-error` via continuation.
- On `join-ack` (from either transport): update `reconnectToken`/`iceServers` from response payload, mark registered, exit reconnecting state, show "reconnected" UI, **reopen any peer connections whose data channel is closed/closing or whose peer connection state is `failed`** (lines 594-604 — collect-then-reopen pattern to avoid mutating the map while iterating), restart heartbeat, return.
- On failure: only `slot_not_held`, `ws-closed`, `wt-net` reasons are retryable; anything else (or exhausting 13 attempts) clears `reconnectToken`, stops heartbeat, and transitions to an "ended" terminal UI state.

### `WebRTC/PeerConnectionManager.swift` (service, event-driven + streaming)

**Analog:** `client/src/phone.ts` lines 634-713 (`openChannelToPeer`)

**Locked data channel config** (line 639 — non-negotiable, D-05/Pattern 2 in RESEARCH.md):
```typescript
const dc = pc.createDataChannel('sensor', { ordered: false, maxRetransmits: 0 });
```
Swift: `RTCDataChannelConfiguration()` with `isOrdered = false; maxRetransmits = 0` — leave `maxPacketLifeTime` untouched (mutually exclusive per spec; RESEARCH.md's explicit anti-pattern warning).

**Event wiring to replicate** (lines 641-712):
- `onconnectionstatechange`: on `'failed'`, if the channel was previously open, decrement the open-channel counter and update UI; if still `registered`, delete the stale peer entry, close the old `pc`, and **re-invoke `openChannelToPeer` for the same peer** (self-healing reconnect-on-failure, not just a passive log).
- `onnegotiationneeded`: `setLocalDescription()` then send an `offer` envelope via `signalSend`. Log the SDP's `a=setup:` value for debugging (relevant to the DTLS-passive-role quirk documented in `room.ts`/RESEARCH.md Pattern 2).
- `onicecandidate`: send `ice-candidate` envelope per non-null candidate; log candidate-gathering-done on null candidate.
- `dc.onopen`: mark channel open, increment open-channel counter, update UI, send `rtc-channel-ready` envelope; if this is a recovery reopen (`isRecovery` flag), also send a `phone-state` envelope with `state: 'channel-recovered'`.
- `dc.onclose`: unless this was an intentional close (`flagClose()` was called — WR-11 pattern, an explicit "don't treat this close as a failure" flag stored per-peer), decrement counter, update UI, send `phone-state` with `state: 'channel-lost'`.
- Peer state stored in a map keyed by peer ID: `{ pc, dc, channelOpen, flagClose }` (line 712) — Swift equivalent: a dictionary of a small struct/class with the same four fields (the `flagClose` closure becomes a stored `Bool` flag set before intentional teardown).

### `Sensor/SensorPacketEncoder.swift` (utility/codec, transform)

**Analog:** `client/src/sensor/encode.ts` (full file — 200 lines, already read in full, no re-read needed)

**Exact D-14 byte layout to replicate bit-for-bit** (`encode.ts` lines 61-79, transcribed verbatim in RESEARCH.md's Code Examples section and already reproduced there as a starting Swift draft):
```
offset  0 : uint8   schema version (= 1)
offset  1 : uint16  seq mod 65536              (little-endian)
offset  3 : uint32  timestamp (ms since session start)
offset  7 : float16 qw
offset  9 : float16 qx
offset 11 : float16 qy
offset 13 : float16 qz
offset 15 : float16 dx   (gesture displacement — 0 in 06.2, D-01/D-09)
offset 17 : float16 dy
offset 19 : float16 dz
offset 21 : float16 px   (dead-reckoning position — 0 in 06.2, D-01/D-09)
offset 23 : float16 py
offset 25 : float16 pz
offset 27 : float32 driftConfidence            (0 in 06.2, D-01/D-09)
offset 31 : uint8   touchActive (1 or 0)
offset 32 : uint16  touchX (round(clamp01(x) * 65535))
offset 34 : uint16  touchY (round(clamp01(y) * 65535))
total  36 bytes, little-endian throughout
```
**Input sanitisation to replicate** (`encode.ts` lines 39-47, `safeFloat`): every float field must be guarded against `NaN`/`±Infinity`/null before writing — fallback to `0`. This guard is applied uniformly to every field in `encodePacket` (lines 94, 97-100, 103-105, 108-110, 113), not just user-facing ones.
**Buffer reuse pattern** (`encode.ts` lines 28-33): a single module-scope buffer is allocated once and reused every tick, to avoid per-tick GC pressure at 60Hz — CONTEXT.md explicitly calls this out (`_packetBuf`-reuse pattern) as something the Swift port should mirror with an equivalent fixed-buffer-reuse approach (e.g. a reused `Data`/`UnsafeMutableRawBufferPointer` rather than allocating a new `Data` per packet).
**Touch clamp pattern** (lines 118-123): clamp to `[0,1]` THEN scale by 65535 THEN round — order matters for edge-case correctness.

Swift starting-point code is already drafted in RESEARCH.md's "Code Examples → Byte-identical Swift packet encoder" section — flagged there as `[ASSUMED]` syntax needing an on-device build + fixture-byte-comparison unit test before trusting it; treat that draft as the implementation seed, not a verified-correct reference.

### `Sensor/CoreMotionSource.swift` (service, event-driven)

**No direct TS file analog** (CoreMotion is a new native capability; the web client's nearest conceptual counterpart is its `DeviceOrientationEvent`/`DeviceMotionEvent` listener setup inside `startSensorPipeline` in `phone.ts`, but that function also does Madgwick/ZUPT/Kalman work this phase explicitly does NOT port — see D-01/D-09).

**What to replicate conceptually:** a single continuous orientation-quaternion stream, sourced from the OS's already-fused output, with **no secondary fusion pass on top** (explicit anti-pattern in RESEARCH.md, mirroring the same anti-pattern already avoided in `phone.ts`'s comments around `DeviceOrientationEvent`). Use `CMMotionManager.startDeviceMotionUpdates(using: .xArbitraryZVertical, to: sensorQueue, withHandler:)` at `1/60` interval; map `CMQuaternion(x,y,z,w)` → packet fields `(qw,qx,qy,qz) = (q.w, q.x, q.y, q.z)`, flagged `checkpoint:human-verify` per RESEARCH.md Pitfall 1 (axis-convention risk, not proven safe without on-device rotation testing). Position/gesture/driftConfidence fields stay hard-zero in this phase (D-09).

### `QRTokenParser.swift` (utility, transform) — extend in place

**Current state** (already read in full, 15 lines):
```swift
enum QRTokenParser {
    static func token(from payload: String) -> String? {
        guard let components = URLComponents(string: payload),
              let token = components.queryItems?.first(where: { $0.name == "token" })?.value,
              !token.isEmpty else {
            return nil
        }
        return token
    }
}
```
**Required extension** (Pitfall 5 in RESEARCH.md): add a sibling accessor (e.g. `static func host(from payload: String) -> String?`) that extracts `URLComponents(string: payload)?.host`, since the native app has no `location.hostname` equivalent and the web client derives every WT/WS URL from the page's own host (`phone.ts` lines 252, 335, 378). Scanned URL shape is `https://{base_url}/phone?token={token}` per the server's `room_registry.rs` pairing-link format (cited in RESEARCH.md Pitfall 5) — `host` extraction is a straightforward `URLComponents.host` read alongside the existing `token` extraction, no new parsing logic needed.

### `ContentView.swift` / new `ActiveSessionView.swift` (component/view, request-response UI state)

**Analog:** `client/src/phone.ts`'s view-state functions (`showView` lines 87-100, `updateConnectingUI` lines 715-728, `showReconnecting`/`showReconnected` lines 493-505)

## Shared Patterns

### Envelope dispatch (`type`/`from`/`to`/`payload` JSON, transport-agnostic)
**Source:** `client/src/phone.ts` `signalSend()` (lines 156-162)
**Apply to:** `TransportManager`, `PeerConnectionManager`, `HeartbeatTimer` — anything that sends a signaling message must go through one dispatcher that picks WT-vs-WS based on current active-transport state, not call transport-specific send functions directly.

### `myId` persistence across reconnects
**Source:** `client/src/phone.ts` line 257 comment: `if (!myId) { myId = crypto.randomUUID(); }  // preserve myId across reconnects`
**Apply to:** `TransportManager` — generate once (UUID), never regenerate on reconnect; both WT and WS registration paths must reuse the same value.

### Retryable-vs-terminal error reason classification
**Source:** `client/src/phone.ts` line 611: `const retryable = errReason === 'slot_not_held' || errReason === 'ws-closed' || errReason === 'wt-net';`
**Apply to:** `TransportManager`'s reconnect loop — only these three reasons continue the retry loop; anything else (e.g. a genuine `pair-error`/`join-error` reason from the server) is terminal and should transition straight to the ended/error state.

### `ordered:false, maxRetransmits:0` data channel config
**Source:** `client/src/phone.ts` line 639
**Apply to:** `PeerConnectionManager` — locked value, must not use `maxPacketLifeTime` as a substitute (see RESEARCH.md Anti-Patterns).

### Host derivation from scanned QR URL (replaces `location.hostname`)
**Source:** `QRTokenParser.swift` (existing) extended per Pitfall 5; consumed by `client/src/phone.ts` lines 252, 335, 378 conceptually (those lines use `location.hostname`, which has no native equivalent — this is a genuine platform gap, not a copy-paste port)
**Apply to:** `WebTransportSignaling`, `WebSocketSignaling`, and any other URL construction — must all derive from the same scanned-host value threaded down from pairing, never hardcoded.

### Safe-float sanitisation before every binary-encoded float write
**Source:** `client/src/sensor/encode.ts` lines 39-47 (`safeFloat`)
**Apply to:** `SensorPacketEncoder` — every float field (quaternion, displacement, position, driftConfidence, touch coords) must be guarded against NaN/Infinity before being written into the packet buffer.
