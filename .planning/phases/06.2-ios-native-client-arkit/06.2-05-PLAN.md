<objective>
Attempt the native WebTransport-over-HTTP/3 client — the D-05 spike — as a `SignalingTransport` conformance built on Apple's `Network.framework` QUIC (`NetworkConnection { QUIC(alpn:["h3"]) }` on iOS 26.5, falling back to `NWProtocolQUIC` if needed). This is genuinely greenfield low-level protocol work; no production-ready native Swift WebTransport client exists (RESEARCH Package Legitimacy Audit, A4).

**Time-box (resolves RESEARCH Open Question 3, D-05):** The spike gets exactly ONE implementation cycle (this plan) plus ONE on-device debug session (Plan 09's WebTransport checkpoint). Success = open a QUIC/h3 connection, complete the extended-CONNECT handshake, open a bidirectional stream, and round-trip (or at minimum successfully send) a `register` JSON envelope against the running dev server. If that is not demonstrated within the on-device session in Plan 09, WebSocket-only (Plan 04) becomes the working path for 06.2 and the spike findings are documented for a future revisit. Per D-04/D-05, WS-only must be the OUTCOME of this documented spike, never a shortcut taken upfront.

Because `TransportManager` (Plan 07) already tries WT first then falls back to WS at runtime, a failed spike requires NO code restructure — the class simply fails to connect and the manager degrades. No ARKit (D-01).

Purpose: Give the phase a genuine shot at true WT/WS parity with the web client (D-04) rather than defaulting to WS.
Output: `Http3Framing.swift` (pure, unit-tested framing helpers), `WebTransportSignaling.swift` (the connection + handshake), and framing tests.
</objective>
