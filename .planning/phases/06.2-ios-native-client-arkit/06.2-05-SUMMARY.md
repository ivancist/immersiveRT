## Accomplishments
- `Http3Framing.swift`: pure, Foundation-only, side-effect-free byte helpers — QUIC varint encode/decode (RFC 9000 §16), a minimal HTTP/3 SETTINGS frame (RFC 9114 §7.2.4), a QPACK static-table-only encoder for the extended-CONNECT pseudo-header set (RFC 9204 §4.5), and the RFC 9220 §4.2 WebTransport stream-type prefix
- 16 XCTest cases green, including RFC 9000 Appendix A.1's canonical varint worked examples and a hand-derived byte-exact QPACK fixture for a full extended-CONNECT header block
- `WebTransportSignaling.swift`: a `SignalingTransport` conformance that opens a QUIC/ALPN-h3 connection, exchanges an HTTP/3 SETTINGS frame on a control stream, performs the extended-CONNECT handshake using Task 1's framing helpers, ports `phone.ts`'s `sendWtRequest`/`sendWtMessage`/`listenForServerPushes`/`processWtPush` stream patterns, and generates+sends the initial `register` envelope after starting the push listener (matching `startPhoneClient()`'s exact ordering)
- Confirmed the real `Network.framework` Swift API surface in this Xcode 26.6 toolchain by reading the actual `.swiftinterface` (not guessing from the WWDC25 sketch RESEARCH.md flagged as `[ASSUMED]`/uncited) — `NetworkConnection<QUIC>.openStream(directionality:)`, `.inboundStreams(handler:)`, `NetworkChannel.send/receive`, `QUIC.Stream.streamID` all confirmed to exist with the signatures this file now uses

## Decisions Made
- Extended-CONNECT header order and static-vs-literal encoding choice per pseudo-header (see `key-decisions` in frontmatter) — followed the plan's explicit instruction (`:method`/`:scheme` indexed, `:path`/`:authority` literal-with-name-reference, `:protocol` literal-without-name-reference since no static entry exists)
- `extendedConnectHeaders()` returns the QPACK field section only, not an HTTP/3 HEADERS-frame-wrapped block — callers compose the frame wrapper themselves from `encodeVarint`, avoiding a redundant dedicated helper
- Polled `NetworkConnection.state` instead of using the `@isolated(any) @Sendable` `onStateUpdate` callback for readiness, to sidestep cross-actor closure-capture complexity under this project's `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` build setting — a deliberate simplicity trade-off appropriate for a time-boxed spike
- `connect()` also generates `myId` and sends `register` (not just the raw QUIC/HTTP3 handshake) — re-reading `client/src/phone.ts`'s `startPhoneClient()` shows WT-ready + myId-generation + push-listener-start + register-send are one contiguous control-flow block for the WT path, and `SignalingTransport.swift`'s own doc comment documents `connect()` as sending the initial register envelope

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed incorrect hand-computed test fixtures for `wtStreamTypePrefix`**
- **Found during:** Task 1 (first real `xcodebuild test` run)
- **Issue:** Initial test fixtures asserted `wtStreamTypePrefix(bidi: false) == [0x54]` and `bidi: true == [0x41]` (single-byte), but 0x54 (84) and 0x41 (65) both exceed the QUIC varint's 6-bit/63-value single-byte boundary (RFC 9000 §16) — the correct minimal-length encoding for both is 2 bytes.
- **Fix:** Corrected the test fixtures to `[0x40, 0x54]` and `[0x40, 0x41]` (the real `encodeVarint` 2-byte encoding). The implementation was correct from the start; only the hand-derived test literals were wrong.
- **Files modified:** mobile/ios-app/immersiveRTTests/Http3FramingTests.swift (pre-commit, part of Task 1's single commit — no separate fix commit needed since caught before the task commit)
- **Verification:** `xcodebuild test -only-testing:immersiveRTTests/Http3FramingTests` — all 16 cases green after the fix
- **Committed in:** 9161a03 (Task 1 commit — fixture correction was made before committing, not a separate follow-up commit)
