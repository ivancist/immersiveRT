<objective>
Port the 36-byte binary sensor packet encoder (schema v1, D-14 layout) to Swift byte-for-byte, and extend the existing `QRTokenParser` to also extract the server host from the scanned pairing URL.

This satisfies PHONE-05 for the native client. Per D-09, orientation (qw/qx/qy/qz) will carry real CoreMotion data (wired in Plan 03/07), while gesture displacement (dx/dy/dz), dead-reckoning position (px/py/pz), and driftConfidence stay hard-zero in 06.2 — deferred to 06.3's ARKit integration (D-01). The encoder is the fixed producer contract for the unchanged Phase 6 desktop decoder, so byte-identity is verified against a fixture captured from the real `encode.ts`, not eyeballed.

Purpose: The desktop `decode.ts` (Phase 6, unchanged) is the fixed consumer — any byte drift silently corrupts every rendered frame.
Output: `SensorPacketEncoder.swift`, a fixture generator + captured fixture, byte-equality tests, and the host-extraction extension to `QRTokenParser`.
</objective>

| CoreMotion sensor → packet encoder | Raw float sensor values (possibly NaN/Infinity on a glitch) enter the binary encoder. |
| Encoder → desktop decoder (P2P data channel) | Encoded bytes cross the network to the unchanged Phase 6 decoder. |

ASVS Level 1; block on high severity.

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06.2-03 | Tampering | SensorPacketEncoder float fields | medium | mitigate | Port `safeFloat` and apply to every float field before writing so a transient CoreMotion NaN/Infinity cannot produce a poison float16 the desktop decoder reads (V5, RESEARCH Security Domain). A fixture test exercises the NaN→0 path. |
</threat_model>
