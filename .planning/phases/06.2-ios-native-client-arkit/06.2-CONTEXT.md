**Gathered:** 2026-07-12
**Status:** Ready for planning

<domain>
## Phase Boundary

**Scope narrowed during discussion — read this before ROADMAP.md.** ROADMAP.md's current 06.2 description bundles transport parity (WebTransport/WS signaling, WebRTC data-channel fan-out, pairing, heartbeat, binary packet encoding) together with ARKit position/orientation tracking. This discussion decided to **split that into two phases**, mirroring how Phase 4 (transport) and Phase 5 (sensor fusion) split the web client:

- **This phase (06.2) is transport parity only:** reimplement the phone client's transport stack natively in Swift — WebTransport-primary/WebSocket-fallback server communication (signaling AND ongoing game-state communication, not just the pair handshake), join-room/pairing flow (building on the existing `QRScannerView.swift`/`QRTokenParser.swift`), WebRTC unreliable data channel fan-out to all desktops in the room, heartbeat, and the 36-byte binary sensor packet schema v1 — with orientation+position values sourced from whatever placeholder/stub is simplest until the tracking phase lands (do not build ARKit integration in this phase).
- **A follow-on phase (not yet in ROADMAP.md, tentatively "06.3: ARKit Position & Orientation Tracking") covers ARKit** — `ARSession` world tracking for both position AND orientation, the mandatory on-device precision verification checkpoint, and recenter/reset UX.

<decisions>
## Implementation Decisions

### Phase split
- **D-01:** Phase 06.2 = transport parity only (WebTransport/WS + WebRTC + pairing + packet encoding). ARKit tracking is deferred to a new follow-on phase, not built here.

### Orientation & position source (applies to the future ARKit phase, captured now so it isn't lost)
- **D-02:** Use ARKit's `ARSession`/camera-transform pose for **both** position and orientation — it provides orientation "for free" alongside position from the same VIO fusion. Do **not** run a separate CoreMotion (`CMDeviceMotion`) orientation pipeline in parallel; this differs from Phase 06.1's Android/WebXR approach, which kept `DeviceOrientationEvent` for orientation and only sourced position from WebXR.
- **D-03 (risk/research flag):** ARKit's camera-transform axis convention may not match `DeviceOrientationEvent`'s convention used by the existing desktop decode/SLERP pipeline (Phase 6). Phase 6 already hit an axis-inversion bug once porting position (`quick/260710-w83`). The future ARKit phase's research step MUST verify the axis mapping against the existing pipeline before assuming compatibility.

### iOS signaling & server-communication transport
- **D-04:** WebTransport is not scoped to signaling alone — it's the general server-communication channel used throughout gameplay (matching the web client's `phone.ts` dual-path design, where WebTransport is used for ongoing communication, not just the initial pair handshake). Mirror the web client's exact dual-path pattern: **WebTransport primary, WebSocket fallback** — do not default to a WebSocket-only shortcut.
- **D-05 (research flag):** No mature first-party native Swift WebTransport/QUIC client is confirmed to exist yet. Research must find and validate the concrete implementation approach (e.g. `Network.framework`'s HTTP/3 support, or a viable third-party library) before planning locks in an architecture. If no viable path is found, the WebSocket fallback (already implemented server-side, per `client/src/phone.ts`'s WT/WS dual path) becomes the working path — but this is a fallback of last resort, not the default choice.

### Test device & distribution
- **D-06:** Development/testing distribution is via **Xcode-tethered debug builds** (free Apple ID; note the standard 7-day app-expiry unless periodically re-signed/reinstalled from Xcode). No TestFlight/paid Developer Program needed for now.
- **D-07:** User confirmed an iPhone is available for on-device verification — the go/no-go ARKit checkpoint should NOT get blocked on missing hardware the way Phase 06.1's Android checkpoint (06.1-04) did.

### ARKit recenter/reset UX (forward note for the future ARKit-tracking phase — NOT built in 06.2)
- **D-08:** Both an automatic origin reset AND a manual "recenter" control should exist. Auto-reset timing (raw pair-time vs. after a hold-still calibration step, mirroring the web client's `runCalibration`-before-sensor-pipeline convention in `client/src/sensor/encode.ts`) is left for that phase's research/planning to evaluate — not decided here.

### Claude's Discretion
- Exact library/dependency-manager choice for WebRTC on iOS (e.g. SPM vs CocoaPods, which prebuilt WebRTC.xcframework) — no user preference stated; research/planner decides.
- UI treatment for the transport-parity phase beyond what's needed to prove the connection works (existing `ContentView.swift`/`TokenDetailsView.swift` are placeholders) — no explicit request for a polished UI in this phase.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `QRScannerView.swift` + `QRTokenParser.swift`: pairing-token capture already works end-to-end; the transport-parity phase starts from a scanned token, not from scratch on QR handling.
- `client/src/sensor/encode.ts`'s `_packetBuf`-reuse pattern (single allocated buffer, no per-tick GC) — Swift port should use an equivalent fixed-buffer reuse approach for the 60Hz packet encode path.

### Established Patterns
- Dual-path transport with WebTransport preferred, WebSocket fallback, decided per-connection-attempt (see `phone.ts` `startPhoneClient()` / `tryWtReconnect()`) — the native client must replicate this decision structure, not just the WebSocket path.
- `ordered:false, maxRetransmits:0` RTCDataChannel config is enforced from first use across the whole project — native WebRTC data channel setup must match exactly.

### Integration Points
- Server-side signaling/pairing protocol (Rust `wt_server`/`ws_server`, room registry) is already built (Phases 1-3) and is transport-agnostic between web and native clients — no server changes anticipated for this phase.
- Desktop decode/SLERP pipeline (Phase 6, `client/src/sensor/decode.ts` + `playerStore.ts`) is the fixed consumer contract: whatever the native client sends over the wire must byte-match schema v1 regardless of which platform produced it.

</code_context>

<specifics>
## Specific Ideas

- "ARKit gives orientation for free" — explicit user rationale for D-02: don't duplicate fusion work CoreMotion would otherwise do, since ARKit's `ARFrame.camera.transform` already yields a full pose (rotation + translation).
- "WebTransport is not only for signaling. It will be used for server communications in games." — explicit user correction that reframed D-04; WebTransport's role in this phase is broader than the initial pairing handshake.
- "You'll find the way to use it" — user's direction that research, not the discussion, resolves whether/how native WebTransport is implemented (D-05); a WS-only shortcut should not be assumed as the primary plan.

</specifics>
