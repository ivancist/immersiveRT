<objective>
Build the `TransportManager` state machine — the orchestrator that ties transports, pairing, WebRTC fan-out, heartbeat, and the sensor send loop together — as a direct port of `phone.ts`'s `startPhoneClient()` + `attemptReconnect()`.

This is where D-04's dual-path requirement is enforced: WebTransport is attempted first every time, WebSocket is the fallback — never a WS-only happy path. Because the manager is transport-agnostic (it drives whichever `SignalingTransport` connected), a failed WebTransport spike (Plan 05) degrades here purely at runtime with zero code change. The manager also starts the CoreMotion-driven encode/send loop where orientation is real and position/gesture/drift are zero-stubbed (D-09). No ARKit (D-01).

Purpose: The single source of truth for connect → pair → fan-out → stream → reconnect, matching the web client's proven control flow exactly.
Output: `TransportManager.swift` and its state-machine tests using fake transports.
</objective>
