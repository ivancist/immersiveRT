## Summary

The native Swift phone client's transport layer breaks into three independent problems, each with a different confidence level. **WebRTC** is solved: `stasel/WebRTC` is the community-standard SPM-installable xcframework (Google deprecated its own CocoaPods binary distribution at M80), and its `RTCDataChannelConfiguration` API maps 1:1 onto the web client's `{ordered: false, maxRetransmits: 0}` contract. **WebSocket** is solved even more simply: `URLSessionWebSocketTask` is a first-party Foundation API since iOS 13 — zero dependency risk, and it is the literal fallback-of-last-resort path D-05 anticipates. **WebTransport is the hard problem.** As of this research (mid-2026), there is no production-ready, first-party, or mature third-party native Swift WebTransport client. `URLSession`'s built-in HTTP/3 support only accelerates ordinary HTTP requests — it does not expose the WebTransport session/stream API (extended CONNECT, bidirectional/unidirectional streams, datagrams). The two Swift QUIC efforts found (`hironichu/Quiver`, `swift-quic` org) are both early-stage: Quiver is explicitly described by its own author (Feb 2026 Swift Forums post) as AI-assisted, Linux-tested-only, "cannot say anything about the reliability" beyond that; `swift-quic`'s `cloudflare-quiche` bindings show low commit/release activity with the last issue closed roughly two years ago. Neither is a defensible production dependency for a phone-motion-controller platform whose core value proposition is sub-20ms reliability.

What DOES exist, and what this research recommends as the primary path, is Apple's own low-level QUIC transport: `Network.framework`'s `NWProtocolQUIC` (iOS 15+) can open a raw QUIC connection with custom ALPN negotiation, and iOS 26's new declarative `NetworkConnection` API (matching this project's 26.5 deployment target) composes QUIC as a swappable protocol-stack component (`NetworkConnection(to:) { QUIC() }`) with async/await send/receive built in. Because the existing Rust server uses the real `wtransport` crate (RFC 9220-compliant WebTransport-over-HTTP/3, not a bespoke lightweight protocol), a native client — whether hand-rolled on `Network.framework` or using a third-party library — MUST speak the full WebTransport-over-HTTP/3 handshake to interoperate: ALPN `h3`, an HTTP/3 SETTINGS frame exchange, an extended CONNECT request (`:method: CONNECT`, `:protocol: webtransport`), and then WT-stream-type-prefixed QUIC streams. This is a real, scoped, but nontrivial protocol implementation — not a simple wrapper. Given that, the honest recommendation is: **spike it first, time-boxed, before committing the phase's architecture to it.** If the spike succeeds, the phase gets true WT/WS parity with the web client. If it doesn't land within budget, WebSocket-only (a robust, zero-dependency, already-proven-server-side path) is the documented fallback, matching D-05's explicit framing exactly.

The CoreMotion axis question resolves more cleanly than expected but is not risk-free. `CMAttitude`'s device-local axis convention (X = right, Y = up/top, Z = out of the screen toward the viewer, when the device is held in portrait) matches the W3C `DeviceOrientationEvent` device-frame convention used by `eulerToQuat()` in the web client — this is corroborated by the W3C spec itself, which explicitly states its absolute-orientation quaternion convention is "similar to... the `xMagneticNorthZVertical` option for Core Motion." However, the web client's iOS branch does NOT use absolute/compass-referenced orientation (`deviceorientationabsolute` never fires on iOS Safari; iOS falls into the generic, device-relative `deviceorientation` fallback) — so the closest CoreMotion reference-frame analog for parity is `CMAttitudeReferenceFrame.xArbitraryZVertical`, not `.xMagneticNorthZVertical`. This mapping is reasoned from documentation, not verified on-device, and is flagged as the phase's primary open risk (Pitfall 1) — exactly the kind of axis-convention bug this project has already hit once (see `.planning/debug/huge-position-drift-makes-pho.md`).

**Primary recommendation:** Build the Swift transport stack as: (1) a `SignalingTransport` protocol with two conformances — `WebTransportSignalingTransport` (custom `Network.framework`-based RFC 9220 client, built behind a time-boxed spike) and `WebSocketSignalingTransport` (`URLSessionWebSocketTask`, zero-dependency, always-available fallback); (2) `stasel/WebRTC` via SPM for the data-channel fan-out, replicating `{isOrdered: false, maxRetransmits: 0}` and the iOS-Safari-DTLS-passive-role SDP patch already proven necessary server-side in `room.ts`; (3) `CMMotionManager.startDeviceMotionUpdates(using: .xArbitraryZVertical)` at 60Hz for orientation, remapped from `CMQuaternion(x,y,z,w)` into the packet's `(qw,qx,qy,qz)` fields, position/gesture/driftConfidence stubbed to zero pending 06.3's ARKit integration; (4) a hand-written, dependency-free 36-byte packet encoder using Swift's native `Float16` type, byte-identical to `encode.ts`'s D-14 layout.

### Pattern 1: Minimal WebTransport-over-HTTP/3 client on `Network.framework` (the D-05 spike)

**What:** Interoperating with the existing Rust `wtransport` 0.7.1 server requires the FULL WebTransport-over-HTTP/3 handshake — this is not a lightweight custom protocol shortcut, because `wtransport::Endpoint::server(...)` implements the real spec (`IncomingSession` → `request.accept()` → `conn.accept_bi()/open_bi()` per `server/src/wt_server.rs`). The handshake sequence a native client must implement:
1. Open a QUIC connection to `host:4433` with ALPN `h3` (`NWProtocolQUIC` options, or iOS 26's `NetworkConnection { QUIC() }`).
2. Exchange minimal HTTP/3 control-stream SETTINGS frames (client and server each open a unidirectional control stream and send a SETTINGS frame — for a minimal client, an empty/default SETTINGS frame is sufficient; `wtransport`/`h3` implementations require this before accepting requests).
3. Open a bidirectional stream and send an HTTP/3 request with `:method: CONNECT`, `:protocol: webtransport`, `:scheme: https`, `:path: /` (or whatever path the server's accept logic uses — `wt_server.rs`'s `IncomingSession` logs `request.path()` but does not appear to gate on it, so `/` should suffice), encoded via QPACK (static-table-only encoding is sufficient for this fixed, small header set — no dynamic table needed for a client that never varies its headers).
4. On receiving a `200` response on that stream, the WebTransport session is established; that stream becomes the "CONNECT stream" for the session's lifetime.
5. Subsequent QUIC streams (bidi and uni) are prefixed with a WT stream-type varint per RFC 9220 to associate them with the session — this is exactly what `phone.ts`'s `transport.createBidirectionalStream()` does under the hood in the browser.

**When to use:** As the primary attempt for the WebTransport transport, time-boxed. If a working proof-of-concept (open a connection, send the `register` envelope, receive nothing back but the connection stays open — matching `wt_server.rs`'s no-response-on-register behavior) cannot be demonstrated within the spike's time budget, fall back to WebSocket-only for this phase and record the spike's findings for a future revisit.

**Example (iOS 26+, matches project's 26.5 deployment target):**

### Anti-Patterns to Avoid

- **Defaulting to WebSocket-only without attempting WebTransport first:** D-04/D-05 explicitly reject this as the default; it must be the outcome of a failed, documented spike, not a shortcut taken to save implementation time.
- **Skipping the QPACK/HTTP-3-framing layer and trying to open raw QUIC streams directly against `wtransport`:** the server implements the real WebTransport-over-HTTP/3 spec (confirmed by reading `wt_server.rs` — it uses `wtransport::Endpoint::server` and `IncomingSession`/`request.accept()`, not a raw QUIC echo protocol). A client that skips the HTTP/3 CONNECT handshake will never get past `IncomingSession`.
- **Using `maxPacketLifeTime` instead of `maxRetransmits`:** WebRTC's data channel spec treats these as mutually exclusive: the web client's locked config is `maxRetransmits: 0` specifically (not a time-based equivalent) — the Swift `RTCDataChannelConfiguration` must set `maxRetransmits = 0` and leave `maxPacketLifeTime` untouched/default.
- **Running a secondary orientation-fusion pass on top of CoreMotion's `attitude`:** exactly the same anti-pattern the web client's comments warn against for `DeviceOrientationEvent` — CoreMotion has already fused gyro+accelerometer(+magnetometer if using a magnetic reference frame); do not re-run Madgwick/Mahony on top of it in this phase.

## Common Pitfalls

### Pitfall 1: CoreMotion attitude quaternion axis/reference-frame mismatch vs. the desktop's expected convention

**What goes wrong:** The desktop pipeline (`decode.ts` → `THREE.Quaternion.set(qx, qy, qz, qw)`) expects a quaternion produced the same way the web client's `eulerToQuat(alpha, beta, gamma)` produces one — using the exact W3C Z-X-Y Euler formula, device-relative (non-compass) on iOS specifically, because `deviceorientationabsolute` never fires in iOS Safari. If the Swift client instead reads `CMAttitude.quaternion` directly (Apple's own internal sensor-fusion output) and maps its `(x,y,z,w)` fields naively into the packet's `(qw,qx,qy,qz)` slots, there is a real risk the rotation "feels" wrong on the rendered cube — either because (a) the wrong `CMAttitudeReferenceFrame` was chosen (compass-referenced `.xMagneticNorthZVertical` vs. device-relative `.xArbitraryZVertical` — these are NOT interchangeable; picking the wrong one changes yaw behavior entirely), or (b) CoreMotion's internal fusion algorithm produces a numerically different (though physically equivalent) quaternion representation than the W3C Euler-angle-to-quaternion formula, even when device axes nominally match. This project has already been burned once by a perceived axis-inversion bug (`.planning/debug/huge-position-drift-makes-pho.md`) that turned out to be a ZUPT re-anchoring artifact, not a literal sign bug — but that investigation shows how easy it is to misdiagnose orientation/position bugs without careful on-device isolation testing.

**Why it happens:** CoreMotion and the W3C DeviceOrientationEvent spec are independently-implemented sensor-fusion stacks that happen to share a device-local axis convention (per NSHipster + W3C spec cross-reference — MEDIUM confidence) but were never designed to be bit-compatible with each other, and the two platforms' "no compass" behaviors (`.xArbitraryZVertical` vs. iOS Safari's non-absolute `deviceorientation`) are similar in spirit but not guaranteed identical in implementation.

**How to avoid:** (1) Use `CMAttitudeReferenceFrame.xArbitraryZVertical` as the starting point — it is the closest match to iOS Safari's actual current behavior (device-relative, no compass). (2) Add an on-device verification task early in the plan: rotate the phone through known reference rotations (90° yaw, 90° pitch, 90° roll) and confirm the rendered desktop cube rotates in the matching direction/axis, comparing directly against the existing web client's behavior on the SAME physical device if possible. (3) Do not assume `(qw,qx,qy,qz) = (attitude.quaternion.w, .x, .y, .z)` is correct without this verification — flag it as a `checkpoint:human-verify` task, not an assumption baked silently into the encoder.

**Warning signs:** Cube rotates around the wrong axis, rotates in the opposite direction from the phone's motion, or yaw drifts/snaps unexpectedly (would indicate a reference-frame mismatch, e.g. compass corrections firing when not expected).

### Pitfall 2: TLS trust — mkcert's local CA is not trusted on the iPhone by default

**What goes wrong:** Every `NWConnection`/`NetworkConnection`/`URLSessionWebSocketTask` connection to the dev server over TLS will fail with a trust-validation error, because the server uses an mkcert-issued certificate (per CLAUDE.md's TLS-for-local-development stack) whose root CA is installed in the desktop browser's/OS's trust store, NOT the iPhone's. Unlike a desktop browser hitting the same server, there is no "click through the warning" UX path for `Network.framework`/`URLSession` TLS validation failures — the connection simply fails.

**Why it happens:** mkcert's local CA must be explicitly installed as a trusted root certificate profile on each device that needs to trust it; this is a one-time manual step per test device that is easy to forget when moving from "worked on desktop Chrome" to "testing on the iPhone."

**How to avoid:** Document (and the plan should include a task for) exporting mkcert's root CA (`mkcert -CAROOT` on the dev machine), transferring it to the iPhone (AirDrop or a hosted URL), installing it as a configuration profile (Settings → General → VPN & Device Management), AND enabling full trust for it (Settings → General → About → Certificate Trust Settings) — both steps are required; installing the profile alone is not sufficient for TLS chain validation.

**Warning signs:** Every connection attempt fails immediately with a TLS/trust error, even though the same server works fine from a desktop browser or the web phone client.

### Pitfall 3: CoreMotion delivery queue and thread-safety with SwiftUI/`@MainActor`

**What goes wrong:** `CMMotionManager.startDeviceMotionUpdates(to: someQueue, withHandler:)` delivers callbacks on whatever `OperationQueue` is passed — if that queue is the main queue at 60Hz while also encoding+sending over WebRTC synchronously in the handler, UI responsiveness can degrade; if it's a background queue, touching `@Published`/SwiftUI state from within the handler without hopping back to `@MainActor` will trigger runtime warnings/crashes in strict-concurrency builds (the project's `SWIFT_APPROACHABLE_CONCURRENCY = YES` / `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` build settings, confirmed in `project.pbxproj`, make this a live concern).

**Why it happens:** The web client's `devicemotion` event handler runs on the browser's normal event loop with no explicit threading model to reason about; the Swift port introduces a real choice (which queue) with real concurrency-safety consequences that has no web-client analog to copy.

**How to avoid:** Run `CMMotionManager` updates on a dedicated background `OperationQueue`, do the packet encode + `RTCDataChannel.sendData()` call entirely off the main actor (both are non-UI operations), and only hop to `@MainActor` for UI-visible state (connection status, channel count) at a throttled rate — not per-packet.

**Warning signs:** Dropped frames / UI stutter during active sensor streaming, or Swift concurrency runtime warnings/crashes about actor isolation in the Xcode console.

### Pitfall 4: `isIdleTimerDisabled` must be reset on backgrounding / app termination, unlike the web Wake Lock API's auto-release semantics

**What goes wrong:** The web client's Wake Lock API auto-releases when the tab is backgrounded and the code explicitly re-requests it on `visibilitychange` → visible (see `phone.ts`'s `visibilitychange` listener). `UIApplication.shared.isIdleTimerDisabled = true` has different lifecycle semantics — it persists until explicitly set back to `false`, and forgetting to reset it can cause user-facing battery/UX complaints unrelated to this specific session if the app doesn't clean up on backgrounding.

**Why it happens:** No 1:1 API lifecycle parity between the browser Wake Lock API and iOS's idle-timer API — this is a genuine platform-semantic difference, not a straightforward port.

**How to avoid:** Set `isIdleTimerDisabled = true` when entering the active sensor-streaming state (mirroring `requestWakeLock()`'s call sites) and explicitly set it back to `false` in `sceneDidEnterBackground`/`applicationWillResignActive` or equivalent SwiftUI `scenePhase` handling, re-enabling on return to foreground — mirroring the intent of `phone.ts`'s `visibilitychange` handler even though the underlying API shape differs.

**Warning signs:** Screen never locks even after leaving the app, draining battery.

### Pitfall 5: `QRTokenParser` currently discards the host — the native app has no `location.hostname` equivalent

**What goes wrong:** The web client derives all of its server URLs (`https://{location.hostname}:4433`, `wss://{location.hostname}:9090`) from the page's own URL, because it IS served from that host. The native app is not served from anywhere — it's a standalone binary — so it has no equivalent of `location.hostname`. The existing `QRTokenParser.token(from:)` only extracts the `token` query parameter and silently discards the rest of the scanned URL (including the host), per the code read this session (`QRTokenParser.swift`, `URLComponents(string: payload)?.queryItems?.first(where: {$0.name == "token"})`).

**Why it happens:** The scaffold was built to prove QR scanning + token extraction work, before the transport layer (and its need for a server host) existed.

**How to avoid:** Extend `QRTokenParser` (or add a sibling function) to also extract `URLComponents(string: payload)?.host` from the scanned pairing URL (`https://{base_url}/phone?token={token}` per `room_registry.rs`'s `format!("{}/phone?token={}", base_url, pairing_token)`), and thread that host string through to the transport layer's WT/WS URL construction, replacing every `location.hostname` reference in the ported `phone.ts` logic.

**Warning signs:** Hardcoding a dev-machine IP/hostname into the Swift app instead of deriving it from the scan (works on one dev machine, breaks the moment the server's IP changes — trivial but easy to miss if not flagged upfront).

### Byte-identical Swift packet encoder (D-14 schema v1, 36 bytes)

```swift
// Source: direct structural port of client/src/sensor/encode.ts —
// [VERIFIED: client/src/sensor/encode.ts, read this session; byte offsets/types below are an
// exact transcription of that file's documented D-14 layout, not independently re-derived]
//
// D-14 byte layout (little-endian throughout):
//   offset  0 : uint8   schema version (= 1)
//   offset  1 : uint16  seq mod 65536
//   offset  3 : uint32  timestamp (ms since session start)
//   offset  7 : float16 qw
//   offset  9 : float16 qx
//   offset 11 : float16 qy
//   offset 13 : float16 qz
//   offset 15 : float16 dx (gesture displacement x — stubbed 0 in 06.2)
//   offset 17 : float16 dy
//   offset 19 : float16 dz
//   offset 21 : float16 px (dead-reckoning position x — stubbed 0 in 06.2)
//   offset 23 : float16 py
//   offset 25 : float16 pz
//   offset 27 : float32 driftConfidence (stubbed 0 in 06.2)
//   offset 31 : uint8   touchActive (1 or 0)
//   offset 32 : uint16  touchX (round(clamp01(x) * 65535))
//   offset 34 : uint16  touchY (round(clamp01(y) * 65535))
//   total  36 bytes
struct SensorPacket {
    var seq: UInt16
    var timestampMs: UInt32
    var qw, qx, qy, qz: Float
    var dx, dy, dz: Float
    var px, py, pz: Float
    var driftConfidence: Float
    var touchActive: Bool
    var touchX, touchY: Float // [0,1]
}

func encodePacket(_ pkt: SensorPacket, into buf: inout Data) {
    buf.removeAll(keepingCapacity: true)
    buf.append(1) // schema version
    withUnsafeBytes(of: pkt.seq.littleEndian) { buf.append(contentsOf: $0) }
    withUnsafeBytes(of: pkt.timestampMs.littleEndian) { buf.append(contentsOf: $0) }
    for v in [pkt.qw, pkt.qx, pkt.qy, pkt.qz, pkt.dx, pkt.dy, pkt.dz, pkt.px, pkt.py, pkt.pz] {
        withUnsafeBytes(of: Float16(v).bitPattern.littleEndian) { buf.append(contentsOf: $0) }
    }
    withUnsafeBytes(of: pkt.driftConfidence.bitPattern.littleEndian) { buf.append(contentsOf: $0) }
    buf.append(pkt.touchActive ? 1 : 0)
    let tx = UInt16((min(1, max(0, pkt.touchX)) * 65535).rounded())
    let ty = UInt16((min(1, max(0, pkt.touchY)) * 65535).rounded())
    withUnsafeBytes(of: tx.littleEndian) { buf.append(contentsOf: $0) }
    withUnsafeBytes(of: ty.littleEndian) { buf.append(contentsOf: $0) }
}
```

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | No user-account authentication in this platform (session/pairing-token based, not credential-based) |
| V3 Session Management | yes | Pairing/reconnect tokens — the native client must treat `reconnect_token` with the same care the web client does: `localStorage`/`sessionStorage` equivalents on iOS are `UserDefaults`/Keychain; given STATE.md's existing precedent (T-03-09: "never log the reconnect token value"), the Swift port MUST NOT log the token either, and should prefer Keychain over `UserDefaults` for at-rest storage (UserDefaults is not encrypted at rest by default; Keychain is) — this is a STRICTER control than the web client's `localStorage` use, worth flagging as an improvement rather than a strict port. |
| V4 Access Control | yes | Server-side `envelope.from != my_id` spoofing check (confirmed in `wt_server.rs`) already defends against a malicious client claiming another client's identity — no client-side control needed beyond correctly using the server-issued `myId`/UUID, matching `phone.ts`'s `crypto.randomUUID()` pattern (Swift equivalent: `UUID().uuidString`). |
| V5 Input Validation | yes | `decodePacket`-equivalent validation is desktop-side (Phase 6, unchanged) and out of scope here; the Swift ENCODER should still run all float fields through a `safeFloat`-equivalent NaN/Infinity guard before writing (mirroring `encode.ts`'s `safeFloat()`) so a transient CoreMotion glitch can't produce a poisoned packet — recommend porting this guard function directly. |
| V6 Cryptography | yes | TLS only (no custom crypto) — QUIC mandates TLS 1.3 per spec; `Network.framework`/`URLSessionWebSocketTask` handle the TLS implementation, never hand-roll. mkcert-issued dev certs are self-signed-by-a-local-CA, not production certs — Pitfall 2 covers the trust-chain setup this requires on the test device. |

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Reconnect-token leakage via logs/crash reports | Information Disclosure | Never log the token value (matches existing `T-03-09` project convention); prefer Keychain storage over `UserDefaults` on iOS. |
| Malicious/spoofed `from` field in signaling envelopes | Spoofing | Already mitigated server-side (`wt_server.rs`'s `envelope.from != my_id` check) — Swift client just needs to consistently use its server-issued ID, no client-side additional control needed. |
| Untrusted/malformed WebRTC data channel payload from a peer | Tampering | Out of scope for the PHONE client (it only SENDS packets, doesn't decode peer-sent ones in this architecture — no phone-to-phone data channel exists); desktop-side validation (Phase 6, unchanged) already handles this for packets IT receives. |
| TLS downgrade / cert-pinning bypass on a hostile network | Tampering/Spoofing | Standard system TLS trust store validation via `Network.framework`/`URLSession` is sufficient for this phase's dev/LAN-testing threat model; production hardening (cert pinning) is out of scope for a phase whose distribution model is Xcode-tethered debug builds (D-06), not public release. |
