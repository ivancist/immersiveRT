# Phase 06.2 Plan 03: CoreMotion Orientation Source + Heartbeat Timer Summary

**CoreMotionSource streams OS-fused attitude quaternions at 60Hz off the main actor (axis mapping flagged for Plan 09 verification); HeartbeatTimer fires idempotent 5s heartbeats via an injectable scheduler, fully unit-tested with zero real-time waits.**

## Accomplishments
- `CoreMotionSource` wraps `CMMotionManager`, streaming `attitude.quaternion` at 1/60s via `.xArbitraryZVertical` on a dedicated serial background `OperationQueue`, with no secondary orientation-fusion pass on top (D-09, anti-pattern avoided per RESEARCH.md)
- `HeartbeatTimer` ports `phone.ts`'s `startHeartbeat()` exactly: idempotent clear-before-restart, fixed 5000ms interval, dispatches through an injected `(SignalingEnvelope) -> Void` sender rather than a hardcoded transport (D-04)
- `HeartbeatScheduler`/`HeartbeatCancelable` protocol pair decouples the timer from `Foundation.Timer`, enabling `HeartbeatTimerTests` to simulate N elapsed intervals deterministically (no real 5s sleeps)
- Both files verified against the real Xcode toolchain: `xcodebuild build` succeeds with `CoreMotionSource` included, and `xcodebuild test -only-testing:immersiveRTTests/HeartbeatTimerTests` passes all 4 tests

## Decisions Made
- `.xArbitraryZVertical` chosen as the CoreMotion reference frame (RESEARCH.md's reasoned starting point for parity with iOS Safari's non-absolute `deviceorientation` behavior) — explicitly commented as `⚠️ UNVERIFIED (Pitfall 1)` pending Plan 09's on-device axis-verification checkpoint
- `HeartbeatScheduler`/`HeartbeatCancelable` protocols introduced (not in the plan's literal text, but implied by "injectable interval/clock") to make the 5s cadence testable without real waits or flaky timing assertions
- Sensor callback runs entirely off the main actor on a dedicated serial `OperationQueue` (`maxConcurrentOperationCount = 1`) — satisfies Pitfall 3's concurrency-isolation guidance under the project's `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` build setting
