## Decisions Made
- Pinned `stasel/WebRTC` to `150.0.0` (the real current release tag Xcode resolved during Task 0's human verification), not the unverified `149.0.0` figure from RESEARCH.md
- `applyRemoteAnswer`/`addRemoteCandidate` accept the raw `SignalingEnvelope` rather than a pre-parsed `RTCSessionDescription`/`RTCIceCandidate`, keeping envelope parsing (payload key names `type`/`sdp` for answers, `candidate`/`sdpMid`/`sdpMLineIndex` for ICE â€” confirmed against `room.ts`'s `RTCSessionDescriptionInit`/`RTCIceCandidateInit` usage) inside this type so a future `TransportManager` can route server pushes straight through
- `peerConnectionShouldNegotiate` creates the offer explicitly via `offer(for:completionHandler:)` then `setLocalDescription(_:completionHandler:)`, since the native binding exposes no single-call equivalent to `phone.ts`'s zero-arg `pc.setLocalDescription()` (which infers offer-vs-answer from signaling state)
- `maxPacketLifeTime` must never be assigned or read anywhere this project touches an `RTCDataChannelConfiguration`/`RTCDataChannel` that uses `maxRetransmits` â€” verified as a source-contract test (grep for the assignment) rather than a runtime property read, since the read itself crashes

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Removed a crashing runtime assertion on `maxPacketLifeTime`, replaced with a source-contract test**
- **Found during:** Task 3 (`PeerConnectionManagerTests`)
- **Issue:** The plan's acceptance criteria calls for "a test asserts `maxPacketLifeTime` is the default/unset (not substituted)." A direct `XCTAssertEqual(config.maxPacketLifeTime, -1)` crashed the test process with `SIGABRT` â€” confirmed via exported crash log that the abort originates inside `WebRTC.framework` itself (M150 binary), not Swift-side code. The property's public header documents `-1` as the "unset" default, but the getter enforces `maxPacketLifeTime`/`maxRetransmits` mutual exclusivity by aborting on read of the unset field, not just on simultaneous write.
- **Fix:** Replaced the runtime property read with a source-contract test (`test_makeDataChannelConfig_neverAssignsMaxPacketLifeTime_sourceContract`) that greps `PeerConnectionManager.swift` for `maxPacketLifeTime =` and asserts it's absent. Documented the hazard prominently in `makeDataChannelConfig()`'s doc comment so no future code accidentally reads the property (e.g. for debug logging), which would crash in production too.
- **Files modified:** `mobile/ios-app/immersiveRT/WebRTC/PeerConnectionManager.swift`, `mobile/ios-app/immersiveRTTests/PeerConnectionManagerTests.swift`
- **Verification:** Full test suite green (`immersiveRTTests`: 27 tests, 0 failures) after the fix; confirmed the original crash via an exported `.ips` crash log showing the abort frame inside `WebRTC.framework`.
- **Committed in:** `3ad794f` (Task 3 GREEN commit)

**2. [Rule 3 - Blocking] Wired `stasel/WebRTC` into the test target, not just the app target**
