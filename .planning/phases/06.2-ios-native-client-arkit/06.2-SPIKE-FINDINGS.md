# 06.2 WebTransport Spike Findings (D-05)

## Outcome: WT works

The hand-rolled RFC 9220 WebTransport-over-HTTP/3 client (`WebTransportSignaling.swift`,
Plan 05) completes the full handshake and carries live traffic on real hardware: connect →
extended-CONNECT → register → pair → WebRTC offer/answer/ICE relay → heartbeat, all over
native QUIC/h3, for multiple independent sessions, with no WebSocket fallback needed. This
exceeds D-05's minimum bar (WS-only fallback is an acceptable outcome) — WT is the primary
transport in practice on this server/client pairing.

`TransportManager` still tries WT first and falls back to WS on any failure (D-04), so this
remains a safe degrade path if WT regresses in a future server/OS version, not a hard
dependency.

## Time-box spent

One implementation cycle (Plan 05) + one on-device debug session (Plan 09) that ran well
beyond the nominal single-session time-box due to the number of real bugs surfaced — see
below. Justified by the outcome: a fully working native WT client, not just a documented
failure.

## Bugs found and fixed during the on-device session

1. **Control stream not retained (`ClosedCriticalStreamError`)** — `openControlStream(_:)`
   held the opened HTTP/3 control stream in a local variable only; ARC released it (and the
   underlying `Network.framework` stream) the instant the function returned. RFC 9114
   §6.2.1 requires the control stream to stay open for the connection's entire lifetime.
   100% reproducible server-side. Fixed by retaining on `self`.

2. **Extended-CONNECT success check too narrow** — the original heuristic looked for a
   single byte (`0xD9`, the fully-static-indexed QPACK encoding of `:status: 200`). The
   real server's QPACK encoder instead uses a Literal Field Line With Name Reference +
   Huffman-compressed literal value — a different, equally spec-valid encoding of the same
   header. The client treated every successful handshake as a failure and tore its own
   connection down, which the server observed as "connection closed by peer" while waiting
   to accept the register stream. Fixed with a minimal QPACK field-line walker
   (`responseIndicatesStatus200(_:)`) recognizing both encodings, plus a narrow Huffman
   decoder scoped to 3-digit ASCII status codes. Regression-tested against the exact bytes
   captured from the real server.

3. **Ack/error envelopes missing `from`/`to` fields** — a server-side (not this phase's)
   pre-existing asymmetry: `pair-ack`/`pair-error`/`join-ack`/`join-error`/`leave-ack` are
   built via `serde_json::json!()` in `room_registry.rs`/`ws_server.rs`, never through the
   typed Rust `SignalingEnvelope` struct, so they only ever send `{"type","payload"}`. The
   existing JS client tolerates this (untyped); Swift's default `Codable` synthesis required
   both fields, so every ack silently failed to decode and was dropped — leaving the pairing
   request hanging forever even after the server paired successfully. Affected both WT and
   WS paths identically. Fixed with a custom `init(from:)` defaulting missing `from`/`to`/
   `payload`.

4. **No timeout on initial WT connect** — observed a 107-second hang before falling back to
   WS on one early test (later determined to likely have been conflated with an unrelated
   desktop-tab WT session — see correction below). Added an 8s bound regardless, since a
   browser's native `WebTransport.ready` self-times-out and this hand-rolled client did not.

5. **Dead connection undetected after backgrounding** — after the app backgrounds long
   enough for the QUIC connection to time out, nothing detected it: `startInboundStreamListener`'s
   reliance on `conn.inboundStreams` throwing on failure doesn't reliably fire. First fix
   attempt (a separate polling `Task` reading `conn.state`) caused a worse regression —
   concurrent access to `conn` while the main flow was actively using it broke even
   fresh first-time connects, not just reconnects. Reverted; replaced with a synchronous
   state check inside `sendAndDrain` (the one function every send/request/heartbeat funnels
   through) — no concurrent access, same call path already touching the connection.

6. **Heartbeat not sent immediately on resume** — `TransportManager.resumeFromBackground()`'s
   own doc comment already documented the intent ("immediate heartbeat send," mirroring
   `phone.ts`), but only called `heartbeatTimer.start()`, which schedules the *next*
   heartbeat 5s out rather than sending one immediately. Combined with finding 5, this meant
   dead-connection detection (and reconnect) could lag up to 5s after returning to the
   foreground. Added `HeartbeatTimer.fireNow()`, called on resume.

## Not this phase's bug (flagged, not fixed)

The desktop web client throws `Uncaught TypeError: Cannot read properties of undefined
(reading 'add')` in its packet decode handler on the first packet from each new phone
connection (`room-*.js`, minified). Packets still arrive and the cube still renders/rotates
correctly despite the error, so it did not block verification, but it's a real bug in
non-iOS code — out of scope for this iOS-only phase (per project conventions). Worth a
follow-up ticket.

## Correction on an earlier over-claim

An early diagnosis in this session attributed a ~107-second WT hang directly to the phone's
own connect attempt, based on server logs showing "WT session accepted" then a long gap
before failure. The user correctly pointed out the desktop browser tab also opens its own
WT signaling connection to the same server, and the long-lived session in that log window
was very likely the desktop's, not the phone's — there was no way to distinguish the two
from `client_id` alone in that log excerpt. The only firmly-established, repeatedly-
reproduced fact is the `ClosedCriticalStreamError` (finding 1) and, after fixing it, the
QPACK encoding mismatch (finding 2) — both confirmed via exact byte-level evidence, not
timing inference.
