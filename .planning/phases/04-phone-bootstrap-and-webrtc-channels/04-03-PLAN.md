<objective>
Deliver the session-durability slice: once a phone reaches the active view, it stays a live
controller. It heartbeats every 5s (so the server never evicts it prematurely), holds the screen
awake via Wake Lock, reports every state transition to the room, and dynamically grows/shrinks its
WebRTC mesh as desktops join and leave.

Purpose: Satisfy PHONE-06 (heartbeat + disconnect-within-65s) and PHONE-07 (Wake Lock), plus the
D-06/D-07 dynamic mesh and D-16/D-17/D-18 self-healing + state relay that make `player-ready` a
durable, observable session rather than a one-shot event.
Output: `handle_heartbeat` + `handle_heartbeat_miss` + `phones_missing_heartbeat` + `handle_phone_state`
+ peer-joined/peer-left pushes on the server, the background heartbeat monitor in main.rs, and the
Wake Lock / heartbeat / state-notification / peer-mesh handlers in phone.js.
</objective>

## Phase Goal (user story)

**As a** phone player joining a room via a QR scan, **I want to** grant motion access and connect
to every desktop in the room over unreliable WebRTC data channels while keeping my session alive,
**so that** my phone becomes a live low-latency motion controller the moment the game signals
`player-ready`.

> This plan delivers the durability slice: heartbeat + Wake Lock + state relay + dynamic mesh keep
> the active session alive and self-healing.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| phone → server (WT) | Phone-supplied heartbeat and phone-state messages cross here |
| server background monitor → room state | Timer-driven slot transitions act on server-held state only |
| server → desktops (relay) | phone-state and peer events are relayed to all room desktops |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-09 | DoS | heartbeat / phone-state flood from a phone | medium | mitigate | Handlers are O(1) stateless updates/relays with no allocation growth; existing 64 KiB message-size cap + per-connection semaphore (MAX_WT_CONNECTIONS) bound the rate; `from` guard prevents a phone spoofing another's heartbeat |
| T-04-10 | Spoofing | forged phone-state / heartbeat claiming another phone's identity | high | mitigate | `envelope.from` != connection-registered id is dropped before dispatch (wt_server ~line 200 / ws_server); heartbeat/phone-state resolve the slot by the registered phone_client_id, not client-asserted fields |
| T-04-11 | Tampering | peer-left injection to force-close a rival's channel | medium | mitigate | peer-left is server-originated (from handle_join/on_client_disconnect), never accepted from clients; the default broker arm cannot deliver a client-forged `peer-left` to the phone because only the server calls route_to_phone |
| T-04-12 | Denial of service | slot never released after heartbeat miss | low | mitigate | handle_heartbeat_miss reuses the bounded hold timer (`hold_ttl_secs`); slot is released by `release_slot_if_disconnected` if not reclaimed — no permanent leak |
| T-04-SC | Tampering | npm/cargo installs | low | accept | No new dependencies in Phase 4 (RESEARCH Package Legitimacy Audit: none) |
</threat_model>
