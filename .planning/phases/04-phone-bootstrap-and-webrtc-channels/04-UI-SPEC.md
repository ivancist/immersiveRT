## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding, status-dot margins |
| sm | 8px | Compact vertical spacing between label and value |
| md | 16px | Default vertical rhythm between form elements |
| lg | 24px | Phone card internal padding (replaces desktop 48px) |
| xl | 32px | Vertical padding above/below full-screen centred card |
| 2xl | 48px | Not used on phone client — reserved for desktop cards only |
| 3xl | 64px | Not used on phone client |

Exceptions:
- Touch target minimum 44px enforced on all interactive elements (encoded in `.btn { min-height: 44px }`).
- Phone card uses `padding: 24px` (not the desktop 48px) — mobile viewport constraint.
- Motion indicator circle diameter: 32px (not a spacing token, a fixed shape size).

**Source:** Inferred from existing `padding`, `gap`, `margin` values in `index.html`; mobile constraint applied to card padding.

## Phone Client: State Machine and Layout

The phone client renders exactly one view at a time. Each view is a full-viewport centred card (same pattern as `#view-phone` in `index.html`).

Card container (applies to all views):
- `display: flex; align-items: center; justify-content: center; min-height: 100vh; padding: 16px;`
- Inner card: `max-width: 360px; width: 100%; background: var(--color-surface); border-radius: 8px; padding: 24px; text-align: center;`

### View 1: Permission Gate (initial load)

Shown immediately on page load, before any WebTransport connection attempt.

| Element | Spec |
|---------|------|
| Heading | 20px / 600 — "ImmersiveRT" |
| Body text | 15px / 400 / secondary color — "Tap the button to enable motion control." |
| CTA button | `.btn` full-width — label: **"Grant Motion Access"** |
| Spacing above button | 24px (`margin-top: 24px`) |

**Interaction rules:**
- Button is the ONLY interactive element. No other links, no spinner, no connection attempt before tap.
- On tap: call `DeviceMotionEvent.requestPermission()` synchronously in the click handler.
- Android (no `requestPermission`): button tap triggers a brief "Activating..." disabled state (100ms), then auto-advances to View 2.
- iOS granted: advance to View 2.
- iOS denied: advance to View 5 (Error — permission denied).

### View 2: Connecting (after permission grant)

Shown between permission grant and `player-ready` event.

| Element | Spec |
|---------|------|
| Spinner | `.spinner` (24px, existing component, centred) |
| Status text | 15px / 400 — "Connecting... **X/Y** channels" where X and Y are live integers |
| X update | Increments by 1 each time an `RTCDataChannel` reaches `readyState === 'open'` |
| Y | Total desktop peers from `pair-ack.peers.length`, fixed after pair-ack received |
| Spacing below spinner | 16px |

**Interaction rules:**
- No interactive elements. Passive status display only.
- If `pair-ack` has `peers: []` (empty room, no other desktops), X/Y shows "0/0 channels" and the view advances immediately when `player-ready` fires.
- Channel counter format: bold the numbers — `Connecting... <strong>2</strong>/<strong>3</strong> channels`.

### View 3: Active / Ready (after `player-ready`)

Shown after server broadcasts `player-ready` and all channels confirmed.

| Element | Spec |
|---------|------|
| Player name | 28px / 600 — player's own username from `pair-ack.slot` |
| Room code | 13px / 600 / secondary color — "Room **ABCD**" (room code in monospace font) |
| Channel count | 15px / 400 — "**N/N** connected" with a status dot left of text |
| Status dot | 8px circle; `--color-status-connected` (#22c55e) when all connected |
| Spacing below channel count | 24px |
| Motion indicator label | 13px / 600 / secondary color — "Motion" |
| Motion indicator | 32px circle, centred; see animation spec below |
| Spacing below indicator | 8px |
| Hint text | 13px / 400 / secondary color — "Keep this screen open." |

**Motion indicator animation spec:**
- Element: `width: 32px; height: 32px; border-radius: 50%;`
- Idle state: `background: var(--color-status-empty)` (#444444)
- Active trigger: `devicemotion` event linear acceleration magnitude > 0.5 m/s² (Phase 5 calibrates exact threshold; use 0.5 as placeholder)
- Active state: `background: var(--color-accent)` (#7c6af7) + `box-shadow: 0 0 8px var(--color-accent)` + `transform: scale(1.3)`
- Transition in: `transition: background 0.1s ease-out, transform 0.1s ease-out, box-shadow 0.1s ease-out`
- Return to idle: 300ms after last motion event exceeds threshold — same transition, reversed

**Interaction rules:**
- No interactive elements on this screen. This is a passive status display.
- Wake Lock is active at this point (requested after `player-ready` — D-15).
- Channel count dot changes from `--color-status-hold` to `--color-status-connected` when all channels confirmed.

### View 4: Session Ended (slot lapsed — no reconnect in Phase 4)

Shown when the server closes the session or the slot eviction window (60s) expires without reconnect.

| Element | Spec |
|---------|------|
| Heading | 20px / 600 — "Session ended" |
| Body text | 15px / 400 / secondary color — "Your session timed out or the room closed." |
| Recovery hint | 13px / 600 / secondary color — "Scan the QR code again to start a new session." |

**Interaction rules:**
- No button — reconnect UX is deferred to Phase 5.
- No spinner, no retry logic.
- Color: all text in non-destructive colors (text-primary / text-secondary). This is an informational end state, not an error.

### View 5: Error — Permission Denied (iOS)

Shown when `DeviceMotionEvent.requestPermission()` returns "denied".

| Element | Spec |
|---------|------|
| Heading | 20px / 600 / `--color-destructive` — "Motion access denied" |
| Body text | 15px / 400 — "Motion control is required to use this controller." |
| Recovery hint | 13px / 600 / secondary color — "Enable it in iOS Settings > Safari > Motion & Orientation Access, then reload this page." |

**Interaction rules:**
- No retry button (reload instruction is sufficient; adding a button would re-trigger the denied state).
- No connection attempt initiated before this resolves.

### View 6: Error — Pairing Failed (server pair-error or no token)

Shown when the server returns a `pair-error` message, or when the page loads with no `token` query parameter, or when the token is expired/invalid.

| Element | Spec |
|---------|------|
| Heading | 20px / 600 / `--color-destructive` — "Pairing failed" |
| Body text | 15px / 400 — server error message or "This pairing link is invalid or has expired." |
| Recovery hint | 13px / 600 / secondary color — "Go back and scan the QR code again." |

**Interaction rules:**
- Reuses the existing `#error-state` pattern from `index.html #view-phone` — identical visual structure.
- No retry logic on the phone page itself.
