<objective>
Deliver the thinnest end-to-end phone slice: a phone player scans the QR code, sees a single
"Grant Motion Access" button, taps it, and the phone connects to the server over WebTransport,
pairs with its slot, and receives the full desktop roster plus ICE servers — ready to open
WebRTC channels in Plan 02.

Purpose: Stand up the new `phone.html` + `phone.js` artifact served at `/phone`, satisfy the iOS
permission gate (PHONE-02) and no-install QR delivery (PHONE-01), and extend the server `pair-ack`
so the phone has everything it needs (peers[], ice_servers) to fan out WebRTC in the next slice.
Output: `phone.html` (6-view shell), `phone.js` (permission gate + WT pair bootstrap), one-line
nginx change, enhanced `PairAckPayload`, and `SlotInfo.phone_client_id`.
</objective>

## Phase Goal (user story)

**As a** phone player joining a room via a QR scan, **I want to** grant motion access and connect
to every desktop in the room over unreliable WebRTC data channels while keeping my session alive,
**so that** my phone becomes a live low-latency motion controller the moment the game signals
`player-ready`.

> This plan delivers the first slice of that story: QR load → permission gate → paired with roster.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| phone browser → Rust server (WebTransport :4433) | Untrusted phone-supplied pairing token + client_id cross here |
| QR URL → phone browser | Pairing token travels in the URL query string |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-01 | Spoofing | `envelope.from` on WT/WS pair message | high | mitigate | Existing guard in wt_server.rs (~line 200) / ws_server drops any envelope whose `from` != the connection's registered id; phone_client_id recorded in SlotInfo is the registered id, not an attacker-chosen value |
| T-04-02 | Elevation of privilege | pairing token replay | high | mitigate | `PairingTokenStore::validate_and_consume` is single-use (T-03-01 precedent); a replayed token returns `pair-error` |
| T-04-05 | Information disclosure | pairing token in `/phone?token=` URL | low | accept | Phase 3 precedent (QR-scan context); token is short-lived (`pairing_ttl_secs` = 300) and single-use |
| T-04-06 | Input validation | malformed pair payload | medium | mitigate | `raw_payload["token"].as_str()` defensive extraction returns `pair-error`/`invalid_payload`; `parse_envelope` returns None on malformed bytes (never panics) |
| T-04-SC | Tampering | npm/cargo installs | low | accept | No new npm or Cargo dependencies in Phase 4 (RESEARCH Package Legitimacy Audit: none) — no install task, no [ASSUMED]/[SUS] packages |
</threat_model>
